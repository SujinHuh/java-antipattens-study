# 004. Point Refund Refactoring Plan

## 목표

포인트 환불 API의 기존 동작 의도를 유지하되, Controller, Service, Repository, Util 책임을 분리하고 면접에서 설명 가능한 구조로 개선한다.

## 기존 동작 보존 기준

- 결제 ID와 사용자 ID로 환불 대상 결제를 확인한다.
- 이미 환불된 결제는 다시 환불하지 않는다.
- 환불 성공 시 상태를 `REFUNDED`로 변경한다.
- 환불 이력을 사용자별로 조회할 수 있다.

## 리팩터링 우선순위

1. Controller가 `PointPayment`를 직접 Request/Response로 사용하지 않게 한다.
2. Service가 요청 객체를 신뢰하지 않고 Repository에서 기존 결제를 조회하게 한다.
3. 조회 실패, 이미 환불됨, 소유자 불일치, 정책 차단을 에러 코드가 있는 예외로 분리한다.
4. Repository 반복 조회를 제거하고 조건 조회 메서드를 둔다.
5. `static final List` 공유 상태를 인스턴스 기반 저장소로 바꾼다.
6. Util에 있던 환불 정책과 알림 책임을 `Policy`, `Notifier`로 분리한다.
7. 성공/실패/경계 테스트를 추가한다.

## 계층별 변경 이유

- Controller: HTTP 요청/응답과 요청 검증 경계만 담당한다.
- Service: 환불 유스케이스, 상태 전이, 트랜잭션 경계의 중심이 된다.
- Repository: 저장소 조회와 저장만 담당한다.
- Policy: 환불 가능 여부 판단을 담당한다.
- Notifier: 환불 알림 발송을 담당한다.

## 바꾸지 않은 부분과 이유

- 실제 Spring/JPA 의존성은 넣지 않는다. 1~8회차는 짧은 레거시 코드 리뷰 훈련이 목적이다.
- 외부 API 장애 대응은 직접 구현하지 않는다. 알림 실패 정책은 9회차 이후 확장 후보로 둔다.

## 테스트로 고정할 동작

- 정상 환불 성공
- 없는 결제 환불 실패
- 다른 사용자의 결제 환불 실패
- 이미 환불된 결제 환불 실패
- 사용자별 환불 이력 조회

## 트레이드오프

- 인메모리 저장소는 학습용 예시일 뿐 실제 운영 저장소가 아니다.
- `Optional`은 Repository 조회 반환에만 제한적으로 사용한다.
- 알림 발송 실패를 환불 실패로 볼지, 환불 성공 후 별도 재시도 대상으로 볼지는 도메인 정책 확인이 필요하다.
