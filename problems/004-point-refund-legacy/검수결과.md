# 004. Point Refund Review 교차검증 결과

기준 문서: `docs/15. AI 교차검증 실행 프롬프트.md`

## 교차검증 총평

`review-answer.md`의 기술 방향은 전반적으로 적절하다.

특히 아래 판단은 면접 기준으로 안전하다.

- Entity를 Request/Response로 직접 쓰는 문제
- Controller/Service/Repository 직접 `new` 생성 문제
- Service가 클라이언트 입력 상태를 신뢰하는 문제
- `findAll` 후 반복 `findById`를 "N+1 형태의 반복 Repository 조회"로 표현한 점
- Service에서 `ResponseEntity`를 반환하자는 사용자 답변을 위험 표현으로 잡은 점
- DTO 개선을 Lombok보다 API 계약/validation 중심으로 보정한 점
- static Util과 `@Component`의 의미 충돌을 잡은 점

## 틀렸거나 위험한 부분

### 1. `@Transactional` 설명은 "필수"보다 "필요할 수 있음"으로 말하는 것이 더 안전하다

현재 리뷰에서는 환불 상태 변경 유스케이스에 트랜잭션 경계가 필요하다고 설명했다.
방향은 맞지만, 이 문제 코드는 실제 DB/JPA가 아니라 인메모리 저장소 예제다.

면접에서는 아래처럼 말하는 것이 더 안전하다.

```text
실제 DB를 사용하는 Repository라면 조회, 상태 변경, 저장이 하나의 유스케이스이므로 Service 계층에 트랜잭션 경계를 두는 것이 일반적입니다.
현재 예제는 인메모리 코드지만 운영 저장소로 바뀌면 반드시 확인해야 할 지점입니다.
```

### 2. `PointRefundException` 자체가 나쁜 것은 아니다

리뷰에서 원본의 `PointRefundException`도 예외 정책 부재의 일부로 언급했다.
정확히는 `PointRefundException`을 만든 것 자체가 문제라기보다, 에러 코드/상태 매핑 없이 문자열 메시지만 전달하는 구조가 문제다.

더 안전한 표현:

```text
커스텀 예외를 둔 방향은 나쁘지 않지만, 에러 코드나 공통 예외 처리와 연결되지 않아 실패 원인을 구조적으로 구분하기 어렵습니다.
```

### 3. 알림 발송 실패 정책은 현재 코드만으로 단정하면 안 된다

`sendRefundNotice` 실패 정책 부재를 지적한 것은 좋다.
다만 실제 도메인에서는 알림 실패가 환불 실패와 분리될 수도 있고, 반드시 이벤트/큐로 분리해야 한다고 단정하면 위험하다.

더 안전한 표현:

```text
알림 실패 시 환불을 롤백할지, 환불은 성공시키고 알림만 재시도할지는 도메인 정책 확인이 필요합니다.
```

## 추가 관점

### 1. 권한/소유자 검증을 더 상단 핵심 문제로 올려도 된다

현재 리뷰에는 "환불 대상 결제가 요청한 사용자 소유인지 검증하지 않는다"가 `놓친 본질`에 들어 있다.
이 문제는 결제/환불 도메인에서는 매우 핵심적이므로 `핵심 문제 요약`이나 `반드시 잡았어야 하는 문제`에 더 강하게 넣어도 된다.

### 2. `PointRefundRequest`가 존재하지만 사용되지 않는 점도 명시하면 좋다

문제 코드에는 `PointRefundRequest`가 따로 있는데 Controller는 `PointPayment`를 받고 있다.
이건 "DTO가 있는데도 사용하지 않는 레거시 코드"라는 관점으로 볼 수 있다.

### 3. Repository의 `findAll()`이 내부 mutable List를 그대로 반환하는 문제는 중요하다

현재 `놓친 본질`에는 포함되어 있다.
다음 리뷰에서는 "외부에서 저장소 내부 컬렉션을 수정할 수 있다"는 실무 영향을 더 명확히 적으면 좋다.

## 더 안전한 면접 답변

```text
이 코드는 Controller가 PointPayment를 직접 요청과 응답에 사용하고 있어 API 계약과 내부 모델이 결합되어 있습니다.
또한 Service가 클라이언트가 보낸 payment 상태를 신뢰하고, 조회 실패 시 요청 객체를 그대로 환불 대상으로 사용하기 때문에 없는 결제나 조작된 금액도 환불될 수 있습니다.
실제 DB 기반 Repository라면 환불 상태 변경은 Service 계층의 트랜잭션 경계 안에서 처리하고, 조회 실패/이미 환불/소유자 불일치 같은 실패는 에러 코드가 있는 도메인 예외로 분리해야 합니다.
환불 이력 조회는 findAll 후 반복 findById가 아니라 userId와 status 조건으로 한 번에 조회하는 방식이 좋습니다.
```
