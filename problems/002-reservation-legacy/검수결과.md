# 🔍 002-reservation-legacy 코드 리뷰 검수 결과

## ✅ 학습자가 정확히 잡은 것 (칭찬)

### 1. Controller — 개별 문자열 파라미터 파싱 문제
실제 코드를 보면 `createReservation(String userId, String roomId, String startHour, String endHour, String status)`처럼 파라미터를 개별 문자열로 받고 있다. 학습자는 이 부분을 "dto로 받지 않고 각각 받음"이라고 정확히 짚었다. Controller는 요청 전체를 DTO 하나로 받아야 입력 유효성 검증(`@Valid`)과 구조 관리가 쉬워진다. **잘 잡은 핵심 포인트다.**

### 2. Controller — 수동 if 검증과 뭉뚱그린 예외 처리
`if (request.startHour < 0 || request.endHour > 24 ...)` 같은 도메인 검증 로직이 Controller에 들어 있고, `catch (Exception e)` 하나로 모든 예외를 잡아 `"500 INTERNAL_SERVER_ERROR"` 문자열을 반환한다. 학습자는 두 가지 모두 지적했다. 예외를 단일 타입으로 뭉개면 잘못된 요청인지 시스템 오류인지 클라이언트가 판단할 수 없어 운영 추적과 UX 모두 나빠진다. **정확한 지적이다.**

### 3. Repository — static List와 동시성·데이터 정확성 문제
`private static final List<Reservation> RESERVATIONS = new ArrayList<>()` 는 JVM 내에서 전체 인스턴스가 같은 리스트를 공유한다. 학습자는 "데이터 충돌", "데이터 정확성" 문제를 언급했고, 이는 실제로 멀티스레드 환경에서 ConcurrentModificationException 이나 레이스 컨디션으로 이어질 수 있다. **이전 문제(001)보다 훨씬 발전한 관찰이다.**

### 4. Repository — println 사용 문제
`System.out.println("reservation saved: " + reservation.id)` 코드를 학습자가 직접 지목하며 `log.`를 사용해야 한다고 언급했다. 프로덕션 코드에서 `println`은 로그 레벨·포맷·비동기 처리가 없어 성능·운영 모두 문제다. **실용적인 포인트를 잘 잡았다.**

### 5. Repository — return null NPE 위험
`findById()`가 값이 없으면 `return null`을 반환하는 것을 보고 NPE 우려를 언급했다. `Optional<Reservation>`이나 명시적 예외로 대체해야 한다는 방향이 맞다. **올바른 감지다.**

### 6. 문자열 상태값 오타 취약성과 enum 제안
`"CANCELLED"`, `"CONFIRMED"` 같은 문자열 리터럴이 코드 여러 곳에 흩어져 있고, 오타 하나로 버그가 생긴다는 점을 학습자가 여러 번 반복해서 지적하고 enum을 언급했다. 실제로 `existsOverlapping()`의 `!"CANCELLED".equals(...)`, Service의 `"BLOCKED".equals(...)`, `cancel()`의 `"CANCELLED".equals(...)` 등 다섯 곳 이상에 분산되어 있다. **중요한 문제를 일관되게 잡았다.**

### 7. Test — 단일 케이스와 JUnit 미사용
`@Test` 없이 성공 케이스 하나만 확인하는 테스트 구조를 잡았다. 중복 예약, 시간 오류, 취소 경우의 수를 검증하지 않는다는 점도 언급했다. **테스트 관점이 성장하고 있다.**

## 🟡 방향은 맞지만 표현/이해가 부족한 것

### 1. Service — "Reservation을 여기서 생성해야 해?"
학습자가 말하려 한 것은 아마도 **객체 생성과 필드 수동 할당이 한 메서드에 너무 많이 몰려 있다**는 느낌이었을 것이다. 그러나 표현이 "Setter/Getter를 사용해야 하는 것 아니냐"는 방향으로 흘러서 핵심에서 벗어났다.

더 정확한 표현은 이렇다:
> `reserve()` 한 메서드가 중복 여부 확인 → 상태 검증 → Entity 생성 → id 부여 → 저장 → 응답 문자열 생성까지 전부 처리한다. **단일 책임 원칙(SRP)** 위반이다. Entity 생성은 팩토리 메서드나 생성자로 분리하고, 응답 문자열 조립은 Controller가 담당해야 한다.

Setter/Getter 자체는 이 문제의 핵심이 아니다. 핵심은 **한 메서드의 책임 과부하**다.

### 2. @Transactional 위치에 대한 혼란
"트랜잭션은 Controller에 있어야 하는지 Service에 있어야 하는지"라고 방향을 고민했는데, 방향 자체는 절반 맞다. 학습자가 `@Transactional`의 위치를 의심하고 있다는 점에서 문제 의식은 생겼다.

더 정확한 이해는:
> `@Transactional`은 **유스케이스(업무 단위)의 경계**에 둔다. 예약을 생성하거나 취소하는 전체 흐름이 하나의 트랜잭션이어야 하므로 Service 메서드에 붙이는 것이 표준이다. Controller는 HTTP 요청/응답 경계를 담당하기 때문에 트랜잭션 범위를 여기서 열면 DB 커넥션이 불필요하게 오래 잡힌다.

### 3. AOP/횡단관심사 언급
"@Transactional은 횡단관심사이니까 한 번만 걸어줘도 좋다"고 언급했는데, 방향은 맞지만 논리가 약하다. 횡단관심사이기 때문에 한 번만 쓰는 것이 아니라, **self-invocation 문제** 때문에 `saveReservation()`의 별도 `@Transactional`이 실제로는 동작하지 않는다는 점을 잡아야 했다.

## ❌ 완전히 놓쳤거나 개념이 혼동된 것

### 1. DTO와 Entity 역할 혼동 (개념 역전)
학습자가 쓴 답변:
> "db와 직접 주고 받는 걸 dto가 하는 건가? entity가 하는 건가? **사용자한테 요청을 받아서 들어오는 게 entity가 받는 거 맞지?**"

이것은 반대다.

| 구분 | 역할 | 예시 |
|------|------|------|
| **DTO** | 외부(클라이언트) 요청/응답 데이터를 운반 | `ReservationRequest`, `ReservationResponse` |
| **Entity** | DB 저장 상태를 표현하는 도메인 객체 | `Reservation` |

사용자 요청은 **DTO**가 받고, Service가 검증한 뒤 **Entity**를 생성해서 저장한다. Entity를 API 경계에 직접 노출하면 DB 컬럼 구조와 API 구조가 강하게 결합되고, 클라이언트가 수정하면 안 되는 `createdAt`, `id` 같은 필드까지 받아버릴 수 있다. 지금 코드의 `ReservationRequest`에 `status`, `createdAt`이 들어 있는 것 자체가 이 문제다.

### 2. Self-Invocation 문제를 완전히 놓침
실제 코드를 보면:
```java
// ReservationService.java
@Transactional
public String reserve(ReservationRequest request) {
    ...
    saveReservation(reservation);  // ← 같은 클래스 내부 직접 호출
    ...
}

@Transactional
public void saveReservation(Reservation reservation) {
    reservationRepository.save(reservation);
}
```

Spring AOP는 **프록시 패턴**으로 동작한다. 외부에서 `service.reserve()`를 호출하면 프록시를 통해 들어오므로 `@Transactional`이 적용된다. 그런데 `reserve()` 내부에서 `this.saveReservation()`을 호출하면 **프록시를 거치지 않고 실제 객체를 직접 호출**한다. 따라서 `saveReservation()`의 `@Transactional`은 실제로 아무런 효과가 없다.

이것은 Spring을 쓰는 개발자라면 반드시 알아야 하는 함정이고, 가장 놓치기 쉬운 문제 중 하나다.

### 3. Service가 HTTP 상태 문자열을 반환하는 문제
```java
// ReservationService.java — cancel()
if (reservation == null) return "404 NOT_FOUND";
if ("CANCELLED".equals(reservation.status)) return "409 ALREADY_CANCELLED";
return "200 OK";
```

Service가 `"404 NOT_FOUND"`, `"409 ALREADY_CANCELLED"`, `"200 OK"` 같은 **HTTP 의미의 문자열**을 직접 반환하고 있다. Service가 HTTP 프로토콜 표현에 묶여버린 상태다. Service가 HTTP를 알아야 할 이유가 없다.

올바른 구조:
- Service는 도메인 결과(도메인 예외 또는 결과 객체)만 반환한다.
- Controller(또는 `@ExceptionHandler`)가 그것을 HTTP 응답으로 변환한다.

학습자는 Controller의 예외 처리가 뭉뚱그려져 있다고 지적했지만, **Service 자체가 HTTP 코드를 반환하는 더 근본적인 문제**는 짚지 못했다.

### 4. Repository의 save()가 비즈니스 정책을 판단하는 책임 위반
```java
// ReservationRepository.java
public void save(Reservation reservation) {
    if ("CANCELLED".equals(reservation.status) && reservation.endHour <= reservation.startHour) {
        throw new ReservationException("invalid");
    }
    RESERVATIONS.add(reservation);
    ...
}
```

학습자는 `existsOverlapping()`의 조건 로직은 지적했지만, `save()` 안에 비즈니스 검증이 들어 있다는 점은 명확히 짚지 못했다. Repository의 책임은 **저장과 조회**뿐이다. 예약 상태가 유효한지, 시간 범위가 올바른지 판단하는 것은 Service(또는 도메인 객체)의 책임이다. Repository가 이런 검증을 하면 Service를 우회해서 Repository를 직접 쓸 때 정책이 꼬이고, 테스트 작성도 어려워진다.

### 5. Controller와 Service 모두에서 의존성 주입(DI) 미사용
```java
// ReservationController.java
private final ReservationService reservationService = new ReservationService();

// ReservationService.java
private final ReservationRepository reservationRepository = new ReservationRepository();
```

Controller에서 Service를, Service에서 Repository를 `new`로 직접 생성한다. 학습자가 Controller의 `new ReservationService()`는 언급했지만, **의존성 주입(DI)이 전혀 없다는 점**을 명시적인 문제로 지적하지는 않았다. DI가 없으면 단위 테스트에서 Mock 객체를 주입할 수 없고, 의존 관계를 교체하려면 코드를 직접 바꿔야 한다.

## 🎯 72점 점수 평가

**72점은 적절하다. 오히려 약간 관대한 편에 속한다.**

### 점수를 지지하는 근거 (잘한 부분)
- static List, println, null 반환, 문자열 상태값 오타 위험, 테스트 구조 문제 등 **표면적으로 눈에 보이는 문제들을 골고루 잡았다**.
- 001 문제보다 관찰 범위가 넓어졌다는 성장이 보인다.
- "왜 문제인가"에 대한 설명은 짧지만, 방향은 맞다.

### 점수에서 깎인 이유 (놓친 부분)
- DTO/Entity 역할을 **완전히 반대로** 이해했다. 오해가 아니라 역전이기 때문에 감점 비중이 크다.
- Self-invocation 문제는 Spring AOP 핵심 개념이고, 이 코드의 가장 중요한 숨은 버그인데 **전혀 언급이 없었다**.
- Service가 HTTP 문자열을 반환하는 **계층 책임 혼재**를 놓쳤다.
- Repository의 `save()`에 비즈니스 정책이 들어 있다는 **책임 원칙 위반**을 명확히 지적하지 못했다.

시니어 관점에서 72점은 "기본기 감각은 있는 주니어"에 해당한다. 눈에 보이는 코드 냄새는 잡을 수 있지만, 아키텍처 계층 책임과 Spring 내부 동작 원리에서 구멍이 있다.

## 📌 다음 학습에서 집중해야 할 것 TOP 3

### 🥇 1순위: DTO / Entity 역할 구분 — 개념을 완전히 재정립할 것

**이유**: 이번 리뷰에서 가장 큰 개념 오류다. DTO와 Entity가 뒤바뀐 상태에서는 레이어드 아키텍처 전체를 잘못 이해하게 된다. 아래 흐름을 손으로 그리며 외울 것.

```
클라이언트 요청
    → Controller   : DTO로 받음 (ReservationRequest)
    → Service      : DTO 검증 → Entity 생성 (Reservation)
    → Repository   : Entity 저장
    → Service      : Entity → 응답 DTO 변환 (ReservationResponse)
    → Controller   : 응답 DTO → HTTP 응답 변환
```

단순 암기가 아니라 **"왜 분리해야 하는가"**(DB 변경 격리, 클라이언트 조작 방지, API 안정성)를 설명할 수 있어야 한다.

### 🥈 2순위: Spring AOP 프록시와 Self-Invocation — 작동 원리를 이해할 것

**이유**: `@Transactional`, `@Cacheable`, `@Async` 같은 Spring AOP 기반 애노테이션은 **모두 같은 함정**을 갖고 있다. 프록시가 어떻게 동작하는지 모르면 애노테이션을 붙여도 효과가 없는 코드를 만들고, 원인을 재현하기도 어렵다.

직접 실험해볼 것:
1. `@Transactional`이 붙은 메서드가 같은 클래스의 다른 `@Transactional` 메서드를 호출할 때 트랜잭션이 몇 개 열리는지 확인한다.
2. 내부 호출 시 두 번째 `@Transactional`이 실제로 무시된다는 것을 로그로 증명해본다.

### 🥉 3순위: 계층 책임 원칙 — "이 코드가 여기 있어야 하나?"를 질문하는 습관

**이유**: 학습자는 이미 "이게 여기에 있어야 하나?"라는 질문을 여러 번 하고 있다(매우 좋은 신호). 하지만 그 답을 아직 체계적으로 내리지 못하고 있다. 다음 기준을 기억할 것.

| 계층 | 책임 |
|------|------|
| **Controller** | HTTP 요청 수신, 입력 DTO 검증, 응답 포맷 변환 |
| **Service** | 비즈니스 로직, 트랜잭션 경계, 도메인 조합 |
| **Repository** | 저장·조회·삭제, 비즈니스 판단 없음 |
| **Entity/Domain** | 도메인 상태와 불변 규칙 보호 |

코드를 볼 때마다 **"이 코드의 계층은 어디인가? 이 코드가 여기 있으면 어떤 문제가 생기는가?"** 를 자문하는 습관을 들인다. Service가 HTTP 코드를 반환하거나 Repository가 비즈니스를 판단하는 것이 모두 이 기준에서 걸러진다.
