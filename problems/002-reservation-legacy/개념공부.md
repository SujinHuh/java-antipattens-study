## 🔍 1. Request DTO에 서버 전용 필드가 포함된 설계 결함

### 📍 문제 코드 비교

[ReservationRequest.java](src/main/java/com/example/reservation/ReservationRequest.java) (DTO)
```java
public class ReservationRequest {
    public long userId;
    public long roomId;
    public int startHour;
    public int endHour;
    public String status;    // ❌ 서버가 결정해야 할 값
    public String createdAt; // ❌ 서버가 현재 시각으로 채워야 할 값
}
```

[Reservation.java](src/main/java/com/example/reservation/Reservation.java) (Entity)
```java
public class Reservation {
    public long id;          // ← 서버/DB 생성
    public long userId;
    public long roomId;
    public int startHour;
    public int endHour;
    public String status;    // ← 서버 비즈니스 로직이 결정
    public String createdAt; // ← 서버가 now()로 채움
}
```

두 클래스를 나란히 놓고 보면 거의 똑같습니다.
`id`만 없을 뿐, `status`와 `createdAt`이 Request DTO에도 그대로 들어 있어요.
이건 **"클라이언트가 서버 내부 상태를 직접 조작할 수 있는 구멍"** 이 뚫린 것입니다.

### 💥 문제 1: `status`를 클라이언트가 직접 보낸다

[ReservationService.java](src/main/java/com/example/reservation/ReservationService.java) 24번째 줄:
```java
reservation.status = request.status; // ← 클라이언트가 보낸 값을 그대로 씀
```

**조작 위협 시나리오**:
```
정상 요청:
  클라이언트 → { status: "CONFIRMED" } → DB 저장 ✅

악의적 조작:
  클라이언트 → { status: "PAID" }      → 결제 완료로 DB에 기록됨 💥
  클라이언트 → { status: "ADMIN" }     → 관리자 예약으로 DB에 기록됨 💥
  클라이언트 → { status: "BLOCKED" }   → (서비스에서 예외가 터지긴 하지만,
                                          이걸 서비스에서 막는 게 아니라 DTO 자체에서 받지 말아야 함)
```

**올바른 설계**:
```java
// ReservationService.java
Reservation reservation = new Reservation();
reservation.userId = request.userId;
reservation.roomId = request.roomId;
reservation.startHour = request.startHour;
reservation.endHour = request.endHour;
reservation.status = "CONFIRMED"; // ← 서버가 직접 결정!
reservation.createdAt = LocalDateTime.now().toString();
```

## 🔍 2. `@Transactional`이 실제로 동작하지 않는다

### 📍 문제 코드

[Transactional.java](src/main/java/com/example/reservation/Transactional.java) (직접 만든 애노테이션)
```java
// Spring의 @Transactional이 아닌, 직접 만든 껍데기 애노테이션
public @interface Transactional { }
```

[ReservationService.java](src/main/java/com/example/reservation/ReservationService.java)
```java
@Transactional          // ← 이게 진짜 Spring @Transactional이 아님!
public String reserve(ReservationRequest request) { ... }

@Transactional          // ← 마찬가지로 가짜
public void saveReservation(Reservation reservation) { ... }
```

### 💥 왜 문제인가?

Spring의 `@Transactional`은 Spring AOP가 프록시를 생성해서 트랜잭션을 열고 닫아주는 방식으로 동작합니다.
직접 만든 `@interface Transactional`은 **이름만 같을 뿐, 실제 트랜잭션 기능이 전혀 없습니다**.

```
실제 동작 비교:
  Spring @Transactional     직접 만든 @Transactional
  ───────────────────────   ───────────────────────
  트랜잭션 시작              아무 일도 일어나지 않음
  로직 실행                  로직 실행 (트랜잭션 없이)
  성공 → 커밋               성공 → 그냥 끝남
  실패 → 롤백               실패 → 롤백 없이 데이터 반쪽만 저장됨 💥
```

**실제 위험 시나리오**:
```
reserve() 메서드 안에서:
  1. 예약 저장 → 성공 (DB에 반영)
  2. 알림 발송 → 예외 발생!

진짜 @Transactional이라면: 1번도 롤백 → 데이터 일관성 유지
가짜 @Transactional이라면: 1번은 그대로 커밋됨 → 예약은 됐는데 알림은 없는 불일치 상태 💥
```

### ✅ 올바른 해결

```java
// import를 올바르게!
import org.springframework.transaction.annotation.Transactional; // ← 진짜 Spring 것
```

## 🔍 4. Repository가 비즈니스 판단을 하고 있다 (책임 위반)

### 📍 문제 코드

[ReservationRepository.java](src/main/java/com/example/reservation/ReservationRepository.java) 10~12번째 줄:
```java
public void save(Reservation reservation) {
    // ❌ "CANCELLED 상태이고 endHour <= startHour면 예외" → 이건 비즈니스 규칙!
    if ("CANCELLED".equals(reservation.status) && reservation.endHour <= reservation.startHour) {
        throw new ReservationException("invalid");
    }
    RESERVATIONS.add(reservation);
}
```

### 💥 왜 문제인가?

각 계층의 역할:
```
Controller  → 요청 수신 / 응답 반환 / 입력값 파싱
Service     → 비즈니스 규칙 적용, 흐름 제어
Repository  → 저장소 접근 (저장, 조회, 삭제) 만!
```

Repository가 `"CANCELLED 상태이면서 시간이 이상하면 예외를 던진다"`는 **비즈니스 판단**을 하면:
- 같은 검증 로직이 Service에도 있을 경우 **중복 검증**이 생깁니다.
- Repository를 테스트할 때 비즈니스 조건까지 같이 테스트해야 해서 **테스트가 복잡**해집니다.
- 나중에 비즈니스 규칙이 바뀌면 Repository도 같이 바꿔야 해서 **변경에 취약**합니다.

### ✅ 올바른 구조

```java
// Service에서 비즈니스 판단
public String reserve(ReservationRequest request) {
    if (request.endHour <= request.startHour) {
        throw new InvalidReservationTimeException(...); // Service가 판단
    }
    // ...
    reservationRepository.save(reservation); // Repository는 그냥 저장만
}

// Repository는 저장만
public void save(Reservation reservation) {
    RESERVATIONS.add(reservation); // 판단 없이 순수하게 저장
}
```

## 🔍 5. `@Transactional` Self-Invocation 문제 (AOP 우회)

### 📍 문제 코드

[ReservationService.java](src/main/java/com/example/reservation/ReservationService.java) 27번째 줄:
```java
@Transactional
public String reserve(ReservationRequest request) {
    // ...
    saveReservation(reservation); // ← 같은 클래스의 메서드를 직접 호출!
}

@Transactional  // ← 이 어노테이션은 이 호출에서는 아무 의미 없음
public void saveReservation(Reservation reservation) {
    reservationRepository.save(reservation);
}
```

### 💥 왜 문제인가?

Spring `@Transactional`은 **Spring AOP 프록시** 방식으로 동작합니다.

```
외부에서 호출할 때 (AOP 프록시 정상 적용):
  Controller
    │
    └─→ [Spring Proxy(프록시)]  ← AOP가 여기서 트랜잭션을 시작함
              │
              └─→ ReservationService.reserve()  ✅ 트랜잭션 적용

같은 클래스 내부에서 호출할 때 (Self-Invocation):
  reserve()
    │
    └─→ this.saveReservation()  ← 프록시를 거치지 않고 직접 호출!
                                   ❌ 트랜잭션 미적용
```

**핵심**: `this.saveReservation()`은 Spring 프록시 객체가 아닌 **자기 자신(this)**을 직접 호출하기 때문에, `saveReservation()`에 붙은 `@Transactional`이 무시됩니다.

단, 현재 이 코드의 `@Transactional`은 직접 만든 가짜 어노테이션이라 어차피 동작 자체가 안 되지만, 진짜 Spring `@Transactional`을 사용했더라도 self-invocation이면 동일하게 AOP가 우회됩니다.

### ✅ 해결 방법

```java
// 방법 1: self-invocation이 필요한 경우 별도 클래스로 분리
@Service
public class ReservationSaver {
    @Transactional
    public void save(Reservation reservation) {
        reservationRepository.save(reservation);
    }
}

// 방법 2: saveReservation() 자체를 없애고 reserve() 안에 직접 처리
@Transactional
public String reserve(ReservationRequest request) {
    // ...
    reservationRepository.save(reservation); // 직접 repository 호출
}
```

## 🔍 6. `status`를 문자열(String)로 관리한다 (enum 미사용)

### 📍 문제 코드

코드 곳곳에 문자열이 하드코딩되어 있습니다:
```java
// ReservationService.java
if ("BLOCKED".equals(request.status)) { ... }
reservation.status = request.status;

// ReservationRepository.java
if ("CANCELLED".equals(reservation.status) ...) { ... }
&& !"CANCELLED".equals(reservation.status)
```

### 💥 왜 문제인가?

```
오타가 나도 컴파일 에러 없음:
  "CANCLED"  → 오타지만 컴파일 통과 → 런타임에서 조건 불일치 버그 💥
  "confirmed" → 소문자지만 컴파일 통과 → equals() 비교 실패 버그 💥

허용되지 않는 값도 막을 수 없음:
  reservation.status = "아무말이나"; → 컴파일 통과 → 데이터 오염 💥

IDE 자동완성 불가:
  유효한 상태값이 뭔지 코드만 봐서는 알 수 없음
```

### ✅ enum으로 해결

```java
// 상태값을 enum으로 정의
public enum ReservationStatus {
    CONFIRMED, CANCELLED, BLOCKED
}

// 이제 오타 → 컴파일 에러로 즉시 잡힘
if (ReservationStatus.BLOCKED == request.status) { ... } // ✅

// 허용되지 않는 값 → 컴파일 에러
reservation.status = "아무말이나"; // ❌ 컴파일 에러!
reservation.status = ReservationStatus.CONFIRMED; // ✅
```

## 🔍 7. 테스트가 JUnit 형식을 따르지 않는다

### 📍 문제 코드

[ReservationServiceTest.java](src/test/java/com/example/reservation/ReservationServiceTest.java):
```java
public class ReservationServiceTest {
    // ❌ @Test 어노테이션 없음 → JUnit이 이 메서드를 테스트로 인식 못함
    public void reserveRoom() {
        ReservationService service = new ReservationService();

        ReservationRequest request = new ReservationRequest();
        request.status = "CONFIRMED"; // ❌ 클라이언트가 status를 직접 지정

        String result = service.reserve(request);

        // ❌ assertThat() 대신 if문으로 직접 검증 → 실패 메시지가 불명확
        if (!result.contains("status=CONFIRMED")) {
            throw new AssertionError("reservation failed");
        }
    }
    // ❌ @BeforeEach로 static RESERVATIONS 초기화 없음 → 테스트 격리 안됨
}
```

### 💥 문제 목록

| 문제 | 영향 |
|------|------|
| `@Test` 없음 | Maven/Gradle 빌드 시 테스트가 아예 실행 안 됨 |
| `if`문으로 검증 | 실패 시 어떤 값이 나왔는지 알 수 없음 |
| `@BeforeEach` 없음 | 테스트 간 static 리스트 데이터 공유 (001과 동일한 문제) |
| `request.status` 직접 세팅 | DTO 설계 결함을 테스트가 그대로 수용하고 있음 |

### ✅ 올바른 테스트

```java
class ReservationServiceTest {
    @BeforeEach
    void setUp() {
        // static 리스트 초기화 (테스트 격리)
        ReservationRepository.RESERVATIONS.clear();
    }

    @Test
    void 정상_예약_성공() {
        ReservationService service = new ReservationService();

        ReservationRequest request = new ReservationRequest();
        request.userId = 1L;
        request.roomId = 10L;
        request.startHour = 10;
        request.endHour = 12;
        // status는 Request에 없어야 하므로 세팅 자체를 안 함

        String result = service.reserve(request);

        // assertThat으로 명확하게 검증 + 실패 시 실제값 출력
        assertThat(result).contains("CONFIRMED");
    }
}
```
