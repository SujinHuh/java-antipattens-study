# Refactoring Plan

이 문서는 003 풀이 후 비교 학습용이다. 원본 코드가 유일하게 틀렸다는 뜻이 아니라, 면접에서 설명하기 쉬운 개선 방향 한 가지를 정리한다.

## 기존 동작 보존 기준

- 쿠폰 코드와 사용자 id를 받아 쿠폰 적용 결과를 반환한다.
- 사용자의 주문 요약을 조회한다.
- 쿠폰을 사용할 수 없으면 명확한 예외로 실패한다.
- 쿠폰 적용 성공, 쿠폰 없음, 이미 사용된 쿠폰, 다른 사용자 쿠폰, 주문 요약 조회, 잘못된 요청 케이스를 테스트로 고정한다.

## 리팩터링 우선순위

1. Controller가 Entity를 직접 받지 않도록 Request/Response DTO를 분리한다.
2. Controller의 수동 검증과 상태 코드 문자열 반환을 제거한다.
3. Service 반환 타입을 `Object`에서 명확한 응답 DTO로 바꾼다.
4. Service가 요청 객체를 그대로 신뢰하지 않고, 기존 쿠폰을 조회해 존재 여부, 소유자, 상태를 검증하게 한다.
5. Service에 트랜잭션 경계를 둔다.
6. 반복문 안 Repository 조회를 한 번의 조회로 바꾼다.
7. Util의 쿠폰 정책 로직을 정책 객체로 이동한다.
8. 문자열 상태값을 enum으로 바꾼다.
9. RuntimeException 직접 사용을 도메인 예외와 에러 코드로 바꾼다.
10. 실행 가능한 테스트 케이스를 추가한다.

## 계층별 변경 이유

### Controller

- 요청 DTO를 받고 Service를 호출하는 역할로 좁힌다.
- 잘못된 요청은 validation/예외 처리로 분리하고, Controller가 비즈니스 판단을 하지 않게 한다.

### Service

- 쿠폰 적용 유스케이스를 담당한다.
- 요청 DTO의 `code`로 기존 쿠폰을 조회하고, 쿠폰 없음/소유자 불일치/이미 사용 또는 만료 상태를 검증한다.
- 상태 변경과 저장이 있으므로 트랜잭션 경계를 Service에 둔다.
- 응답 문자열이나 Entity 반환 대신 응답 DTO를 반환한다.

### Repository

- 주문 id별 금액을 반복 조회하지 않고, 사용자 주문 목록을 한 번에 조회한다.
- null 반환과 static 공유 상태를 피한다.
- 인메모리 예제에서는 발급된 쿠폰 목록을 들고 있고, Service는 이 저장소에서 기존 쿠폰을 조회한다.

### Policy

- 원본 `CouponUtil`에 있던 쿠폰 사용 가능 여부와 할인 계산은 비즈니스 정책이다.
- 단순 Util이 아니라 `CouponPolicy_Refactored`로 분리해 테스트와 변경을 쉽게 한다.

## 바꾸지 않은 부분과 이유

- 실제 Spring import는 넣지 않고 어노테이션은 주석으로 유지한다. 현재 저장소가 실제 Spring Boot 프로젝트가 아니기 때문이다.
- 실제 DB 구현은 넣지 않는다. 면접용 짧은 레거시 코드 비교가 목적이다.

## 테스트로 고정할 동작

- VIP 쿠폰 적용 성공
- 존재하지 않는 쿠폰 실패
- 이미 사용된 쿠폰 실패
- 다른 사용자 소유 쿠폰 실패
- 주문 요약 조회
- 잘못된 요청 입력 실패

## 면접용 설명 문장

이 코드는 Controller가 Entity를 직접 받고 상태 코드를 문자열로 반환하며, Service는 Object를 반환하고 클라이언트가 보낸 쿠폰 상태를 그대로 신뢰합니다. 또 Repository 조회를 반복문 안에서 수행하고, 쿠폰 정책이 Util에 들어가 있어 계층 책임이 흐려져 있습니다. 개선한다면 요청/응답 DTO를 분리하고, Service는 쿠폰 코드를 기준으로 서버 저장소의 기존 쿠폰을 조회한 뒤 존재 여부, 소유자, 사용 상태를 검증하게 하겠습니다. 쿠폰 상태는 enum으로 제한하고, 쿠폰 정책은 별도 정책 객체로 분리하며, Repository는 필요한 데이터를 한 번에 조회하도록 바꾸겠습니다.
