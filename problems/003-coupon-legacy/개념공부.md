# 003-coupon-legacy 개념 정리

3번 문제(쿠폰 발급 및 사용) 코드 리뷰를 통해 발견한 안티패턴과 핵심 개념을 정리합니다.

## 1. Controller 파라미터로 Entity 직접 수신
- **문제**: Controller가 클라이언트 요청을 받을 때 `CouponRequest` 같은 DTO를 쓰지 않고 `Coupon` Entity를 직접 받고 있습니다.
- **결과**: 사용자가 임의로 조작하면 안 되는 값(예: `status`, `used`, `discountAmount`)까지 외부에서 덮어쓸 수 있고, API 규격이 내부 저장/도메인 모델에 강하게 묶입니다.
- **관련 개념**: [DTO vs Entity (03-dto-entity.md)](../../docs/공통개념/03-dto-entity.md)

## 2. Util 클래스에 비즈니스 로직(정책) 포함
- **문제**: 쿠폰 사용 가능 여부 판별, 할인율 계산 등 '핵심 비즈니스 규칙'이 `CouponUtil` 같은 Util 클래스에 들어있습니다.
- **결과**: 순수한 도구여야 할 Util이 비즈니스 정책을 알게 되어, 정책이 바뀔 때마다 공통 유틸리티를 수정해야 하는 결합도 문제가 생깁니다.
- **관련 개념**: [Util (Utility) 클래스 (09-util-classes.md)](../../docs/공통개념/09-util-classes.md)

## 3. 반복문 내부의 Repository 반복 조회
- **문제**: Service 계층에서 `for`문 같은 반복문을 돌면서, 그 내부에서 매번 `couponRepository.findOrderAmount()`를 호출합니다.
- **결과**: 현재 예제는 인메모리 코드지만, 실제 DB나 외부 저장소라면 데이터가 1000개일 때 조회도 1000번 반복되어 서버 지연이나 Timeout을 유발할 수 있습니다.
- **관련 개념**: [반복문 내부의 DB 조회 (10-db-loop-antipattern.md)](../../docs/공통개념/10-db-loop-antipattern.md)

## 4. 트랜잭션(@Transactional) 부재
- **문제**: 쿠폰의 상태를 변경(`USED`)하고 저장소에 다시 저장하는 로직이 있는데도 해당 Service 메서드에 `@Transactional`이 없습니다.
- **결과**: 현재 예제는 인메모리 코드라 실제 트랜잭션이 동작하지는 않지만, 실제 DB나 영속 저장소를 사용하는 코드라면 중간 실패 시 롤백이 되지 않아 데이터 정합성이 깨질 수 있습니다.
- **관련 개념**: [Spring AOP와 프록시 (06-spring-aop-proxy.md)](../../docs/공통개념/06-spring-aop-proxy.md)

## 5. Controller의 책임 위반 (비즈니스 검증)
- **문제**: `request.orderAmount < 0` 같은 도메인/비즈니스 관점의 검증 로직이 Controller에 하드코딩되어 있습니다.
- **결과**: 계층의 역할이 섞여 Service 로직만 따로 떼어서 재사용하거나 테스트하기 어려워집니다.
- **관련 개념**: [계층별 책임 (05-layer-responsibility.md)](../../docs/공통개념/05-layer-responsibility.md)
