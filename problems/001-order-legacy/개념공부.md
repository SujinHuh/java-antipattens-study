## 🔍 1. static 리스트를 데이터베이스(DB) 대용으로 사용하는 안티패턴

### 📍 문제 코드
[OrderRepository.java](file:///Users/huhsujin/Documents/Playground/java-antipatterns/problems/001-order-legacy/src/main/java/com/example/order/OrderRepository.java#L7)
```java
public class OrderRepository {
    private static final List<Order> ORDERS = new ArrayList<>();
    
    // ...
}
```

### 💥 문제의 현상과 한계

#### A. 테스트 격리(Test Isolation)의 붕괴
- JVM 메모리에 `static` 변수가 평생 잔존하므로, 테스트 A가 수정한 데이터가 테스트 B에 영향을 미칩니다.
- **실제 테스트 실패 시나리오**:
  ```java
  // Test Class
  class OrderRepositoryTest {
      private final OrderRepository repo = new OrderRepository();

      @Test
      void test1_주문저장() {
          repo.save(new Order(1L, 10000));
          assertThat(repo.findById(1L)).isNotNull(); // ✅ 성공
      }

      @Test
      void test2_주문이비어있어야함() {
          // 기대값: 새로 실행되었으므로 저장소는 비어있어야 함
          // 결과: test1에서 저장된 데이터가 남아있어 리스트 사이즈가 1이 됨 -> ❌ 실패!
          assertThat(repo.findAll()).isEmpty(); 
      }
  }
  ```
- 이에 대한 상세한 `static` 메모리 동작 방식은 [공통개념.md - 1. Java static 키워드와 메모리 구조](file:///Users/huhsujin/Documents/Playground/java-antipatterns/docs/공통개념.md#1-java-static-키워드와-메모리-구조)를 참조하세요.

#### B. 동시성(Concurrency) 문제
- 실서비스에서 동시에 100명이 주문을 진행한다면, 여러 스레드가 동시에 `ORDERS.add(order)`를 실행합니다.
- `ArrayList`는 Thread-Safe하지 않아 인덱스가 중복 지정되거나 데이터가 덮어써져 유실될 위험이 높습니다.
- 관련 이론 및 해결 가능한 Thread-Safe 컬렉션은 [공통개념.md - 2. 동시성과 Thread-Safe](file:///Users/huhsujin/Documents/Playground/java-antipatterns/docs/공통개념.md#2-동시성concurrency과-thread-safe)를 참고하세요.

## 🛡️ 2. Request DTO에 `finalPrice` 필드가 포함된 설계 결함

### 📍 문제 코드
[OrderRequest.java](file:///Users/huhsujin/Documents/Playground/java-antipatterns/problems/001-order-legacy/src/main/java/com/example/order/OrderRequest.java#L7)
```java
public class OrderRequest {
    public String customerType;
    public int itemCount;
    public int pricePerItem;
    public int finalPrice; // ◀️ 클라이언트가 금액을 직접 지정해서 보내도록 설계됨!
}
```

### 💥 조작 위협 시나리오
- 원래 `OrderService`에서 `itemCount * pricePerItem`에 할인을 적용해 계산해야 하지만, 클라이언트 요청 필드에 `finalPrice`가 있으면 공격자가 직접 조작된 데이터를 보낼 수 있습니다.
- 예를 들어, 상품 단가 10,000원짜리 10개를 주문하면서 요청 바디로 `finalPrice: 100`을 보냈을 때, 만약 개발자가 실수로 컨트롤러나 서비스에서 `request.finalPrice`를 검증 없이 사용하면 심각한 금전적 손실이 발생합니다.
- 요청 데이터 설계 시 클라이언트를 맹신하지 않고 서버 독립적으로 비즈니스 처리를 수행하는 상세 원칙은 [공통개념.md - 3. DTO 설계 원칙](file:///Users/huhsujin/Documents/Playground/java-antipatterns/docs/공통개념.md#3-dto-데이터-전달-객체-설계-원칙)에서 확인하세요.

```java
        throw new OrderException("invalid order"); // ◀️ 어떤 데이터가 왜 유효하지 않은지 전혀 알 수 없음
    }
    // ...
}
```

[OrderController.java](file:///Users/huhsujin/Documents/Playground/java-antipatterns/problems/001-order-legacy/src/main/java/com/example/order/OrderController.java#L14)
```java
try {
    return orderService.processOrder(customerType, count, price);
} catch (Exception e) {
    return "500 INTERNAL_SERVER_ERROR"; // ◀️ 발생한 Exception e를 로깅하지 않고 삼켜버림
}
```

### 💥 실제 운영 디버깅 곤경
- 실서비스에서 500 에러가 발생했을 때 로그에 `invalid order`라는 단 한 줄만 찍히거나, 에러 로그 자체가 유실됩니다.
- 구체적으로 어떤 유저의 무슨 주문건 때문에 데이터가 잘못 들어왔는지, 소스코드의 어떤 흐름에서 에러가 전파되었는지(`Stack Trace`) 추적할 길이 완전히 막혀버립니다.
- 올바른 예외 전파, 예외 세분화 및 정보 로깅 원칙은 [공통개념.md - 4. 예외 처리 설계 원칙](file:///Users/huhsujin/Documents/Playground/java-antipatterns/docs/공통개념.md#4-예외exception-처리-설계-원칙)을 참고하세요.
