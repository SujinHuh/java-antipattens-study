# 006. Order Cancel Legacy - 문제검수결과

## 총평

현재 006번 문제는 **출제 가능 상태로 재설계 완료**했다.

주문 취소 도메인은 004 포인트 환불, 005 배송지 변경과 다른 업무 흐름이며, 23번 출제 매트릭스의 여러 개념을 반복 학습하기에 적합하다.
사용자가 한 문제 안에서 여러 개념을 반복해서 익히고 싶다고 했으므로, 핵심 개념을 6~7개 수준으로 넓히되 코드 양은 40분 안에 리뷰 가능한 수준으로 유지했다.

05번과 비슷한 `Controller new + Map 응답` 중심 반복을 줄이고, 006번은 Service 내부 상태 변경, 요청값 신뢰, Optional, 외부 환불 호출 순서, OCP 분기 쪽에 문제를 더 숨겼다.

## 보강 완료 항목

### 1. 답변 전 힌트 노출 제거

- `OrderService.java`, `OrderCancelUtil.java`에 있던 `❌ Optional 오남용`, `❌ 소유자 검증 누락`, `❌ SOLID OCP 위반` 같은 정답 주석을 제거했다.
- 기존 `개념공부.md`는 `답변후-개념공부.md`로 변경했다.
- `README.md`에는 특정 개념명이나 정답 힌트를 노출하지 않고, 코드 읽는 순서만 남겼다.

### 2. 문제 구조 보강

- `README.md` 추가
- `src/main/main.iml` 추가
- `src/test/test.iml` 추가
- `.idea/modules.xml`에 006 main/test module 등록
- `javac` 기준 main/test 컴파일 통과 확인

### 3. 23번 출제 매트릭스 반영

006번은 아래 개념을 섞어 낸 문제다.

- `@Valid` 누락과 DTO 검증 경계
- Entity 직접 Response 노출
- Optional `orElse`로 조회 실패를 임시 Entity 생성으로 숨기는 흐름
- 요청 userId/환불금액/상태값을 서버 상태처럼 신뢰하는 문제
- 주문 타입별 취소 정책 if 분기와 OCP
- 외부 환불 호출과 주문 상태 저장 순서
- 시간 기준 검증과 테스트 가능성
- Repository `save`가 생성/수정/중복 기준을 구분하지 않는 문제

23번 기준상 004/005와 완전히 다른 개념만 내는 것이 아니라, 반복 약점은 다시 섞되 코드 모양과 업무 흐름을 바꿔 출제한다.
006번은 "주문 취소"라는 상태 전이/환불 흐름으로 바뀌었고, Controller 중심이 아니라 Service/private helper/Repository/Util에 결함을 분산했기 때문에 004/005와 같은 도메인명만 바꾼 반복은 아니다.

## 남는 주의점

- 한 문제에 여러 개념이 들어 있으므로 사용자가 한 번에 다 찾기 어려울 수 있다.
- 채점 시에는 모든 항목을 동일 가중치로 보지 말고, 핵심 4개를 우선 채점한다.

우선 채점할 핵심 4개:

1. 조회 실패를 `orElse(createTemporaryOrder(request))`로 정상 흐름처럼 숨기는 문제
2. 요청 userId/환불금액/상태값을 서버 상태처럼 신뢰하는 문제
3. 주문 타입별 취소 정책 분기와 OCP
4. 외부 환불 호출과 주문 상태 저장 순서

보조로 볼 항목:

- Controller의 `@Valid` 누락과 Entity 직접 반환
- Repository `save`의 중복 저장 가능성
- 시간 기준 검증과 테스트 불안정성
- static Util과 외부 의존성 대체 어려움
- 테스트 실행/검증 부재

## 결론

006번은 **풀어도 되는 상태**다.

문제는 23번 출제 매트릭스를 참고해 여러 개념을 섞어 만든 것이 맞다.
다만 답변할 때는 모든 문제를 완벽히 찾으려 하기보다, 핵심 4개를 먼저 잡고 나머지를 보조로 추가하는 방식이 좋다.
