## 총평

005 문제는 현재 학습 루프 기준에 적합하다.

Java + Spring Boot 짧은 레거시 코드 템플릿이고, `Controller -> Service -> Repository -> Util` 흐름을 유지한다. 004번에서 드러난 약점도 다시 반영되어 있다.

## 적합한 부분

- 코드량이 40분 안에 리뷰 가능한 수준이다.
- Controller, Service, Repository, Util, DTO/Entity, Exception, Test가 모두 있다.
- JPA/QueryDSL/Liquibase/외부 API 장애 대응을 직접 문제로 넣지 않았다.
- README와 `review-answer.md`에 정답이나 힌트가 노출되지 않았다.
- IntelliJ용 `src/main/main.iml`, `src/test/test.iml`, `.idea/modules.xml` 등록이 되어 있다.
- `javac` 기준 main 코드 컴파일이 통과했다.

## 반영된 004 약점

- Service가 HTTP 응답용 `Map`을 반환한다.
- DTO validation이 없다.
- 조회 실패를 새 객체 생성으로 덮는다.
- 반복 Repository 조회가 있다.
- Util에 비즈니스 로직과 static 호출이 있다.
- `RuntimeException`을 직접 사용한다.
- 테스트가 실행 가능한 테스트 형태가 아니다.

## 섞인 문제 유형

- 명백한 안티패턴: 직접 `new` 생성, `RuntimeException`, 실행되지 않는 테스트
- 돌아가지만 위험한 코드: 조회 실패 시 새 객체 생성, 문자열 상태값, static mutable List
- 개선하면 좋은 코드: DTO validation, enum, 응답 DTO, Logger
- 트레이드오프 판단: 감사 로그 실패를 주소 변경 실패로 볼지 분리할지
- 리팩토링 방향 제안: 주소 검증 정책, 감사 로그, Repository 조건 조회 분리

## 부족하거나 위험한 부분

- Controller와 Service 양쪽에 직접 `new` 생성이 있어 DI 문제가 조금 중복된다.
- `save()`가 항상 `add()`라 중복 저장 문제가 매우 잘 보인다.
- `getChangeHistory()`에는 `userId == null`, `latest == null` 등 NPE 포인트가 많아 결함 밀도가 높은 편이다.
- 테스트는 `@Test`가 전혀 없어 "테스트 부실"보다 "테스트가 아니다"로만 보일 수 있다.
