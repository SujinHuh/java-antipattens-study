# 11. 코드 리뷰 면접 답변 템플릿

이 문서는 코드 분석 결과를 면접장에서 말할 수 있는 답변 순서로 정리하기 위한 답변 전용 템플릿이다.

먼저 전체 요청 흐름을 Controller -> Service -> Repository -> DTO/Entity -> Exception -> Test 순서로 파악했습니다.

제가 우선적으로 본 부분은 입력 검증, 비즈니스 로직 위치, 저장소 접근 방식, DTO/Entity 책임 분리, 예외 처리, 테스트 가능성입니다.

첫 번째로 Controller에서 요청 DTO 검증이 부족해 보입니다.
현재 구조에서는 잘못된 값이 Service 계층까지 들어갈 수 있기 때문에,
DTO에 `@NotNull`, `@NotBlank` 같은 검증을 추가하고 Controller에서 `@Valid`를 적용하는 것이 좋겠습니다.

두 번째로 Service 계층의 트랜잭션 경계가 명확하지 않습니다.
여러 DB 변경이 하나의 유스케이스로 묶여 있다면 Service 메서드에 `@Transactional`을 두고,
외부 API 호출과 DB 변경의 순서를 분리해서 실패 시 정합성이 깨지지 않도록 해야 합니다.

세 번째로 예외 처리가 공통화되어 있지 않습니다.
`RuntimeException`을 직접 던지기보다는 도메인별 커스텀 예외와 에러코드를 두고,
`GlobalExceptionHandler`에서 일관된 응답으로 내려주는 것이 좋겠습니다.

네 번째로 Repository 또는 조회 로직에서 성능 문제가 생길 수 있는 부분을 확인해야 합니다.
반복문 안에서 DB 조회가 발생하거나 전체 조회 후 Java에서 필터링한다면,
데이터가 늘어났을 때 N+1이나 메모리 사용 문제가 생길 수 있습니다.

다섯 번째로 DTO와 Entity의 책임이 섞여 있는지 확인해야 합니다.
요청 DTO가 응답이나 저장 상태까지 들고 있거나, Entity가 화면/API 요구사항에 직접 맞춰져 있다면
계층 간 변경 전파가 커질 수 있으므로 입력 DTO, 응답 DTO, 도메인/영속 Entity의 역할을 분리하는 것이 좋겠습니다.

마지막으로 이 코드는 테스트하기 어려운 구조도 일부 보입니다.
핵심 비즈니스 로직은 Service 단위 테스트로 분리하고,
Controller는 `MockMvc`로 요청 검증 실패와 성공 응답을 함께 검증하는 방식이 좋겠습니다.

## 압축 답변 구조

1. 전체 흐름을 먼저 말한다.
2. 우선순위 기준을 밝힌다.
3. Controller 검증 문제를 말한다.
4. Service 트랜잭션 문제를 말한다.
5. Repository 성능/조회 책임 문제를 말한다.
6. DTO/Entity 책임 분리 문제를 말한다.
7. 예외 처리 문제를 말한다.
8. 테스트 가능성 문제를 말한다.
9. 마지막에 리팩토링 우선순위를 정리한다.

## review-answer.md 문제 단위 누적 포맷

사용자가 특정 계층을 분석한 뒤 "정리해줘", "review-answer.md에 써줘", "기존 코드와 리팩토링 코드도 보여줘"라고 요청하면 아래 포맷을 유지한다.

목표는 정답만 적는 것이 아니라, 사용자가 면접장에서 말할 수 있게 `코드 위치 -> 판단 -> 왜 문제인지 -> 실무 영향 -> 개선 방향 -> 리팩토링 코드 -> 면접용 문장` 순서로 누적하는 것이다.

````markdown
#### 문제 N. 문제 제목

- 기존 코드:

```java
// 문제를 일으키는 핵심 코드 조각
```

- 판단: 이 코드가 왜 의심되는지 한 문장으로 적는다.
- 왜 문제인지: 계층 책임, 결합도, 테스트 가능성, 데이터 정합성, API 계약 중 어떤 기준에서 문제인지 설명한다.
- 실무 영향: 운영 장애, 협업 비용, 회귀 위험, 보안/정합성 문제로 어떻게 이어지는지 적는다.
- 개선 방향: 어느 계층/객체로 책임을 옮기고 어떤 기준으로 바꿀지 적는다.

- 리팩토링 코드:

```java
// 개선된 핵심 코드 조각
```

- 면접용 문장:

```text
면접장에서 20~30초 안에 말할 수 있는 문장으로 정리한다.
```
````

필요하면 아래 항목을 추가한다.

- `판단`: `Optional.empty()`처럼 그 자체는 문제가 아니고 처리 지점이 중요한 경우 반드시 넣는다.
- `진짜 문제 위치`: 문제가 발생한 계층과 실제 책임 계층이 다를 때 넣는다.
- `문제 있는 처리`: 잘못된 후속 코드 흐름을 별도로 보여준다.
- `개선된 테스트 관점`: 시간, 외부 API, static 의존성처럼 테스트 가능성이 핵심일 때 넣는다.
- `암기 포인트`: 사용자가 헷갈린 개념을 한 줄로 남긴다.

## 계층별 작성 순서

`review-answer.md`는 가능하면 아래 계층 순서를 유지한다.

```text
Controller -> Service -> Repository -> Util -> DTO/Entity -> Exception -> Test
```

Service나 Repository처럼 흐름이 중요한 계층은 문제를 쓰기 전에 아래 흐름을 먼저 떠올린다.

```text
입력값 -> 조회 -> 검증 -> 계산 -> 외부 호출 -> 저장 -> 결과 반환
```

## 코드 비교 원칙

- 실제 수정 요청이 있으면 해당 `src/main/java/...` 파일에는 기존 문제 코드를 주석으로 남기고, 리팩토링 코드를 활성 코드로 둔다.
- `review-answer.md`에는 전체 파일을 붙이지 않고 핵심 코드 조각만 남긴다.
- 정답이 하나라는 식으로 쓰지 않고, "이 문제에서는 이 개선 방향이 더 안전하다"는 면접 답변 형태로 쓴다.
