## 006 진행 로그

- 2026-06-30: Controller 계층을 함께 검토했다.
- 사용자는 Entity를 Controller 응답으로 그대로 반환하면 DTO/Entity 책임이 섞이고 API 스펙이 DB 구조에 강하게 결합된다는 점을 짚었다.
- 보완한 개념: `OrderCancelRequest`는 요청 DTO이고, 응답은 `OrderCancelResponse` 같은 별도 응답 DTO로 분리해야 한다.
- 추가 보완: Response DTO를 Controller 내부 임시 클래스로 두면 빠른 예시는 될 수 있지만, 면접 전 암기 패턴으로는 좋지 않다. 정석은 `OrderCancelRequest.java`와 `OrderCancelResponse.java`처럼 요청/응답 DTO를 별도 클래스로 분리하는 것이다.
- 추가로 보완한 개념: Controller는 성공/실패 응답을 HTTP 상태코드로 명확히 매핑해야 하며, 성공 바디 있음은 `200`, 성공 바디 없음은 `204`, 요청값 오류는 `400`, 리소스 없음은 `404`, 상태 충돌은 `409`로 구분해 볼 수 있다.
- 코드 학습 방식: 기존 코드와 리팩토링 코드는 실제 `src/main/java/...` 파일 안에 주석 비교로 남긴다. 빠른 복습을 위해 `review-answer.md`에도 핵심 기존 코드/리팩토링 코드 조각과 문제점, 이유, 실무 영향, 개선 방향을 함께 누적한다.
- 2026-06-30: Service 계층 일부를 함께 검토했다.
- 사용자는 `RuntimeException`과 `OrderCancelException`이 섞여 예외 처리가 일관되지 않고, 문자열 메시지만 있어 에러 코드/HTTP 매핑이 어렵다는 점을 짚었다.
- 사용자는 `NORMAL`, `PREORDER`, `DIGITAL` 타입별 if 분기가 OCP에 약하고 enum 또는 정책 분리로 개선할 수 있다고 짚었다.
- 사용자는 Service가 `order.status`, `order.cancelledAt`, `order.cancelReason`을 직접 바꾸는 것이 맞는지 질문했다. 보완한 개념: Builder는 새 객체 생성에 적합하고, 기존 Entity의 상태 전이는 `order.cancel(reason, now)` 같은 도메인 메서드로 캡슐화하는 것이 더 적절하다.
- 사용자는 외부 PG 환불 API 호출과 DB 저장 순서가 트랜잭션에서 왜 문제가 되는지 질문했다. 보완한 개념: `@Transactional`은 DB 변경만 롤백할 수 있고 외부 PG API 호출은 롤백할 수 없다. 따라서 먼저 DB에 `CANCEL_REQUESTED` 같은 요청 상태를 저장하고, 외부 API 호출 후 성공/실패 결과를 다시 DB에 기록하는 흐름이 더 안전하다.
- Service에서 다시 봐야 할 개념으로 `requestedRefundAmount` 신뢰 문제, `requestedStatus` 신뢰 문제, 요청 userId와 주문 소유자 검증, `LocalDateTime.now()` 테스트 어려움, static Util 외부 의존성 문제를 남겼다.
- 2026-06-30: Repository 계층 일부를 함께 검토했다.
- 사용자는 `save()`가 `add()`만 하면 같은 id 데이터가 계속 쌓여 문제가 될 수 있다고 짚었다. 보완한 개념: 수정 유스케이스에서는 `Map.put(id, entity)`처럼 식별자 기준 덮어쓰기 또는 DB update가 필요하다.
- 사용자는 `Optional.empty()` 처리가 아리송하다고 말했다. 보완한 개념: Repository가 `Optional.empty()`를 반환하는 것은 괜찮고, Service에서 `orElseThrow()`로 조회 실패를 명확히 처리해야 한다. 문제는 `empty`를 임시 객체 생성으로 숨기는 것이다.
- 사용자는 개별 개념은 이해하지만, 요청값 -> 조회 -> 검증 -> 계산 -> 외부 호출 -> 저장으로 이어지는 로직 흐름에서 어느 지점이 위험한지 바로 보이는 감각이 부족하다고 말했다. 007번에서는 개념 태그보다 유스케이스 흐름을 따라가며 문제를 찾는 연습을 다시 넣는다.

## 반복 약점

- Controller 문제는 일부 감지했지만 Service/Repository/DTO/Entity/Exception/Test 쪽 문제 탐지가 부족했다.
- Repository를 단순 저장 로직으로만 보고 계층 책임 위반을 놓쳤다.
- 테스트가 실제로 실행 가능한지, 경계 케이스를 검증하는지 확인하지 못했다.
- DTO와 Entity 역할 구분이 아직 헷갈린다.
- `@Transactional` 위치와 self-invocation 동작 이해가 더 필요하다.
- Exception을 구체적인 예외 타입/에러 코드/공통 처리 관점으로 보는 연습이 필요하다.
- Repository 반복 조회와 조회 방식 문제를 놓쳤다.
- Util에 비즈니스 정책이 들어간 문제를 일부만 잡았다.
- Entity 직접 노출과 Service 응답 타입 문제를 더 연습해야 한다.
- 개념명은 알아도 Service 유스케이스 흐름 속에서 조회 실패 처리, 요청값 신뢰, 외부 호출 순서, 저장 의미를 연결해 보는 연습이 더 필요하다.

## 정확한 표현으로 고친 문장

- Controller는 HTTP 요청/응답과 입력 검증에 집중하고, 비즈니스 정책 판단은 Service로 분리하는 것이 좋다.
- Repository는 저장소 접근을 담당해야 하며, `finalPrice == 0` 같은 도메인 정책 검증을 넣으면 계층 책임이 섞인다.
- DTO는 요청/응답 데이터를 운반하는 객체이고, Entity는 DB에 저장되는 도메인 상태를 표현하는 객체다.
- `@Transactional`은 보통 Service의 유스케이스 경계에 두며, 같은 클래스 내부 호출은 Spring AOP 프록시를 거치지 않는다.
- Entity를 요청/응답에 직접 사용하면 API 모델과 저장 모델이 강하게 결합된다.
- public field는 캡슐화가 깨지고 상태 변경을 통제하기 어렵다.
- 반복문 안 Repository 조회는 실제 DB 환경에서 N+1과 유사한 성능 문제로 이어질 수 있다.

     - **동시성 제어**: 공유 자원 경쟁 상태(Race Condition)를 보여주고, 동시성 유실을 제어하기 위한 동기화(Synchronized, ConcurrentHashMap, DB Lock) 고민 유도.
     - **예외 전파와 트랜잭션 롤백**: Checked Exception에 대한 `@Transactional` 롤백 동작 누락, 혹은 비즈니스 예외 발생 시 의도치 않은 하위 트랜잭션 전파 예외(`UnexpectedRollbackException`) 유발.
- **난이도 및 분량**: 초급-중급에서 실무 시나리오 기반 중급으로 조정하고, 30~40분 내에 리뷰 가능한 콤팩트한 코드를 유지한다.
- **회사 면접 우선순위**: 40분 이내 리뷰 가능한 레거시 코드 내에서, 단순한 '정답 찾기'를 넘어 "두 방식 간의 설계적 트레이드오프"를 설명할 수 있도록 출제한다.
