# Legacy Code Review Interview Rules

## 먼저 볼 핵심 기준

문제 풀이 직후에는 모든 문서를 한 번에 다 보려고 하지 않는다.

우선 아래 5개만 먼저 확인한다.

1. Controller가 Entity를 직접 받거나 반환하는지
2. Controller에 비즈니스 판단이 들어갔는지
3. Service에서 상태 변경/저장이 있는데 트랜잭션 경계가 보이는지
4. 반복문 안에서 Repository 조회가 발생하는지
5. `RuntimeException`, `null`, 문자열 상태값처럼 장애 원인을 숨기는 코드가 있는지

003번처럼 `review-answer.md`가 길어지면 처음부터 끝까지 읽지 않는다.

처음 복습할 때는 아래만 먼저 본다.

1. `review-answer.md`의 `핵심 문제 요약`
2. `review-answer.md`의 `채점`
3. `review-answer.md`의 `반드시 잡았어야 하는 문제`
4. `review-answer.md`의 `문제 유형 분류`
5. `review-answer.md`의 `안티패턴 지침화 후보`
6. `review-answer.md`의 `시니어 개발자의 피드백 (면접관 관점)`

`solution/`과 `refactoring-plan.md`는 1차 복습 뒤에 본다.

목표는 모든 개념을 한 번에 외우는 것이 아니라, 짧은 레거시 코드에서 반복적으로 나오는 핵심 문제를 먼저 안정적으로 찾는 것이다.

## 면접관 기준 채점

답변은 학습자에게 친절하게만 평가하지 말고, 실제 회사 1차 기술면접에서 면접관이 듣는 기준으로 평가한다.

사용자는 면접자이고, Codex는 면접관이다. 답변을 볼 때는 "공부 중인 사람을 격려하는 기준"보다 "이 답변을 실제 면접장에서 들었을 때 합격권인지"를 우선한다.

- 표면적인 문제만 찾으면 낮게 평가한다.
- Controller, Service, Repository, Util 각 계층의 책임 위반을 분리해서 본다.
- "잘 모르겠다", "크게 문제 없는 것 같다"에서 멈춘 부분은 감점한다.
- 놓친 핵심 문제는 "반드시 잡았어야 하는 문제"로 분리한다.
- 가장 중요한 문제는 `핵심 문제 요약`에서 먼저 강조한다.
- 각 문제는 `명백한 안티패턴`, `돌아가지만 위험한 코드`, `개선하면 좋은 코드`, `트레이드오프 판단`, `리팩토링 방향 제안` 중 하나 이상으로 분류한다.
- 반복해서 나온 안티패턴은 `안티패턴 지침화 후보`로 분리해 사용자가 빠르게 누적 패턴을 볼 수 있게 한다.
- 점수는 합격권 기준으로 엄격하게 준다.
- 표현이 모호하면 면접에서 말할 수 있는 정확한 문장으로 고친다.

## 합격권 답변 기준

짧은 레거시 코드 템플릿에서 아래를 빠르게 찾아야 한다.

- 요청 검증 누락, `@Valid` 누락, DTO validation 누락
- Entity를 Request/Response에 직접 사용
- Controller에 비즈니스 로직 존재
- Service 트랜잭션 경계 누락
- Repository 반복 조회, 비효율 조회, null 반환
- RuntimeException 직접 사용, 예외 정책 부재
- 부적절한 HTTP 상태 코드
- 문자열 상태값 하드코딩
- Util에 비즈니스 로직 포함
- 테스트하기 어려운 구조와 실행되지 않는 테스트
- 로그 처리 부족

## 학습 부담 조절 기준

한 번에 모든 개념을 완벽하게 보려고 하지 않는다. 이 문서는 면접관 기준 전체 범위를 정리한 것이고, 사용자가 매 회차마다 전부 외워야 한다는 뜻이 아니다.

문제 풀이 직후에는 아래 순서로만 먼저 본다.

1. `review-answer.md`의 핵심 문제 요약
2. `review-answer.md`의 채점과 반드시 잡았어야 하는 문제
3. `review-answer.md`의 문제 유형 분류와 안티패턴 지침화 후보
4. `시니어 개발자의 피드백 (면접관 관점)`의 부족하게 들리는 지점
5. `이번 문제 개념 키워드`의 면접용 문장
6. `면접관 예상 꼬리질문` 중 내 답변에서 바로 이어질 질문
7. `til.md`의 키워드별 개념 설명

`solution/`, `refactoring-plan.md`, 깊은 꼬리질문은 1차 복습 뒤에 본다.

처음에는 아래 5개만 안정적으로 잡는 것을 목표로 한다.

- Controller가 Entity를 직접 받거나 반환하는지
- Controller에 비즈니스 판단이 들어갔는지
- Service에서 상태 변경과 저장이 있는데 트랜잭션 경계가 보이는지
- 반복문 안에서 Repository 조회가 발생하는지
- RuntimeException, null 반환, 문자열 상태값처럼 장애 원인을 숨기는 코드가 있는지

이 5개를 먼저 잡을 수 있으면, 그 다음에 DTO validation, Util 책임, 테스트 가능성, 로그, 트레이드오프를 넓혀간다.

문제 코드는 실제 Spring Boot 프로젝트를 그대로 실행하는 목적이 아니라, 면접장에서 짧은 시간에 리뷰하기 위한 축약 코드일 수 있다. 이 경우 `@Controller`, `@Service`, `@Repository`, `@Transactional` 등은 주석으로 표시해도 되지만, 피드백에서는 실제 Spring 코드라면 어떤 어노테이션과 책임 배치가 필요한지 설명한다.

## 풀이 후 자료

답변과 채점이 끝난 뒤에는 비교 학습을 위해 아래를 만든다.

- `refactoring-plan.md`: 개선 순서, 유지할 동작, 트레이드오프
- `solution/`: 가능한 개선 코드 예시
- `review-answer.md`: 사용자 답변, 면접관 기준 채점, 개선 답변, 개념 키워드 설명, 예상 꼬리질문, 시니어 개발자의 피드백
- `til.md`: 문제에서 나온 키워드와 키워드별 개념 설명
- 각 문제의 `README.md`와 `review-answer.md`에는 리팩토링된 코드 위치를 명확히 적는다.
- 리팩토링 코드 위치 예: `refactoring-plan.md`, `solution/README.md`, `solution/src/main/java/...`, `solution/src/test/java/...`
- 개선 코드 이름 규칙: `기존이름_Refactored`
- 개선 코드 주석 규칙: 핵심 변경 지점에 `기존 코드:` / `수정 코드:` 비교 주석 작성

`solution/`은 정답이 하나라는 뜻이 아니라, 면접에서 설명 가능한 개선 방향의 예시다.

## review-answer.md 핵심 강조 규칙

`review-answer.md`에서는 긴 설명보다 핵심 문제를 먼저 보이게 한다.

반드시 아래 섹션을 상단부에 둔다.

1. `핵심 문제 요약`
   - 면접에서 놓치면 크게 감점될 문제 3~5개
   - 문제 유형
   - 코드 위치
   - 왜 핵심인지
   - 면접용 한 문장

2. `반드시 잡았어야 하는 문제`
   - 문제별로 `문제`, `유형`, `코드 위치`, `왜 문제인지`, `실무 영향`, `면접용 문장`을 적는다.

3. `문제 유형 분류`
   - `명백한 안티패턴`
   - `돌아가지만 위험한 코드`
   - `개선하면 좋은 코드`
   - `트레이드오프 판단`
   - `리팩토링 방향 제안`

4. `안티패턴 지침화 후보`
   - `7. 안티패턴 지침화 에이전트.md` 기준으로 정리한다.
   - 사용자가 "이런 유형의 문제가 반복되는구나"를 빠르게 알 수 있어야 한다.

## 완료 후 에이전트 확인

사용자가 문제 풀이를 완료했다고 말하면, Codex는 피드백을 작성하기 전에 docs에서 필요한 에이전트 기준을 확인한다.

- 기본 채점: `2. 답변 채점 에이전트.md`
- 문제 의도와 다음 회차 보강: `1. 문제 생성 에이전트.md`, `17. 루프 상태 기록.md`
- 교차검증 필요 판단: `15. AI 교차검증 실행 프롬프트.md`
- 코드, 답변, 개선 코드 통합 검토: `16. 통합 코드리뷰-리팩토링 코치.md`
- 반복 안티패턴 지침화: `7. 안티패턴 지침화 에이전트.md`

필요한 모든 에이전트를 항상 실행한다는 뜻은 아니다. 문제 성격과 사용자의 답변에 맞는 기준을 꺼내 적용하고, 중요한 판단이 애매할 때만 교차검증을 요청한다.

## 개념 키워드 기록

각 문제의 피드백에는 아래를 반드시 포함한다.

- 이번 문제에 숨긴 키워드
- 사용자가 잡은 키워드와 놓친 키워드
- 키워드별 짧은 개념 설명
- 해당 키워드가 코드의 어느 위치에서 드러났는지
- 면접에서 사용할 수 있는 한 문장

예: `@Transactional`을 문제에 넣었다면 단순히 "트랜잭션 누락"이라고만 쓰지 말고, 상태 변경과 저장이 하나의 유스케이스로 묶여야 하는 이유, Service 계층에 트랜잭션 경계를 두는 이유, AOP 프록시/self-invocation과 연결될 수 있는 꼬리질문까지 기록한다.

## 면접관 예상 꼬리질문 기록

`review-answer.md`에는 사용자의 답변을 기준으로 실제 면접관이 이어서 물을 질문을 남긴다.

- 답변에서 바로 파생되는 질문
- 놓친 개념을 확인하는 질문
- 모호한 표현을 정확한 용어로 바꾸게 하는 질문
- 질문 의도
- 20~30초 답변 방향

예: 사용자가 "`@Transactional`이 있어야 한다"라고만 말했다면, "왜 Controller가 아니라 Service에 트랜잭션 경계를 두나요?", "self-invocation 상황에서는 `@Transactional`이 어떻게 동작하나요?", "읽기 전용 조회와 상태 변경 메서드의 트랜잭션 설정을 다르게 둘 수 있나요?"처럼 이어서 묻는다.

## 시니어 개발자의 피드백 (면접관 관점)

`review-answer.md`에는 아래 섹션을 유지한다.

```md
## 시니어 개발자의 피드백 (면접관 관점)

### 인정되는 답변

### 실제 면접에서 부족하게 들리는 지점

### 다음 리뷰에서 먼저 확인할 것
```

이 섹션은 칭찬을 위한 공간이 아니라, 실제 면접관이 답변을 들었을 때의 총평을 남기는 공간이다. 인정할 부분은 인정하되, 합격권 기준에서 부족한 부분을 구체적으로 적는다.
