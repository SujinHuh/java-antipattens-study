# 25. 수요일 20회 반복 목표

이 문서는 수요일(2026-07-01)까지 020번 문제까지 푸는 것을 목표로, 문제 출제 속도와 피드백 깊이를 조절하기 위한 운영 문서다.

## 목표

- 현재 005번 이후부터 020번까지 최대한 많은 짧은 Java/Spring Boot 레거시 코드 리뷰를 반복한다.
- 목표는 모든 개념을 한 번에 완벽히 외우는 것이 아니라, 면접에서 짧은 코드의 문제를 빠르게 찾고 근거를 말하는 것이다.
- 답변 후에는 반드시 "왜 문제인지 -> 실무 장애 -> 어느 계층에서 어떻게 고칠지 -> 면접용 문장"까지 정리한다.

## 006~015 운영 방식

006~015는 회차별로 주제를 딱 나누지 않는다.
개념 후보는 `23. 개념별 출제 확장 매트릭스.md`를 기준으로 고르고, 아래 목록은 고정 범위가 아니라 우선적으로 자주 섞을 예시다.
매 문제마다 `23`에서 기본 3~5개를 골라 섞되, 006~015 빠른 반복 구간에서는 핵심 리뷰 포인트를 6~7개까지 허용한다.
아래 목록에 없는 개념도 `23`에 있거나 직전 채점에서 드러난 약점이면 출제할 수 있다.

- Controller 책임, 상태코드, Response DTO, `Map<String, Object>` 응답
- Service 책임, Transaction Script, HTTP 응답 책임 침범
- Repository 조건 조회, `null` 반환, 내부 컬렉션 노출, 반복 조회
- Optional 오남용, `optional.get()`, `orElse` vs `orElseGet`
- 예외 흐름, `RuntimeException`, 400/404/409 구분, 공통 예외 매핑
- DTO/Entity 분리, 서버 결정값을 클라이언트 입력으로 받는 문제
- Validation, `@Valid`, Bean Validation으로 부족한 도메인 검증
- SOLID 중 SRP/OCP/DIP
- Transaction boundary, readOnly, 외부 호출과 트랜잭션 순서
- 권한/소유자 검증, 요청 userId 신뢰 문제
- 시간/상태 변경, 서버 시간 기준 만료/취소 가능 시간
- Util/Policy/Component 경계
- 테스트 가능성, 실패 케이스 누락, static 저장소 테스트 격리 문제
- `23`의 그 외 개념: Java record/DTO 설계, Mapper/변환 책임, AOP/self-invocation, 문자열 상태값, Entity responsibility, Logging/Monitoring, 협업/API 계약, 리팩토링 우선순위 등

## 016~020 운영 방식

016~020은 면접 실전 압축으로 운영한다.

- 10~15분 안에 핵심 문제 5개를 먼저 찾는다.
- 30초 답변과 1분 답변을 바로 만든다.
- 꼬리질문에 대비해 정확한 용어로 다시 말한다.
- 새로운 개념을 많이 늘리기보다 006~015에서 반복해서 틀린 개념을 재출제한다.
- 문제 코드와 직접 연결된 깊은 이론은 짧게라도 반드시 확인한다.

## 문제 생성 기준

- 한 문제의 핵심 결함은 기본 3~5개로 제한한다. 다만 006~015 빠른 반복 구간에서는 반복 약점 2개, 덜 본 개념 2개, 실무 장애/트레이드오프 1개를 포함해 핵심 리뷰 포인트를 6~7개까지 허용한다.
- 출제 개념은 항상 `23. 개념별 출제 확장 매트릭스.md`에서 고르고, 이 문서의 006~015 목록은 빠른 반복을 위한 우선순위 예시로만 본다.
- 같은 개념을 다시 내더라도 코드 모양, 계층, 도메인, 장애 영향을 바꾼다.
- 답변 전에는 정답, 힌트, 개념 태그를 공개하지 않는다.
- 사용자가 힌트를 요청하면 `21. 힌트 제공 가이드.md`와 `22. 계층별 반복 패턴 체크리스트.md` 기준으로 해당 계층만 도와준다.
- 문제 생성 후에는 `23. 개념별 출제 확장 매트릭스.md`와 `17. 루프 상태 기록.md`를 갱신한다.
