# 12. 에이전트 호출 판단 체크리스트

이 문서는 작업 중 **어떤 에이전트 문서를 같이 봐야 하는지** 빠르게 고르는 라우팅 문서다.

상세한 운영 흐름은 `20. 에이전트 운영 상세 가이드.md`에 분리한다.
이 문서는 길게 읽는 문서가 아니다.

## 먼저 볼 복습 순서

`review-answer.md`가 길면 아래 순서만 먼저 본다.

1. `핵심 문제 요약`
2. `채점`
3. `반드시 잡았어야 하는 문제`
4. `문제 유형 분류`
5. `안티패턴 지침화 후보`
6. `시니어 개발자의 피드백 (면접관 관점)`

`solution/`과 `refactoring-plan.md`는 1차 복습 뒤에 본다.

## 새 문제 생성 시 문서 확인 순서

새 문제를 만들 때는 아래 순서로 문서를 확인한다.

1. `12. 에이전트 호출 판단 체크리스트.md`
   - 지금 작업이 새 문제 생성인지, 풀이 완료 피드백인지 먼저 구분한다.
2. `17. 루프 상태 기록.md`
   - 현재 회차, 직전 약점, 다음 회차 필수/보조 주제를 확인한다.
3. `25. 수요일 20회 반복 목표.md`
   - 수요일까지 020번 문제 목표, 006~015 혼합 출제, 016~020 실전 압축 기준을 확인한다.
4. `23. 개념별 출제 확장 매트릭스.md`
   - 이미 나온 방식, 보충해야 할 방식, 변형 축, 과밀 방지 기준을 확인한다.
5. `1. 문제 생성 에이전트.md`
   - 문제 구조, Java/Spring Boot 고정, IntelliJ 호환성, README 힌트 제한을 확인한다.
6. `13. 실제 루프 실행 프롬프트.md`
   - 실제 루프 운영 방식, 답변 전 정답 비공개, 문제 생성 후 검수 기준을 확인한다.
7. 필요 시 `19. 헤렌 공고 기반 docs 보강 예정.md`
   - 회사 공고 관점이 필요한 경우에만 참고한다.

새 문제 생성 후에는 `23`의 `이미 나온 방식`을 반드시 갱신하고, `17`의 다음 회차 지시도 함께 갱신한다.

## 풀이 완료 시 고정 순서

사용자가 "작성했어", "다 풀었어", "완료했어"라고 말하면 아래 순서로 진행한다.

1. `2`로 실제 면접관 기준 채점과 `review-answer.md` 작성
2. `15`로 `review-answer.md` 교차검증 및 `검수결과.md` 작성
3. `3`으로 놓친 개념 설명 보강
4. `7`로 안티패턴 지침화 후보 정리
5. `antipattern-kit/java-spring-coding-guardrails.md` 반영 여부 판단
6. `23`으로 이미 나온 방식, 보충해야 할 방식, 현재 약점 갱신
7. `17`로 누적 약점과 다음 문제 상태 업데이트
8. `4`, `5`로 문제 품질 회고와 다음 문제 조건 보강
9. `6`으로 TIL 기록

## 산출물 구분

- `문제검수결과.md`: 새 문제 출제 직후 문제 품질과 개념 확장 여부를 검수한 결과
- `검수결과.md`: `review-answer.md` 작성 후 기술 오류, 위험 표현, 빠진 관점을 교차검증한 결과
- `antipattern-kit/java-spring-coding-guardrails.md`: 반복 안티패턴을 실제 코딩 가드레일로 승격한 결과
- `답변후-개념공부.md`: 답변 전에는 보지 않는 복습 자료. 답변 후 이번 문제 개념과 관련 docs 링크를 정리한 결과

## 주의

- 모든 에이전트를 항상 길게 실행하지 않는다.
- 답변 전에는 사용자의 요청 없이 정답, 힌트, 개선 코드를 공개하지 않는다.
- 사용자가 풀이 중 힌트를 요청하면 `21` 기준으로 단계별 힌트만 제공한다.
- 피드백 문서에서는 `핵심 문제 요약`, `문제 유형 분류`, `안티패턴 지침화 후보`를 빠뜨리지 않는다.
- 새 문제를 낸 직후에는 문제 품질 검수 결과를 `문제검수결과.md`로 남긴다.
- `review-answer.md` 작성 후에는 `15` 기준으로 교차검증 결과를 남긴다.
- 회사 공고 관점은 `19`에 후보로 관리하고, 초반 문제에는 과하게 섞지 않는다.
