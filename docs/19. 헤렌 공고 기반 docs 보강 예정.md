# 19. 헤렌 공고 기반 docs 보강 예정

이 문서는 헤렌 백엔드 채용공고를 기준으로, 현재 학습 루프 docs에 이미 반영된 내용과 앞으로 보강해야 할 내용을 분리해서 기록한다.

출처: https://www.wanted.co.kr/wd/360821

## 이미 반영된 방향

현재 docs에는 아래 기준이 이미 반영되어 있다.

1. Java + Spring Boot 기반 문제로 고정
2. 짧은 레거시 코드 템플릿을 빠르게 리뷰하는 방식
3. Controller -> Service -> Repository -> Util 중심 구조
4. RDBMS 관점은 JPA 사용법보다 트랜잭션, 격리수준, 인덱스, 실행계획을 의심하는 코드 리뷰로 반영
5. PO, FE, QA 협업 관점에서 API 계약, 상태 코드, 예외 메시지, 테스트 시나리오를 피드백에 포함
6. AI 도구를 사용했다는 말보다 본인이 코드의 의도와 판단 근거를 설명하도록 훈련
7. JPA/QueryDSL/Liquibase는 회사 기술 스택에는 있으나 1~8회차에는 직접 출제하지 않는다. 009에서는 확장 여부를 사용자에게 제안만 하고, 실제 JPA/QueryDSL/Liquibase 문제는 사용자가 동의한 뒤 010 이후 별도 트랙으로 시작한다.

## 추가 보강이 필요한 부분

공고에는 현재 루프에 더 보강하면 좋은 관점이 있다.

### 1. 비즈니스와 사용자 경험 관점

공고에서는 기술적 선택이 사용자 경험과 비즈니스 지표에 어떤 영향을 주는지 설명할 수 있는 사람을 기대한다.

앞으로 피드백에는 아래 질문을 추가한다.

- 이 코드 문제가 실제 사용자 경험에 어떤 영향을 주는가?
- 장애나 오류가 PO/CS/QA에게 어떻게 전달될 수 있는가?
- 상태 코드, 에러 메시지, 응답 형태가 제품 운영에 어떤 비용을 만드는가?
- 기술적 개선이 비즈니스 지표나 운영 효율과 어떻게 연결되는가?

### 2. 외부 API와 데이터 파이프라인 관점

공고에는 외부 SNS/인플루언서 마케팅 플랫폼 API 연동과 데이터 수집/정제 파이프라인 운영이 포함되어 있다.

다만 1~8회차 문제에는 외부 API 장애 대응을 직접 구현 주제로 넣지 않는다.
초반에는 아래 정도를 꼬리질문이나 피드백 보조 관점으로만 다룬다.

- 외부 API 응답을 그대로 신뢰하는 코드
- 실패/timeout/null 응답을 구분하지 않는 코드
- 재시도 가능 오류와 비재시도 오류를 구분하지 않는 코드
- 수집 데이터 검증 없이 저장하는 코드
- 운영자가 추적할 수 없는 로그와 에러 메시지

010 이후에는 사용자가 동의한 경우 별도 문제로 확장할 수 있다.

### 3. 모니터링과 운영 지표 관점

공고에는 Sentry/Prometheus 기반 모니터링과 운영 지표 정의가 포함되어 있다.

초반 문제 피드백에는 구현 기술명을 직접 묻기보다 아래 관점을 추가한다.

- 실패 로그에 추적 가능한 식별자가 있는가?
- 예외가 삼켜져서 운영자가 원인을 알 수 없게 되는가?
- 성공/실패/지연/중복 처리 같은 운영 지표를 남길 수 있는 구조인가?
- 장애 발생 시 어느 계층에서 원인을 추적할 수 있는가?

### 4. 테스트 자동화 관점

공고에는 JUnit 5, TestContainers, 테스트 자동화가 포함되어 있다.

초반 문제에서는 TestContainers를 직접 요구하지 않는다.
대신 아래를 먼저 본다.

- 테스트가 실제로 실행 가능한 형태인가?
- 성공 케이스만 있고 실패/경계 케이스가 빠져 있지 않은가?
- 저장소, 시간, 외부 통신 때문에 테스트가 불안정하지 않은가?
- 리팩토링 후에도 기존 동작을 보존하는 테스트가 있는가?

010 이후 JPA/RDBMS 문제로 확장할 때 TestContainers를 별도 학습 후보로 둔다.

### 5. 글로벌 확장과 안정성 관점

공고에는 글로벌 서비스 확장과 시스템 안정성이 강조되어 있다.

초반 문제에서는 대규모 분산 시스템 설계로 바로 확장하지 않는다.
대신 코드 리뷰 피드백에서 아래 질문을 덧붙인다.

- 문자열 상태값, 날짜 문자열, 하드코딩 메시지가 다국어/타임존/정책 변경에 취약하지 않은가?
- 장애가 났을 때 부분 실패를 설명할 수 있는가?
- 데이터 정합성을 지키기 위한 트랜잭션 경계가 보이는가?
- 단순한 구조를 유지하면서도 변경 가능성을 열어두었는가?

## docs 반영 예정 위치

이 문서의 내용을 실제 운영 규칙으로 반영할 때는 아래 파일을 수정한다.

1. `docs/1. 문제 생성 에이전트.md`
   - 문제 생성 기준에 비즈니스 영향, 운영 지표, 외부 API/데이터 파이프라인 관점을 추가한다.

2. `docs/2. 답변 채점 에이전트.md`
   - 채점 기준에 사용자 경험, 협업 비용, 운영 추적 가능성을 추가한다.

3. `docs/3. 설명 에이전트.md`
   - 설명 기준에 "왜 비즈니스/운영에서 문제가 되는지"를 추가한다.

4. `docs/13. 실제 루프 실행 프롬프트.md`
   - 실제 루프 피드백에 공고 기반 추가 질문을 반영한다.

5. `docs/17. 루프 상태 기록.md`
   - 010 이후 JPA/QueryDSL/Liquibase/TestContainers/외부 API 문제 확장 후보를 기록한다.

## 적용 우선순위

### 지금 바로 적용할 것

- 코드 리뷰 피드백에서 비즈니스 영향, 사용자 경험, 협업 비용을 한 줄이라도 묻기
- 로그와 예외를 운영자가 추적할 수 있는지 확인하기
- 테스트가 실제 실행 가능하고 실패 케이스를 포함하는지 확인하기

### 5~8회차에 보조 관점으로 넣을 것

- 외부 API 실패를 직접 구현하지는 않되, 응답 신뢰/에러 구분/로그 추적 관점을 꼬리질문으로 넣기
- 글로벌 확장 관점에서 문자열 날짜, 문자열 상태값, 하드코딩 메시지의 위험을 묻기

### 010 이후 별도 확장할 것

- JPA
- QueryDSL
- Liquibase
- TestContainers
- 외부 API 연동 실패 처리
- 모니터링 지표 설계

## 면접 답변에 추가할 관점

면접에서는 단순히 "이 코드는 나쁩니다"에서 끝내지 않는다.

아래처럼 말하는 연습을 한다.

```text
이 문제는 코드 스타일 문제만이 아니라, 운영과 협업 비용으로 이어질 수 있습니다.
예를 들어 예외가 RuntimeException 문자열로만 처리되면 FE/QA는 실패 원인을 구분하기 어렵고,
운영자는 Sentry나 로그에서 어떤 사용자, 어떤 요청, 어떤 상태에서 실패했는지 추적하기 어렵습니다.
그래서 DTO 검증, 구체적인 에러 코드, 트랜잭션 경계, 실패 케이스 테스트를 같이 보강해야 합니다.
```
