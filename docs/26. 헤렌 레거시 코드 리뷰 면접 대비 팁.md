# 헤렌 1차 기술면접 대비: 레거시 코드 리뷰 준비 팁

## 1. 면접에서 주어지는 코드의 성격

헤렌 1차 기술면접에서 보는 코드는 실제 운영 서비스의 거대한 레거시 코드 전체라기보다는, 예전에 사용했던 면접용 레거시 코드 템플릿에 가깝다.
코드 양은 많지 않고, 컨트롤러에서 서비스, 레포지토리, 유틸까지 이어지는 단순한 흐름의 코드라고 이해하면 된다. 다만 코드가 짧다고 해서 쉬운 문제라는 뜻은 아니다. 코드 안에는 개발자로서 하면 안 되는 코드 습관들이 의도적으로 많이 들어가 있고, 그중 몇 개를 찾고 설명할 수 있는지가 중요하다.
따라서 준비 방향은 대규모 프로젝트를 분석하는 연습이 아니라, 짧은 레거시 코드에서 문제점을 빠르게 찾고 말로 설명하는 연습이다.

---

## 2. 코드 보는 시간과 접근 방식

코드 보는 시간은 1시간 이상 넉넉하게 주어지는 방식은 아니고, 약 40분 정도라고 생각하면 된다.
그래서 처음부터 모든 코드를 완벽하게 이해하려고 하면 오히려 시간이 부족할 수 있다. 먼저 전체 요청 흐름을 잡고, 그다음 계층별로 문제를 찾아야 한다.

추천하는 읽기 순서는 다음과 같다.
```text
Controller → Service → Repository → Util
```

처음 코드를 받으면 바로 문제점을 지적하기보다, 아래처럼 말하면서 시작하는 것이 좋다.
> "먼저 전체 요청 흐름을 Controller → Service → Repository → Util 순서로 파악하고, 입력 검증, 트랜잭션 경계, 예외 처리, 계층별 책임 분리, 테스트 가능성 순서로 보겠습니다."

이렇게 말하면 처음 보는 코드 앞에서도 기준을 가지고 보고 있다는 인상을 줄 수 있다.

---

## 3. 이해가 안 되는 부분은 오래 고민하지 말고 질문하기

이해가 안 되는 부분을 혼자 오래 붙잡고 있지 말라. 면접관에게 질문해도 되며, 오히려 도메인 의도나 코드 의도가 불명확한 부분은 질문하면서 파악하는 편이 좋다.

면접에서 사용할 수 있는 질문 예시는 다음과 같다.
- "이 상태값이 어떤 라이프사이클을 갖는지 확인해도 될까요?"
- "이 로직은 실패했을 때 전체 요청을 롤백해야 하는 구조인가요, 아니면 일부 실패만 분리해서 처리해도 되는 구조인가요?"
- "이 유틸 로직이 여러 도메인에서 공통으로 쓰이는 로직인지, 아니면 특정 도메인의 비즈니스 규칙인지 확인하고 싶습니다."

중요한 것은 "모르겠습니다"에서 끝나는 게 아니라, 확인 질문을 통해 코드 리뷰 방향을 잡는 것이다.

## 4. 코드 리뷰 방식으로 답변하기

이 면접은 정답 코드를 바로 작성하는 시험이라기보다, 코드 리뷰 방식에 가깝다.

- 컨트롤러 단 로직이라면 `@Valid` 같은 검증 어노테이션이 있어야 할 수 있고, 비즈니스 로직이라면 `@Transactional`이 필요한데 빠져 있는지 볼 수 있다.

따라서 답변은 단순히 "이거 빠졌습니다"가 아니라, 아래 흐름으로 말하는 것이 좋다.
```text
문제 위치 → 왜 위험한지 → 어떻게 개선할지 → 테스트는 어떻게 할지
```

**예시:**
> "Controller에서 요청값 검증이 부족해 보입니다. 현재 구조에서는 잘못된 값이 Service 계층까지 들어갈 수 있고, 그 결과 비즈니스 로직 안에서 예외가 발생하거나 잘못된 데이터가 저장될 수 있습니다. 따라서 Request DTO를 분리하고 필수값에는 @NotNull, @NotBlank 같은 검증을 추가한 뒤, Controller에서 @Valid로 검증하는 방식이 좋겠습니다. 테스트는 필수값 누락 시 400 응답이 내려가는지 확인하겠습니다."

---

## 5. 반드시 봐야 할 핵심 체크포인트

### 5.1 Controller
Controller에서는 요청과 응답의 경계를 본다.

**확인할 내용:**
- 요청값 검증이 있는가
- `@Valid`가 빠져 있지 않은가
- Request DTO와 Response DTO가 분리되어 있는가
- Entity를 직접 Request 또는 Response로 사용하고 있지 않은가
- Controller에 비즈니스 로직이 들어가 있지 않은가
- 실패 상황에서도 무조건 200 응답을 내려주고 있지 않은가

**면접 답변 예시:**
> "Controller에서 Entity를 직접 받고 있다면 외부 요청 구조와 내부 DB 모델이 강하게 묶일 수 있습니다. 요청 DTO와 응답 DTO를 분리해서 API 스펙과 내부 모델의 변경 영향을 분리하는 것이 좋겠습니다."

### 5.2 Service
Service에서는 비즈니스 책임과 트랜잭션 경계를 본다.

**확인할 내용:**
- `@Transactional`이 필요한데 빠져 있지 않은가
- 조회 메서드에 `readOnly = true`를 고려할 수 있는가
- 외부 API 호출과 DB 변경이 한 트랜잭션 안에 섞여 있지 않은가
- 중복 검증 로직이 값을 문자열로 직접 비교하고 있지 않은가
- 예외를 잡고도 흐름을 계속 진행하고 있지 않은가

**면접 답변 예시:**
> "이 Service 메서드는 여러 DB 변경을 하나의 유스케이스로 처리하는데, 트랜잭션 경계가 명확하지 않아 중간 실패 시 데이터 정합성이 깨질 수 있습니다. Service 계층에 @Transactional을 두고 실패 시 롤백 범위를 명확히 하는 것이 좋겠습니다."

### 5.3 Repository
Repository에서는 조회 방식과 성능 문제를 본다.

**확인할 내용:**
- 반복문 안에서 DB 조회가 발생하지 않는가
- 전체 데이터를 가져온 뒤 Java에서 필터링하고 있지 않은가
- 목록 조회에 페이징이 없는가
- 조건절에 자주 쓰이는 컬럼의 인덱스 고려가 필요한가
- 한 방 쿼리에 너무 많은 책임이 들어가 있지 않은가

**면접 답변 예시:**
> "반복문 안에서 Repository 조회가 발생하면 데이터가 늘어났을 때 N+1 문제가 생길 수 있습니다. 필요한 데이터를 한 번에 조회하거나 fetch join, IN 쿼리, batch 조회 방식으로 개선하는 것이 좋겠습니다."

### 5.4 Util
Util에서는 공통 기능과 비즈니스 로직이 섞였는지 본다.

**확인할 내용:**
- Util에 특정 도메인 규칙이 들어가 있지 않은가
- 단순 포맷 변환이 아니라 상태 판단이나 결제/예약 규칙 같은 비즈니스 로직을 처리하고 있지 않은가
- 테스트하기 어려운 static 메서드에 핵심 로직이 몰려 있지 않은가

**면접 답변 예시:**
> "이 Util은 단순 변환 로직이 아니라 특정 도메인의 판단 기준을 포함하고 있습니다. 이 경우 공통 유틸로 두기보다는 도메인 서비스나 별도 Validator로 분리하는 것이 변경 영향 관리에 더 좋겠습니다."

## 6. "몇 개를 찾느냐"가 중요 핵심

코드 템플릿 안에 개발자로서 하면 안 되는 것들이 많이 들어 있고, 결국 몇 개를 찾느냐가 중요하다. 연습할 때는 한 문제에서 최소 10개 정도의 문제점을 찾는 것을 목표로 한다. 

다만 단순히 문제를 많이 나열하는 것보다 중요한 것은 우선순위다.

**우선순위 가이드:**
1. 데이터 정합성이 깨질 수 있는 문제 (가장 중요)
2. 장애나 운영 이슈로 이어질 수 있는 문제
3. 보안 또는 개인정보 노출 가능성이 있는 문제
4. 성능 저하 가능성이 있는 문제
5. 유지보수와 테스트를 어렵게 만드는 문제
6. 코드 스타일이나 가독성 문제

---

## 7. 코드 템플릿만 준비하면 안 된다

코드 템플릿 문제도 중요하지만, 이력서 기반 질문도 까다롭게 들어올 수 있다. 레거시 코드 리뷰 연습과 함께 내가 이력서에 적은 프로젝트 경험도 같이 준비해야 한다.

**주요 대비 질문:**
- 이 기능을 어떻게 구현했는가
- 구현할 때 어떤 기술적 고민을 했는가
- 왜 그 구조를 선택했는가
- 다른 대안은 없었는가
- 운영 중 실패하면 어떻게 대응할 수 있는가
- 테스트는 어떻게 했는가
- 성능이나 트랜잭션 문제는 어떻게 고려했는가

**예시 (GraphQL 경험이 있는 경우):**
- "GraphQL을 왜 도입했나요?"
- "REST로는 왜 부족했나요?"
- "GraphQL을 쓰면서 생긴 단점이나 운영 이슈는 없었나요?"

## 8. AI로 연습하는 방법

1. AI에게 짧은 레거시 코드 템플릿 생성 요청
2. 정답을 보지 않고 직접 문제점 찾기
3. 문제 위치, 위험성, 개선 방향, 테스트 방법까지 작성
4. AI에게 내 코드 리뷰 답변 평가 요청
5. 놓친 문제와 표현을 보완
6. 같은 방식으로 다른 도메인 문제 반복

> "먼저 전체 요청 흐름을 Controller → Service → Repository → Util 순서로 파악하겠습니다. 그다음 입력 검증, 트랜잭션 경계, 예외 처리, 계층별 책임 분리, DB 조회 방식, 테스트 가능성 순서로 보겠습니다. 이해가 안 되는 도메인 의도는 중간에 질문드리면서 확인하겠습니다."

## 10. 실전 답변 템플릿

> "제가 먼저 본 문제는 **[문제 위치]**입니다. 현재 코드는 **[현재 상태]**로 되어 있어서 **[위험성]**이 있을 수 있습니다. 그래서 **[개선 방향]**으로 바꾸는 것이 좋겠습니다. 테스트는 **[검증할 케이스]**를 확인하면 좋겠습니다."

**예시:**
> "제가 먼저 본 문제는 Service의 트랜잭션 경계입니다. 현재 코드는 여러 DB 변경을 처리하고 있는데 @Transactional이 보이지 않아, 중간에 예외가 발생했을 때 일부 데이터만 반영될 가능성이 있습니다. 그래서 Service 유스케이스 단위로 @Transactional을 적용하고, 외부 API 호출이 있다면 DB 트랜잭션과 분리할 수 있는지도 검토하는 것이 좋겠습니다. 테스트는 중간 예외 발생 시 DB 변경이 롤백되는지 확인하면 좋겠습니다."

## 11. 최종 정리

헤렌 1차 기술면접의 레거시 코드 리뷰 준비 방향은 다음과 같다.
- 큰 프로젝트를 분석하는 연습이 아니라, **짧은 레거시 코드 템플릿에서 문제점을 빠르게 찾는 연습**이 필요하다.
- 코드 양은 많지 않을 수 있지만, 개발자로서 하면 안 되는 코드 습관들이 많이 들어 있을 가능성이 높다.
- 따라서 `Controller → Service → Repository → Util` 흐름을 빠르게 파악하고, `@Valid`, `@Transactional`, 예외 처리, DTO 분리, DB 조회 방식, 테스트 가능성 등을 중심으로 리뷰해야 한다.
- 모르는 부분은 오래 고민하지 말고 면접관에게 질문하면서 의도를 확인하는 것이 좋다.
- 마지막으로 코드 리뷰 문제뿐 아니라 **이력서 기반 기술 질문**도 함께 준비해야 한다.
