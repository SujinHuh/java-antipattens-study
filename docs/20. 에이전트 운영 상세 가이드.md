# 20. 에이전트 운영 상세 가이드

이 문서는 `12. 에이전트 호출 판단 체크리스트.md`에서 분리한 상세 운영 문서다.

`12`번은 빠르게 보는 라우팅 문서로 유지하고, 각 단계의 세부 흐름과 문서 역할은 이 문서에서 확인한다.

## 기본 판단 기준

- 새 문제를 만들고 싶으면 `1. 문제 생성 에이전트.md`를 사용한다.
- 새 문제를 낸 직후 문제 품질을 검수하려면 `4. 회고-서브에이전트.md`, `1. 문제 생성 에이전트.md`, `13. 실제 루프 실행 프롬프트.md`, `17. 루프 상태 기록.md`, `25. 수요일 20회 반복 목표.md`, `23. 개념별 출제 확장 매트릭스.md`를 함께 본다.
- 여러 개념을 섞은 문제는 답변 후 복습용 `답변후-개념공부.md`를 만들 수 있지만, 문제 풀이 전에는 열람 금지 자료로 관리한다.
- 내 코드 리뷰 답변을 평가받고 싶으면 `2. 답변 채점 에이전트.md`를 사용한다.
- 특정 개념이나 실무 기준이 이해되지 않으면 `3. 설명 에이전트.md`를 사용한다.
- 이번 문제 품질과 내 답변 방식을 평가하고 싶으면 `4. 회고-서브에이전트.md`를 사용한다.
- 다음 문제 생성 조건을 보강하고 싶으면 `5. 프롬프트 개선 에이전트.md`를 사용한다.
- 문제, 답변, 피드백을 기록하고 싶으면 `6. TIL-로그 에이전트.md`를 사용한다.
- 반복해서 나온 안티패턴을 코딩 지침으로 바꾸고 싶으면 `7. 안티패턴 지침화 에이전트.md`를 사용한다.
- 다른 AI 답변이나 피드백을 검증하고 싶으면 `8. 교차검증 에이전트.md` 또는 `15. AI 교차검증 실행 프롬프트.md`를 사용한다.
- 레거시 프로젝트 구조를 파악하고 싶으면 `9. 프로젝트 온보딩-아키텍처 추출 에이전트.md`를 사용한다.
- 전체 운영 흐름을 다시 확인하고 싶으면 `10. Java Antipatterns Study Loop.md`를 사용한다.
- 면접장에서 말할 답변 순서를 잡고 싶으면 `11. 코드 리뷰 면접 답변 템플릿.md`를 사용한다.
- 실제 문제 풀이를 시작하고 싶으면 `13. 실제 루프 실행 프롬프트.md`를 사용한다.
- 면접에서 AI 활용 경험을 말하고 싶으면 `14. 면접 전 어필할 내용.md`를 사용한다.
- 실제 코드나 문제 코드를 한 번에 분석하고 싶으면 `16. 통합 코드리뷰-리팩토링 코치.md`를 사용한다.
- 여러 회차에 걸친 반복 약점과 다음 문제 생성 상태를 관리하고 싶으면 `17. 루프 상태 기록.md`를 사용한다.
- 작업 성격에 따라 어떤 AI 역할 관점으로 볼지 판단하고 싶으면 `18. AI 역할 분담 가이드.md`를 사용한다.
- 헤렌 공고 기준으로 부족한 관점을 확인하거나 나중에 보강할 내용을 관리하려면 `19. 헤렌 공고 기반 docs 보강 예정.md`를 사용한다.
- 풀이 중 힌트를 요청받으면 `21. 힌트 제공 가이드.md`를 사용한다.
- 계층별 반복 패턴과 조언 템플릿을 확인하려면 `22. 계층별 반복 패턴 체크리스트.md`를 사용한다.
- 문제 출제 다양화와 채점 후 약점 갱신을 관리하려면 `23. 개념별 출제 확장 매트릭스.md`를 사용한다.

## 실행 중 다음 단계 판단

문제를 받은 뒤에는 사용자가 먼저 답변한다.

새 문제를 파일로 만든 직후에는 문제 품질 검수를 먼저 수행한다.

검수 기준은 `4. 회고-서브에이전트.md`, `1. 문제 생성 에이전트.md`, `13. 실제 루프 실행 프롬프트.md`, `17. 루프 상태 기록.md`, `25. 수요일 20회 반복 목표.md`, `23. 개념별 출제 확장 매트릭스.md`를 따른다.

검수 결과는 각 문제 폴더의 `문제검수결과.md`에 남긴다.

사용자가 "작성했어", "다 풀었어", "완료했어"처럼 풀이 완료를 알리면, 현재 회차 문제와 사용자 답변을 기준으로 `12. 에이전트 호출 판단 체크리스트.md`를 먼저 확인하고 필요한 에이전트 문서의 기준을 적용한다.

답변 후 흐름은 `12. 에이전트 호출 판단 체크리스트.md`의 `풀이 완료 시 고정 순서`를 기준으로 한다.

답변 후 개념 복습이 필요하면 해당 문제 폴더의 `답변후-개념공부.md`를 열어 이번 문제에 섞인 개념, 관련 docs 링크, 추가 학습 순서를 확인한다.

`review-answer.md` 작성 후에는 `15. AI 교차검증 실행 프롬프트.md` 기준으로 교차검증 결과를 남긴다.
기술 판단이 특히 애매하거나 다른 관점이 필요하면 `8. 교차검증 에이전트.md` 기준도 함께 적용한다.

같은 안티패턴이 여러 번 반복되면 `7. 안티패턴 지침화 에이전트.md`로 코딩 지침 후보를 만들고, 확정된 지침은 `antipattern-kit/java-spring-coding-guardrails.md`에 누적한다.

답변/채점 후 새로 드러난 개념 출제 방식과 약점은 `23. 개념별 출제 확장 매트릭스.md`에 업데이트하고, 다음 회차 문제에 반영할 누적 약점과 반복 키워드는 `17. 루프 상태 기록.md`에 업데이트한다.

코드 전체를 한 번에 검수하고 리팩토링 우선순위, 개선 코드, 테스트 예시까지 받고 싶으면 `16. 통합 코드리뷰-리팩토링 코치.md`를 사용한다.

## 풀이 완료 트리거

사용자가 풀이 완료를 알리면 아래 순서로 판단한다.

1. `2. 답변 채점 에이전트.md`
   - 실제 면접관 기준으로 점수, 틀린 점, 모호한 표현, 놓친 본질을 분리한다.
   - `review-answer.md`에 핵심 문제 요약, 사용자 답변, 채점, 반드시 잡았어야 하는 문제, 문제 유형 분류, 안티패턴 지침화 후보, 개선 답변, 개념 키워드, 면접관 예상 꼬리질문을 기록한다.
   - `시니어 개발자의 피드백 (면접관 관점)` 섹션에는 인정되는 답변, 실제 면접에서 부족하게 들리는 지점, 다음 리뷰에서 먼저 확인할 것을 남긴다.

2. `15. AI 교차검증 실행 프롬프트.md`
   - `review-answer.md`를 작성한 뒤 기술 오류, 위험한 표현, 빠진 관점, 다음 루프 키워드를 교차검증한다.
   - 결과는 각 문제 폴더의 `검수결과.md`에 남긴다.

3. `3. 설명 에이전트.md`
   - 사용자가 놓친 개념이나 위험하게 말한 개념을 짧게 설명한다.
   - 설명은 암기용 정의보다 면접장에서 말할 수 있는 문장으로 정리한다.

4. `7. 안티패턴 지침화 에이전트.md`
   - 같은 안티패턴이 반복되면 `review-answer.md`의 `안티패턴 지침화 후보`와 `antipattern-kit/java-spring-coding-guardrails.md` 반영 후보를 만든다.
   - 확정 반영할 항목은 실제 `antipattern-kit/java-spring-coding-guardrails.md`에 반영한다.

5. `23. 개념별 출제 확장 매트릭스.md`
   - 사용자가 실제로 잡은 개념과 놓친 개념을 기준으로 `이미 나온 방식`, `보충해야 할 방식`, `수진님 현재 약점과 반복 복습 대상`을 갱신한다.

6. `17. 루프 상태 기록.md`
   - 다음 문제에 다시 섞을 약점, 반복 키워드, 누적 안티패턴 후보를 업데이트한다.

7. `4. 회고-서브에이전트.md`, `5. 프롬프트 개선 에이전트.md`
   - 문제 품질, 답변 품질, 다음 회차 난이도와 반복 출제할 약점을 점검하고 다음 문제 조건을 보강한다.

8. `6. TIL-로그 에이전트.md`
   - `til.md`에 문제 요약, 틀린 부분, 키워드별 개념 설명, 면접용 문장, 안티패턴 후보를 기록한다.

9. `16. 통합 코드리뷰-리팩토링 코치.md`
   - 문제 코드, 사용자 답변, 리팩토링 계획, solution 코드를 함께 검토해야 할 때 적용한다.

모든 단계를 항상 길게 실행하지 않는다. 문제 성격과 답변 수준에 맞춰 필요한 기준을 적용하되, 어떤 기준을 사용했는지는 피드백에 드러나게 한다.

## 전체 파일 목차

### 핵심 에이전트

1. `1. 문제 생성 에이전트.md`
   - 레거시 코드 리뷰형 문제를 생성할 때 사용한다.
   - 사용자가 직접 답변할 수 있도록 문제를 낸 뒤 멈춘다.

2. `2. 답변 채점 에이전트.md`
   - 사용자의 코드 리뷰 답변을 실제 면접 기준으로 채점할 때 사용한다.
   - 틀린 내용, 위험한 표현, 모호한 표현, 놓친 본질을 분리한다.

3. `3. 설명 에이전트.md`
   - 채점 후 이해가 부족한 개념을 실무와 면접 기준으로 설명할 때 사용한다.
   - `@Transactional`, AOP, self-invocation, 예외 처리 같은 개념을 정확한 용어로 정리한다.

4. `4. 회고-서브에이전트.md`
   - 한 문제 풀이가 끝난 뒤 문제 품질과 답변 방식을 평가할 때 사용한다.
   - 다음 문제에서 반복 출제할 약점과 보강할 프롬프트를 찾는다.

5. `5. 프롬프트 개선 에이전트.md`
   - 문제 생성, 답변 채점, 설명 프롬프트를 다음 회차에 맞게 개선할 때 사용한다.
   - 사용자가 틀린 개념과 모호한 표현을 다음 문제에 다시 반영한다.

6. `6. TIL-로그 에이전트.md`
   - 문제, 답변, 피드백, 개선 키워드를 복습 가능한 TIL로 기록할 때 사용한다.
   - 기록이 원본 문제와 어긋나지 않는지 검수한다.

7. `7. 안티패턴 지침화 에이전트.md`
   - 반복해서 나온 Java/Spring 안티패턴을 코딩 지침으로 바꿀 때 사용한다.
   - 최종적으로 Java/Spring Agent Coding Harness로 발전시킬 수 있게 정리한다.

8. `8. 교차검증 에이전트.md`
   - 다른 AI가 만든 문제, 답변, 피드백을 검증할 때 사용한다.
   - 기술 오류, 단정적 표현, 버전 차이, 빠진 실무 관점을 찾는다.

9. `9. 프로젝트 온보딩-아키텍처 추출 에이전트.md`
   - 레거시 프로젝트에 투입됐을 때 코드와 문서 기반으로 구조를 파악할 때 사용한다.
   - 모듈, 도메인, API 흐름, 예외 정책, 트랜잭션 경계, 리팩토링 후보를 추출한다.

### 보조 문서와 실행 프롬프트

- `10. Java Antipatterns Study Loop.md`
  - 사람이 읽는 학습 운영 개요 문서다.
  - 전체 진행 방식, 반복 횟수, 기록 항목을 확인할 때 사용한다.
- `11. 코드 리뷰 면접 답변 템플릿.md`
  - 사용자가 면접장에서 답변할 때 말의 순서를 잡는 템플릿이다.
- `12. 에이전트 호출 판단 체크리스트.md`
  - 상황별로 어떤 에이전트를 호출할지 빠르게 판단할 때 사용한다.
- `13. 실제 루프 실행 프롬프트.md`
  - 실제 문제 풀이를 시작할 때 사용하는 실행용 프롬프트다.
- `14. 면접 전 어필할 내용.md`
  - 면접에서 AI 학습 방식을 어떻게 설명할지 정리한 문서다.
- `15. AI 교차검증 실행 프롬프트.md`
  - 타 AI 모델이 도출한 문제나 답변을 교차검증할 때 사용하는 실행용 프롬프트다.
- `16. 통합 코드리뷰-리팩토링 코치.md`
  - 실제 코드나 문제 코드를 붙여넣고 전체 코드 리뷰와 리팩토링 코칭을 받을 때 사용한다.
- `17. 루프 상태 기록.md`
  - 여러 회차에 걸친 반복 약점, 보강 키워드, 안티패턴 후보를 누적 관리한다.
- `18. AI 역할 분담 가이드.md`
  - 코드 작업, 답변 정리, 교차검증을 어떤 역할 관점으로 나눌지 판단할 때 사용한다.
- `19. 헤렌 공고 기반 docs 보강 예정.md`
  - 헤렌 공고와 현재 docs 사이의 부족한 관점을 관리한다.
- `21. 힌트 제공 가이드.md`
  - 풀이 중 사용자가 힌트를 요청했을 때 정답을 공개하지 않고 단계별로 돕는 기준이다.
- `22. 계층별 반복 패턴 체크리스트.md`
  - Controller, Service, Repository, Util, DTO/Entity, Exception, Test 순서로 볼 반복 패턴과 계층별 조언 템플릿을 관리한다.
- `23. 개념별 출제 확장 매트릭스.md`
  - 같은 개념을 다양한 코드 모양으로 반복 출제하고, 답변/채점 후 약점을 갱신하는 운영 문서다.
- `13-1. 실제 루프 상세 레퍼런스.md`
  - 기존 13번의 상세 기준을 보존한 문서다. 13번으로 빠르게 실행하고, 세부 기준 확인이 필요할 때만 참고한다.
- `24. 학습 운영 변경 로그.md`
  - 공부하면서 문서 구조, 출제 기준, 피드백 방식이 바뀐 이유와 결과를 누적 기록한다.
- `25. 수요일 20회 반복 목표.md`
  - 수요일까지 020번 문제를 목표로 빠른 반복과 깊은 피드백을 어떻게 섞을지 관리한다.
- `legacy.md`
  - 이전 레거시 코드 리뷰 운영 기준을 보관하고 필요 시 참조한다.
- `공통개념.md`, `공통개념/`
  - 개념별 상세 설명과 면접용 문장을 확인하는 레퍼런스다.

## 진행 안내 원칙

각 단계가 끝날 때는 다음에 실행할 에이전트 번호와 역할을 명확히 안내한다.

예시:

```text
다음은 2번 답변 채점 에이전트로 넘어가겠습니다.
이번 답변에서 틀린 부분, 모호한 표현, 놓친 본질을 분리해서 평가하겠습니다.
```

```text
다음은 3번 설명 에이전트로 넘어가겠습니다.
방금 채점에서 나온 @Transactional, self-invocation 개념을 실무 기준으로 정리하겠습니다.
```

```text
다음은 6번 TIL/로그 에이전트로 넘어가겠습니다.
이번 문제, 답변, 피드백, 다음 보강 키워드를 복습 가능한 형태로 기록하겠습니다.
```

## 주의

- 모든 단계를 항상 실행할 필요는 없다.
- 사용자가 이미 이해한 개념은 설명 단계를 짧게 줄일 수 있다.
- `review-answer.md` 작성 후 15번 교차검증은 기본 실행한다.
- 추가 외부 관점이나 더 깊은 기술 검증이 필요할 때 8번 교차검증 에이전트 기준도 함께 사용한다.
- 프로젝트 온보딩/아키텍처 추출은 실제 프로젝트 분석이 필요할 때 별도로 사용한다.
