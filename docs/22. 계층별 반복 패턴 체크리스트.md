# 22. 계층별 반복 패턴 체크리스트

이 문서는 짧은 Java/Spring Boot 레거시 코드 리뷰 면접에서 반복적으로 봐야 할 패턴을 계층별로 정리한다.

목표는 깊은 이론을 전부 외우는 것이 아니라, 40분 안에 핵심 문제를 빠르게 찾고 면접용 문장으로 설명하는 것이다.

## 사용 방법

문제를 볼 때는 아래 순서로 확인한다.

1. Controller
2. Service
3. Repository
4. Util
5. DTO/Entity
6. Exception
7. Test

사용자는 각 계층을 먼저 직접 작성한다.
Codex는 사용자가 조언을 요청한 뒤에만 해당 계층 아래에 `조언`, `표현 보정`, `면접용 문장`을 붙인다.
다음 계층의 힌트나 정답은 사용자가 요청하기 전에는 공개하지 않는다.

각 항목은 아래 4문장으로 정리한다.

```text
문제 위치:
왜 문제인지:
실무 영향:
개선 방향:
```

## 먼저 잡아야 할 10개 반복 패턴

금요일 면접 전까지는 아래 10개를 최우선으로 본다.

1. Controller가 Service를 `new` 하는지
2. Service가 Repository를 `new` 하는지
3. Controller에 비즈니스 로직이 있는지
4. Service가 `Map`이나 `ResponseEntity` 같은 HTTP 응답 구조를 만드는지
5. Entity를 Request/Response에 직접 쓰는지
6. `@Valid`나 DTO 검증이 빠졌는지
7. 조회 실패를 `null`이나 임시 Entity 생성으로 숨기는지
8. `RuntimeException`을 직접 던지는지
9. 문자열 상태값을 하드코딩하는지
10. 반복문 안에서 Repository 조회가 있는지

## 1. Controller에서 볼 것

### 빠른 읽기 순서

1. 요청을 어떤 타입으로 받는지 확인한다.
2. `@RequestBody`, `@Valid`, 파라미터 검증이 필요한데 빠졌는지 본다.
3. Controller가 Service를 어떻게 얻는지, 직접 `new` 하는지 본다.
4. 비즈니스 판단이나 요청 객체 변경이 Controller에 있는지 본다.
5. 응답이 `ResponseEntity<ResponseDTO>`처럼 일관된 API 계약인지 확인한다.

### 반복 패턴

- Service를 `new`로 직접 생성하는지
- Entity를 Request/Response로 직접 받거나 반환하는지
- `Map<String, Object>`로 응답을 만드는지
- 실패 상황을 200 OK처럼 반환하는지
- Controller에서 상태 변경, 우선순위 판단, 가격/정책 판단을 하는지
- 요청 DTO를 직접 수정하는지
- `@RequestBody`, `@Valid`가 필요한데 빠졌는지

### 면접용 문장

```text
Controller는 HTTP 요청/응답 경계와 입력 검증에 집중해야 합니다.
Service 생성, 비즈니스 판단, 요청 객체 변경까지 담당하면 계층 책임이 섞이고 테스트와 변경이 어려워집니다.
```

## 2. Service에서 볼 것

### 빠른 읽기 순서

1. Service가 Repository/Util/Policy를 직접 `new` 하는지 본다.
2. 클라이언트 요청값을 서버 상태처럼 신뢰하는지 확인한다.
3. 조회 실패를 예외로 처리하는지, `null`/임시 Entity/기본값으로 숨기는지 본다.
4. 상태 변경과 저장이 있다면 트랜잭션 경계가 보이는지 본다.
5. Service가 HTTP 응답 구조나 상태코드를 만들고 있는지 확인한다.

### 로직 흐름 추적 순서

Service나 Repository 코드를 볼 때는 아래 순서로 한 줄씩 따라간다.
`review-answer.md`를 작성할 때도 Service/Repository 섹션 위에 이 흐름을 먼저 적고 문제 위치를 연결한다.

1. 입력값: Request DTO에서 들어온 값 중 서버가 믿으면 안 되는 값이 있는지 본다.
2. 조회: Repository 조회 실패가 예외로 중단되는지, 기본값/임시 객체로 계속 진행되는지 본다.
3. 검증: 저장된 Entity 상태 기준으로 검증하는지, 요청값 기준으로 검증하는지 본다.
4. 계산: 금액/상태/시간처럼 서버가 결정해야 하는 값을 클라이언트 입력에서 가져오는지 본다.
5. 외부 호출: PG/API/알림 같은 외부 호출이 DB 저장 전후 어디에 있는지 본다.
6. 저장: 상태 변경이 완전한 한 묶음으로 저장되는지, 일부 필드만 직접 바꾸는지 본다.
7. 결과: Service가 도메인 결과를 반환하는지, HTTP 응답 구조까지 만들어버리는지 본다.

### 반복 패턴

- Repository를 `new`로 직접 생성하는지
- 조회 실패를 예외로 처리하지 않고 정상 흐름처럼 숨기는지
- 클라이언트 요청값을 서버 상태처럼 신뢰하는지
- 상태 변경과 저장이 있는데 트랜잭션 경계가 보이지 않는지
- `ResponseEntity`, `Map status/message/data` 같은 HTTP 응답 구조를 반환하는지
- 문자열 상태값을 직접 비교하거나 대입하는지
- 반복문 안에서 Repository를 다시 호출하는지
- 너무 많은 책임을 한 메서드에서 처리하는지

### 면접용 문장

```text
Service는 유스케이스와 비즈니스 규칙을 담당하되, HTTP 응답 객체를 만들거나 Repository 생성을 직접 책임지면 안 됩니다.
조회 실패와 상태 변경은 서버에 저장된 데이터 기준으로 처리하고, 결과는 Response DTO 같은 비즈니스 결과로 반환하는 편이 좋습니다.
```

## 3. Repository에서 볼 것

### 빠른 읽기 순서

1. 조회 실패를 `null`, 빈 객체, Optional 중 무엇으로 표현하는지 본다.
2. 필요한 조건 조회가 있는지, `findAll()` 후 애플리케이션에서 필터링하는지 확인한다.
3. 반복문 안에서 다시 Repository 조회가 발생하는지 본다.
4. 내부 컬렉션이나 mutable 상태를 그대로 노출하는지 본다.
5. 저장 메서드가 생성/수정/중복 기준을 구분하는지 확인한다.
6. Repository 반환값이 Service 흐름에서 어떻게 처리되는지 함께 본다.

### 반복 패턴

- 조회 실패 시 `null`을 반환하는지
- 내부 컬렉션을 그대로 외부에 반환하는지
- `static List` 같은 공유 mutable 상태를 저장소처럼 사용하는지
- `findAll()` 후 애플리케이션에서 필터링하는지
- Repository 안에 비즈니스 정책 판단이 들어가 있는지
- 저장 메서드가 중복 데이터나 갱신 기준을 고려하지 않는지

### 면접용 문장

```text
Repository는 데이터 접근 책임에 집중해야 합니다.
조회 실패는 null보다 Optional이나 명확한 예외 흐름으로 다루고, 필요한 조건은 Repository 메서드에서 목적에 맞게 조회하는 편이 좋습니다.
```

## 4. Util에서 볼 것

### 빠른 읽기 순서

1. Util이 순수 변환/계산인지, 도메인 정책 판단인지 구분한다.
2. 알림, 로그, 외부 전달 같은 부가 작업이 static으로 묶였는지 본다.
3. 테스트에서 대체하기 어려운 의존성이 있는지 확인한다.
4. 여러 도메인 정책이 공통 Util에 섞여 있는지 본다.
5. Policy, Validator, Notifier, Component로 분리할 책임인지 판단한다.

### 반복 패턴

- 순수 변환/계산이 아니라 비즈니스 정책이 들어가 있는지
- 알림, 로그, 외부 연동 같은 부가 작업을 static 메서드로 처리하는지
- 테스트에서 대체하기 어려운 static 의존성이 있는지
- 여러 계층에서 중복 정책처럼 호출되는지

### 면접용 문장

```text
Util은 순수하고 재사용 가능한 보조 기능에 적합합니다.
정책 판단이나 알림 같은 책임은 별도 Component나 Service로 분리해야 테스트와 변경이 쉬워집니다.
```

## 5. DTO/Entity에서 볼 것

### 빠른 읽기 순서

1. Request DTO가 외부 입력만 담는지, 서버 결정값까지 받는지 본다.
2. Response에 Entity가 그대로 나가는지 확인한다.
3. 검증 어노테이션과 도메인 검증 위치가 분리되어 있는지 본다.
4. Entity 필드가 public이거나 외부에서 직접 바뀌는지 확인한다.
5. 상태값, 금액, 권한 같은 민감한 필드가 API 계약에 노출되는지 본다.

### 반복 패턴

- Request DTO에 서버가 계산해야 할 값이 들어 있는지
- DTO 검증 어노테이션이나 검증 기준이 없는지
- Entity를 API 요청/응답으로 직접 쓰는지
- Entity 필드가 public으로 열려 있는지
- 문자열 상태값 대신 enum을 써야 하는지
- Entity가 너무 빈약하거나 반대로 모든 책임을 다 갖고 있는지

### 면접용 문장

```text
Request DTO는 외부 입력 계약이고 Entity는 내부 도메인 상태입니다.
API 계약과 내부 모델을 분리해야 검증, 보안, 변경 영향을 관리하기 쉽습니다.
```

## 6. Exception에서 볼 것

### 빠른 읽기 순서

1. Service/Controller에서 `throw` 하는 지점을 찾는다.
2. 어떤 예외 타입을 던지는지 확인한다.
3. 그 예외가 `ControllerAdvice` 같은 공통 처리로 매핑되는지 본다.
4. HTTP 상태코드와 응답 DTO가 일관적인지 확인한다.
5. 클라이언트/FE/QA가 실패 원인을 예측할 수 있는지 판단한다.

### 반복 패턴

- `RuntimeException`을 직접 던지는지
- 예외 메시지가 너무 모호한지
- 에러 코드나 도메인 예외 구분이 없는지
- Controller/Service마다 예외 처리 방식이 다른지
- 실패인데 성공 응답으로 감싸는지

### 면접용 문장

```text
예외는 장애 원인과 처리 방식을 드러내야 합니다.
도메인 예외와 에러 코드를 분리하고, Controller 계층에서 일관된 HTTP 응답으로 매핑하는 것이 좋습니다.
```

## 7. Test에서 볼 것

### 빠른 읽기 순서

1. 테스트가 실제로 실행되는지 `@Test`와 설정을 확인한다.
2. assertion이 있는지, 단순 호출만 하는지 본다.
3. 성공 케이스뿐 아니라 실패/경계/권한/상태 불가 케이스가 있는지 확인한다.
4. 직접 `new`, static 저장소, 현재 시간 호출 때문에 테스트 격리가 깨지는지 본다.
5. 테스트가 핵심 정책과 예외 흐름을 고정하는지 판단한다.

### 반복 패턴

- `@Test`가 없어 실행되지 않는 테스트인지
- assertion이 없는지
- 성공 케이스만 있고 실패 케이스가 없는지
- 직접 `new` 생성 때문에 mock이나 fake로 대체하기 어려운지
- 핵심 정책, 예외, 경계값을 검증하지 않는지

### 면접용 문장

```text
테스트는 코드가 실행되는지만 보는 것이 아니라 핵심 정책과 실패 흐름을 고정해야 합니다.
DI가 되어 있어야 Repository나 외부 의존성을 대체해서 Service 단위 테스트를 만들기 쉽습니다.
```

## 현재 회차 풀이 순서

현재 회차에서 어디까지 봤는지는 `17. 루프 상태 기록.md`와 해당 문제의 `review-answer.md`를 기준으로 확인한다.
이 문서는 특정 회차 상태를 고정하지 않고, 모든 문제에서 재사용하는 계층별 체크리스트로 유지한다.

## review-answer.md 상세 누적 템플릿

사용자가 계층별 분석을 한 뒤 "이 내용을 review-answer.md에 정리해줘"라고 요청하면 아래 상세 포맷을 우선 사용한다.
이 포맷은 특히 사용자가 코드 조각을 보며 학습할 때 사용한다.

````markdown
### 계층명

#### 문제 N. 문제 제목

- 기존 코드:

```java
// 문제를 일으키는 핵심 코드 조각
```

- 판단: 이 코드가 왜 의심되는지, 또는 왜 그 자체는 문제가 아니고 처리 지점이 중요한지 적는다.
- 왜 문제인지: 계층 책임, 결합도, 테스트 가능성, 데이터 정합성, API 계약 중 어떤 기준에서 문제인지 설명한다.
- 실무 영향: 운영 장애, 협업 비용, 회귀 위험, 보안/정합성 문제로 어떻게 이어지는지 적는다.
- 개선 방향: 어느 계층/객체로 책임을 옮기고 어떤 기준으로 바꿀지 적는다.

- 리팩토링 코드:

```java
// 개선된 핵심 코드 조각
```

- 면접용 문장:

```text
면접장에서 20~30초 안에 말할 수 있는 문장으로 정리한다.
```
````

상황에 따라 아래 보조 항목을 추가한다.

- `진짜 문제 위치`: Repository의 `Optional.empty()`처럼 코드 위치와 책임 위치가 다를 때 사용한다.
- `문제 있는 처리`: 잘못된 후속 흐름을 보여줄 때 사용한다.
- `개선된 테스트 관점`: `Clock`, fake/mock, 외부 API 대체처럼 테스트 가능성이 핵심일 때 사용한다.
- `암기 포인트`: 사용자가 헷갈린 개념을 한 줄로 고정할 때 사용한다.
- `꼬리질문`: 면접관이 바로 이어서 물을 질문을 1~3개 남긴다.

사용자가 빠르게 작성할 때는 아래처럼 짧게 시작해도 된다.

```text
### Service

1. Repository 직접 생성
   - 위치:
   - 문제:
   - 개선:

2. 조회 실패를 정상 흐름으로 숨김
   - 위치:
   - 문제:
   - 개선:
```

## 계층별 조언 템플릿

사용자가 한 계층의 답변을 작성한 뒤 조언을 요청하면 아래 형식으로 정리한다.
각 문제 제목에는 분류 태그를 붙여서 사용자가 이것이 SOLID 문제인지, Java 안티패턴인지, API 계약 문제인지 바로 알 수 있게 한다.
