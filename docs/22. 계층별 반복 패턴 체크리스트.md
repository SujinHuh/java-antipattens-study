# 22. 계층별 반복 패턴 체크리스트

이 문서는 짧은 Java/Spring Boot 레거시 코드 리뷰 면접에서 반복적으로 봐야 할 패턴을 계층별로 정리한다.

목표는 깊은 이론을 전부 외우는 것이 아니라, 40분 안에 핵심 문제를 빠르게 찾고 면접용 문장으로 설명하는 것이다.

## 사용 방법

문제를 볼 때는 아래 순서로 확인한다.

1. Controller
2. Service
3. Repository
4. Util
5. DTO/Entity
6. Exception
7. Test

사용자는 각 계층을 먼저 직접 작성한다.
Codex는 사용자가 조언을 요청한 뒤에만 해당 계층 아래에 `조언`, `표현 보정`, `면접용 문장`을 붙인다.
다음 계층의 힌트나 정답은 사용자가 요청하기 전에는 공개하지 않는다.

각 항목은 아래 4문장으로 정리한다.

```text
문제 위치:
왜 문제인지:
실무 영향:
개선 방향:
```

## 먼저 잡아야 할 10개 반복 패턴

금요일 면접 전까지는 아래 10개를 최우선으로 본다.

1. Controller가 Service를 `new` 하는지
2. Service가 Repository를 `new` 하는지
3. Controller에 비즈니스 로직이 있는지
4. Service가 `Map`이나 `ResponseEntity` 같은 HTTP 응답 구조를 만드는지
5. Entity를 Request/Response에 직접 쓰는지
6. `@Valid`나 DTO 검증이 빠졌는지
7. 조회 실패를 `null`이나 임시 Entity 생성으로 숨기는지
8. `RuntimeException`을 직접 던지는지
9. 문자열 상태값을 하드코딩하는지
10. 반복문 안에서 Repository 조회가 있는지

## 1. Controller에서 볼 것

### 반복 패턴

- Service를 `new`로 직접 생성하는지
- Entity를 Request/Response로 직접 받거나 반환하는지
- `Map<String, Object>`로 응답을 만드는지
- 실패 상황을 200 OK처럼 반환하는지
- Controller에서 상태 변경, 우선순위 판단, 가격/정책 판단을 하는지
- 요청 DTO를 직접 수정하는지
- `@RequestBody`, `@Valid`가 필요한데 빠졌는지

### 면접용 문장

```text
Controller는 HTTP 요청/응답 경계와 입력 검증에 집중해야 합니다.
Service 생성, 비즈니스 판단, 요청 객체 변경까지 담당하면 계층 책임이 섞이고 테스트와 변경이 어려워집니다.
```

## 2. Service에서 볼 것

### 반복 패턴

- Repository를 `new`로 직접 생성하는지
- 조회 실패를 예외로 처리하지 않고 정상 흐름처럼 숨기는지
- 클라이언트 요청값을 서버 상태처럼 신뢰하는지
- 상태 변경과 저장이 있는데 트랜잭션 경계가 보이지 않는지
- `ResponseEntity`, `Map status/message/data` 같은 HTTP 응답 구조를 반환하는지
- 문자열 상태값을 직접 비교하거나 대입하는지
- 반복문 안에서 Repository를 다시 호출하는지
- 너무 많은 책임을 한 메서드에서 처리하는지

### 면접용 문장

```text
Service는 유스케이스와 비즈니스 규칙을 담당하되, HTTP 응답 객체를 만들거나 Repository 생성을 직접 책임지면 안 됩니다.
조회 실패와 상태 변경은 서버에 저장된 데이터 기준으로 처리하고, 결과는 Response DTO 같은 비즈니스 결과로 반환하는 편이 좋습니다.
```

## 3. Repository에서 볼 것

### 반복 패턴

- 조회 실패 시 `null`을 반환하는지
- 내부 컬렉션을 그대로 외부에 반환하는지
- `static List` 같은 공유 mutable 상태를 저장소처럼 사용하는지
- `findAll()` 후 애플리케이션에서 필터링하는지
- Repository 안에 비즈니스 정책 판단이 들어가 있는지
- 저장 메서드가 중복 데이터나 갱신 기준을 고려하지 않는지

### 면접용 문장

```text
Repository는 데이터 접근 책임에 집중해야 합니다.
조회 실패는 null보다 Optional이나 명확한 예외 흐름으로 다루고, 필요한 조건은 Repository 메서드에서 목적에 맞게 조회하는 편이 좋습니다.
```

## 4. Util에서 볼 것

### 반복 패턴

- 순수 변환/계산이 아니라 비즈니스 정책이 들어가 있는지
- 알림, 로그, 외부 연동 같은 부가 작업을 static 메서드로 처리하는지
- 테스트에서 대체하기 어려운 static 의존성이 있는지
- 여러 계층에서 중복 정책처럼 호출되는지

### 면접용 문장

```text
Util은 순수하고 재사용 가능한 보조 기능에 적합합니다.
정책 판단이나 알림 같은 책임은 별도 Component나 Service로 분리해야 테스트와 변경이 쉬워집니다.
```

## 5. DTO/Entity에서 볼 것

### 반복 패턴

- Request DTO에 서버가 계산해야 할 값이 들어 있는지
- DTO 검증 어노테이션이나 검증 기준이 없는지
- Entity를 API 요청/응답으로 직접 쓰는지
- Entity 필드가 public으로 열려 있는지
- 문자열 상태값 대신 enum을 써야 하는지
- Entity가 너무 빈약하거나 반대로 모든 책임을 다 갖고 있는지

### 면접용 문장

```text
Request DTO는 외부 입력 계약이고 Entity는 내부 도메인 상태입니다.
API 계약과 내부 모델을 분리해야 검증, 보안, 변경 영향을 관리하기 쉽습니다.
```

## 6. Exception에서 볼 것

### 반복 패턴

- `RuntimeException`을 직접 던지는지
- 예외 메시지가 너무 모호한지
- 에러 코드나 도메인 예외 구분이 없는지
- Controller/Service마다 예외 처리 방식이 다른지
- 실패인데 성공 응답으로 감싸는지

### 면접용 문장

```text
예외는 장애 원인과 처리 방식을 드러내야 합니다.
도메인 예외와 에러 코드를 분리하고, Controller 계층에서 일관된 HTTP 응답으로 매핑하는 것이 좋습니다.
```

## 7. Test에서 볼 것

### 반복 패턴

- `@Test`가 없어 실행되지 않는 테스트인지
- assertion이 없는지
- 성공 케이스만 있고 실패 케이스가 없는지
- 직접 `new` 생성 때문에 mock이나 fake로 대체하기 어려운지
- 핵심 정책, 예외, 경계값을 검증하지 않는지

### 면접용 문장

```text
테스트는 코드가 실행되는지만 보는 것이 아니라 핵심 정책과 실패 흐름을 고정해야 합니다.
DI가 되어 있어야 Repository나 외부 의존성을 대체해서 Service 단위 테스트를 만들기 쉽습니다.
```

## 005번 풀이 순서

현재 005번은 Controller와 Service를 봤으므로 다음은 Repository를 보면 된다.

Repository에서는 특히 아래를 확인한다.

1. `findByOrderId`가 실패 시 `null`을 반환하는지
2. `findAll`이 내부 리스트를 그대로 반환하는지
3. `static final List`를 저장소처럼 쓰는지
4. `save`가 항상 추가만 해서 중복이나 갱신 문제가 생기는지
5. 조회 조건을 Repository에서 처리하지 못해 Service 반복 조회를 유발하는지

## 답변 작성 템플릿

```text
### 계층명

- 문제 유형:
- 왜 문제인지:

- 개선 방향:
- 면접용 문장:
```

## 계층별 조언 템플릿

사용자가 한 계층의 답변을 작성한 뒤 조언을 요청하면 아래 형식으로 정리한다.

```text
#### 조언

##### 1. 네가 잘 잡은 부분
- ...

##### 2. 표현을 고치면 좋은 부분
- 기존 표현:
- 면접에서 더 안전한 표현:

##### 3. 놓치기 쉬운 추가 관점
- ...

##### 4. 면접용 문장
...

##### 5. 다음 계층으로 넘어가기 전 체크
- ...
```
