# 21. 힌트 제공 가이드

이 문서는 사용자가 문제 풀이 중 힌트를 요청할 때, 정답을 바로 공개하지 않고 코드 리뷰 사고를 깊게 만들기 위한 기준이다.

목표는 사용자가 짧은 레거시 코드에서 문제를 직접 찾되, 막혔을 때 아래 흐름까지 생각하도록 돕는 것이다.

```text
무엇이 수상한가
-> 왜 문제인가
-> 어떤 장애가 생기는가
-> 어느 계층에서 어떻게 고칠 것인가
-> 면접에서 어떤 문장으로 말할 것인가
```

## 힌트 제공 원칙

- 정답 목록을 바로 공개하지 않는다.
- 사용자가 보고 있는 계층과 코드 위치에 맞춰 힌트를 준다.
- 사용자가 "힌트는 주지 말고 내가 보여달라고 할 때만 달라"고 한 경우, 사용자가 직접 요청하기 전에는 힌트를 먼저 제공하지 않는다.
- 사용자가 계층별 답변을 먼저 작성한 뒤 조언을 요청하면, 힌트가 아니라 `조언`, `표현 보정`, `면접용 문장` 형식으로 피드백한다.
- 힌트는 단계별로 깊어진다.
- 사용자가 더 깊이 물으면 장애 영향과 계층별 개선 방향까지 연결한다.
- 답변 제출 전에는 `solution/`, `refactoring-plan.md`, 전체 문제점 목록을 공개하지 않는다.

## 사용자 작성 후 계층별 조언 방식

사용자가 Controller, Service, Repository, Util, DTO/Entity, Exception, Test를 하나씩 작성하고 조언을 요청하면 아래 형식으로 해당 계층 아래에 붙인다.

```text
#### 조언

##### 1. 네가 잘 잡은 부분
- ...

##### 2. 표현을 고치면 좋은 부분
- 기존 표현:
- 면접에서 더 안전한 표현:

##### 3. 놓치기 쉬운 추가 관점
- ...

##### 4. 면접용 문장
...

##### 5. 다음 계층으로 넘어가기 전 체크
- ...
```

이 방식에서는 사용자가 작성하지 않은 다음 계층의 정답을 먼저 공개하지 않는다.
사용자가 "힌트 보여줘"라고 요청한 경우에만 힌트 단계 형식을 사용한다.

## 힌트 단계

### Level 1. 위치 힌트

어디를 의심해야 하는지만 알려준다.

예시:

```text
이 부분은 Controller가 Service를 어떻게 얻고 있는지 먼저 보세요.
호출 흐름 자체보다 객체 생성 책임을 누가 가지고 있는지가 포인트입니다.
```

### Level 2. 개념 힌트

관련 개념을 짧게 설명한다.

예시:

```text
Spring에서는 Controller가 Service를 직접 new로 만들기보다, Spring Bean으로 등록된 Service를 생성자 주입으로 받는 방식이 일반적입니다.
직접 생성하면 DI, 테스트 대체, AOP 적용에서 문제가 생길 수 있습니다.
```

### Level 3. 장애/실무 영향 힌트

실제로 어떤 장애나 유지보수 문제가 생기는지 묻고 설명한다.

예시:

```text
여기서 한 단계 더 보면, 테스트에서 Mock Service로 바꿔 끼우기 어렵고,
실제 Spring Bean에 붙은 @Transactional 같은 AOP 기능도 적용되지 않을 수 있습니다.
면접에서는 "직접 생성으로 결합도가 높아지고 테스트와 AOP 적용이 어려워진다"까지 말해야 합니다.
```

### Level 4. 계층별 개선 힌트

어느 계층에서 어떻게 고칠지 방향을 제시한다.

예시:

```text
Controller에서는 Service를 생성하지 말고 생성자 주입으로 받아야 합니다.
Service는 Repository를 생성자 주입으로 받고, 상태 변경 유스케이스는 Service 계층에서 트랜잭션 경계를 검토합니다.
```

### Level 5. 면접 답변 힌트

면접장에서 말할 수 있는 문장으로 정리한다.

예시:

```text
면접에서는 이렇게 말하면 됩니다.
"Controller가 Service를 직접 new로 생성하면 Spring DI를 사용하지 않는 구조가 되고, 테스트 대체와 AOP 적용이 어려워집니다. 생성자 주입으로 Bean을 주입받도록 바꾸는 것이 적절합니다."
```

## 힌트 요청별 응답 방식

### 사용자가 "이거 문제야?"라고 물을 때

바로 정답을 단정하지 말고 아래 순서로 답한다.

1. 맞는 방향인지 확인
2. 정확한 용어로 보정
3. 더 깊게 봐야 할 장애 영향 질문
4. 계층별 개선 방향

예시:

```text
방향은 맞습니다. 다만 "JPA N+1"이라고 단정하기보다는 "N+1 형태의 반복 Repository 조회"라고 말하는 게 안전합니다.
그다음에는 실제 DB라면 어떤 쿼리 수 증가가 생기는지, Repository에서 어떤 조건 조회로 바꿀지까지 말해보세요.
```

### 사용자가 "왜 문제야?"라고 물을 때

개념 설명에서 멈추지 말고 장애와 개선까지 연결한다.

```text
왜 문제인지:
실무에서 어떤 장애가 생기는지:
어느 계층에서 고칠지:
면접용 문장:
```

### 사용자가 "어떻게 고쳐?"라고 물을 때

정답 코드를 바로 길게 공개하지 않는다.
먼저 계층별 방향을 말하고, 필요할 때 짧은 코드 조각만 보여준다.

```text
Controller:
Service:
Repository:
DTO/Entity:
Exception:
Test:
```

## 반드시 유도할 질문

힌트를 줄 때 사용자가 아래 질문까지 생각하도록 돕는다.

1. 이 문제는 어떤 유형인가?
   - 명백한 안티패턴
   - 돌아가지만 위험한 코드
   - 개선하면 좋은 코드
   - 트레이드오프 판단
   - 리팩토링 방향 제안

2. 정확히 어떤 장애가 생기는가?
   - 데이터 정합성 문제
   - NPE
   - 중복 저장
   - 권한/소유자 검증 누락
   - 성능 저하
   - 테스트 불안정
   - 운영 추적 불가
   - FE/QA와 API 계약 불명확

3. 어느 계층에서 고칠 것인가?
   - Controller
   - Service
   - Repository
   - Util/Policy/Notifier
   - DTO/Entity
   - Exception
   - Test

4. 면접에서 어떤 문장으로 말할 것인가?

## 계층별 힌트 예시

### Controller

- 이 입력 타입이 외부 요청 DTO인가, 내부 모델인가?
- Controller가 비즈니스 판단을 하고 있지는 않은가?
- Service를 직접 생성하고 있지는 않은가?
- HTTP 응답 상태와 Body가 명확한가?

### Service

- 클라이언트가 보낸 상태를 신뢰하고 있지는 않은가?
- 기존 데이터를 조회한 뒤 상태를 바꾸는가?
- 소유자/권한/상태 전이 검증이 있는가?
- 실제 DB라면 트랜잭션 경계가 필요한가?
- Service가 HTTP 응답 문자열이나 `ResponseEntity`를 만들고 있지는 않은가?

### Repository

- 조회 실패를 `null`로 숨기고 있지는 않은가?
- 반복문 안에서 Repository 조회가 발생하는가?
- 내부 mutable collection을 그대로 반환하는가?
- 저장이 add인지 update인지 구분되는가?

### Util

- 순수 공통 함수인가, 도메인 정책인가?
- static 호출이라 Bean 주입 의미가 사라지지는 않는가?
- 정책, 계산, 알림 같은 서로 다른 책임이 섞여 있지는 않은가?

### DTO/Entity

- 요청/응답 DTO와 내부 모델이 분리되어 있는가?
- 서버가 관리해야 할 상태를 클라이언트가 보내고 있지는 않은가?
- validation이 있는가?
- public field로 상태가 아무 데서나 바뀌지는 않는가?

### Exception

- `RuntimeException` 문자열만 던지고 있지는 않은가?
- 커스텀 예외가 에러 코드와 연결되는가?
- 공통 예외 처리에서 HTTP 상태로 매핑 가능한가?

### Test

- 실제 실행되는 테스트인가?
- 성공 케이스만 있는가?
- 실패/경계 케이스가 있는가?
- static 공유 상태 때문에 테스트가 서로 영향을 주지 않는가?

## 금지할 힌트 방식

- 답변 제출 전에 전체 정답 목록을 공개하지 않는다.
- "이 코드는 이 문제들이 있어요"처럼 바로 문제를 나열하지 않는다.
- 개선 코드 전체를 먼저 보여주지 않는다.
- 사용자가 스스로 볼 기회를 갖기 전에 `solution/` 위치를 안내하지 않는다.

## 좋은 힌트의 형태

```text
맞는 방향입니다.
다만 면접에서는 이 표현을 더 정확히 해야 합니다.

1. 이 코드는 어떤 계층 책임이 깨졌는지
2. 실제로 어떤 장애가 생길 수 있는지
3. 어느 계층에서 어떻게 고칠지

이 세 가지까지 같이 말해보세요.
```
