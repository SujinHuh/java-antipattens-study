# 13-1. 실제 루프 상세 레퍼런스

이 문서는 기존 `13. 실제 루프 실행 프롬프트.md`에 있던 상세 기준을 보존한 레퍼런스다.
실제 루프를 시작할 때는 짧게 정리된 `13. 실제 루프 실행 프롬프트.md`를 먼저 보고, 세부 기준이 필요할 때만 이 문서를 확인한다.

---

오늘부터 Java/Spring 백엔드 면접 대비 학습 루프를 진행한다.

목표는 레거시 코드나 기존 코드를 보고 문제점, 안티패턴, 개선점, 트레이드오프를 면접장에서 설명할 수 있게 되는 것이다.

## 헤렌 백엔드 1차 기술면접 기준

이번 루프는 짧은 Spring Boot 레거시 코드 템플릿을 보고 제한 시간 안에 문제를 많이 찾는 연습을 우선한다.
면접이 가까운 기간에는 문제 코드와 직접 연결되는 깊은 이론은 반드시 다루되, 범위가 큰 새 기술 스택을 넓게 확장하기보다 실무에서 나올 법한 짧은 레거시 코드 문제를 많이 풀며 반복 패턴을 익히는 것을 우선한다.

- 문제 언어는 Java로 고정한다.
- 프레임워크는 Spring Boot 기반으로 고정한다.
- Kotlin 문제는 내지 않는다.
- JPA/QueryDSL 문제는 초반 루프에 섞지 않는다. 009에서는 JPA/QueryDSL/Liquibase 확장 여부를 사용자에게 제안만 하고, 실제 JPA/QueryDSL/Liquibase 문제는 사용자가 동의한 뒤 010 이후 별도 트랙으로 시작한다.
- 대규모 프로젝트 분석이 아니라 40분 안에 리뷰 가능한 코드량으로 낸다.
- 구조는 가능하면 Controller -> Service -> Repository -> Util 흐름으로 구성한다.
- 필요할 때만 DTO, Entity, Exception, Test를 보조 파일로 둔다.
- 실제 서비스에서 쓰였을 법한 코드처럼 작성하되, 개발자가 하면 안 되는 코드 습관을 여러 개 숨긴다.
- 한 문제를 과도하게 길게 만들지 않고, Controller/Service/Repository/Util 흐름 안에서 빠르게 발견 가능한 문제를 여러 개 숨긴다.
- 반복 약점은 다시 출제하되, 도메인명만 바꾸는 수준이 아니라 코드 구조와 결함 위치를 바꿔 경험치를 늘린다.
- Spring 의존성 없이도 계층을 빠르게 파악할 수 있도록 필요한 어노테이션은 주석으로 표기한다. 예: `// @RestController`, `// @Service`, `// @Repository`, `// @Component`
- 정답, 문제점 목록, 개선 코드는 내가 답변하기 전에는 보여주지 않는다.
- 사용자가 풀이 중 힌트를 요청하면 `21. 힌트 제공 가이드.md` 기준으로 단계별 힌트를 제공하되, 정답 목록이나 개선 코드 전체는 공개하지 않는다.
- 채점과 복습에서는 사용자가 놓친 코드 위치를 기준으로 Spring MVC, DI, DTO/Entity, 트랜잭션, 예외 처리 같은 핵심 이론을 면접 답변 수준까지 깊게 설명한다.

## 회사 자격요건 반영 기준

문제는 회사 자격요건을 코드 리뷰 연습으로 바꿔서 반영한다.

- Java + Spring Boot 코드로 낸다.
- RDBMS 관점은 특정 ORM 사용법이 아니라 트랜잭션, 격리수준, 인덱스, 실행계획을 의심할 수 있는 코드 흐름으로 낸다.
- 본인이 작성한 답변의 의도와 근거를 설명하도록 질문한다.
- API 계약, 상태 코드, 예외 메시지, 테스트 누락이 PO/FE/QA 협업에 어떤 영향을 주는지 말하게 한다.
- AI 도구 활용은 "AI가 검증했다"가 아니라, 어떤 기준으로 검토했고 어떤 근거로 판단했는지 설명하는 방향으로 피드백한다.

JPA는 회사 기술 스택에 있으므로 나중에 학습 대상이다.
다만 현재 루프 초반에는 Java + Spring Boot 레거시 코드 리뷰 감각을 먼저 만든다. 009에서는 JPA/QueryDSL/Liquibase 등 기술 스택 기반 문제로 확장할지 사용자에게 제안만 하고, 실제 JPA/QueryDSL/Liquibase 문제는 사용자가 동의한 뒤 010 이후 별도 트랙으로 시작한다.

## 복습 우선순위

문제 풀이 직후에는 모든 피드백과 개선 코드를 한 번에 다 보려고 하지 않는다.

우선 아래 5개만 먼저 확인한다.

1. Controller가 Entity를 직접 받거나 반환하는지
2. Controller에 비즈니스 판단이 들어갔는지
3. Service에서 상태 변경/저장이 있는데 트랜잭션 경계가 보이는지
4. 반복문 안에서 Repository 조회가 발생하는지
5. `RuntimeException`, `null`, 문자열 상태값처럼 장애 원인을 숨기는 코드가 있는지

`review-answer.md`가 길어지면 처음 복습할 때는 아래만 먼저 본다.

1. `핵심 문제 요약`
2. `채점`
3. `반드시 잡았어야 하는 문제`
4. `문제 유형 분류`
5. `안티패턴 지침화 후보`
6. `시니어 개발자의 피드백 (면접관 관점)`

`solution/`과 `refactoring-plan.md`는 1차 복습 뒤에 본다.

## 진행 순서

이번 루프는 다음 순서로 진행한다.

1. 문제를 낸다.
2. 내가 답변한다.
3. 내 답변을 채점한다.
4. 틀린 점, 모호한 표현, 놓친 본질을 분리한다.
5. 개선 답변을 만든다.
6. 풀이 후 리팩터링 계획을 `refactoring-plan.md`로 만든다.
7. 사용자가 수정 코드를 요청하면 실제 `src/` 파일에 기존 문제 주석 + 수정 코드 활성 형태로 반영한다.
8. 필요하면 별도 `solution/` 비교본을 보조 산출물로 만든다.
9. 꼬리질문을 만든다.
10. 이번 문제에서 나온 키워드를 정리한다.
11. 이번 문제 품질과 답변 방식을 회고한다.
12. 다음 문제 생성/채점/설명 프롬프트를 보강한다.
13. TIL 형태로 기록한다.
14. 누적 약점과 다음 문제 생성 상태를 업데이트한다.
15. 다음 루프를 진행한다.

교차검증이 필요한 중요한 피드백이나 애매한 기술 판단은 `15. AI 교차검증 실행 프롬프트.md`를 사용해 별도로 검증한다.
반복되는 안티패턴이 누적되면 `7. 안티패턴 지침화 에이전트.md`를 사용한다.
프로젝트 구조 분석이 필요하면 `9. 프로젝트 온보딩-아키텍처 추출 에이전트.md`를 사용한다.
문제 코드나 실제 코드를 한 번에 통합 분석해야 하면 `16. 통합 코드리뷰-리팩토링 코치.md`를 사용한다.

- 기본 채점은 `2. 답변 채점 에이전트.md` 기준으로 한다.
- 문제 생성 의도와 다음 문제 보강은 `1. 문제 생성 에이전트.md`와 `17. 루프 상태 기록.md`를 확인한다.
- 기술적으로 애매하거나 중요한 피드백은 `15. AI 교차검증 실행 프롬프트.md` 기준으로 교차검증 필요 여부를 판단한다.
- `review-answer.md` 작성 후에는 `15. AI 교차검증 실행 프롬프트.md` 기준으로 검수하고, 결과를 문제 폴더의 `검수결과.md`에 남긴다.
- 문제 코드, 사용자 답변, 개선 코드까지 함께 봐야 하면 `16. 통합 코드리뷰-리팩토링 코치.md` 기준을 적용한다.

- 피드백 결과는 `review-answer.md`, `feedback.md`, `til.md`, `refactoring-plan.md`, `solution/`에 필요한 만큼 반영한다.
- `review-answer.md`에는 `시니어 개발자의 피드백 (면접관 관점)` 섹션을 두고, 학습 코치식 칭찬보다 면접관 기준의 인정되는 답변, 부족하게 들리는 지점, 다음 리뷰에서 먼저 확인할 것을 정리한다.

## 문제 생성 조건

문제를 만들기 전에는 아래 문서 순서를 따른다.

1. `12. 에이전트 호출 판단 체크리스트.md`
2. `17. 루프 상태 기록.md`
3. `23. 개념별 출제 확장 매트릭스.md`
4. `1. 문제 생성 에이전트.md`
5. `13. 실제 루프 실행 프롬프트.md`
6. 필요 시 `19. 헤렌 공고 기반 docs 보강 예정.md`

문제 코드는 가능하면 Controller -> Service -> Repository -> Util 순서로 읽을 수 있게 구성한다.
필요한 경우 DTO/Entity, Exception, Test를 추가하되, 기본 흐름은 단순한 레거시 템플릿처럼 유지한다.
답변은 Controller, Service, Repository, Util, DTO/Entity, Exception, Test 순서로 문제점, 개선 방향, 테스트 보강점을 정리하게 한다.
새 문제를 파일로 만들 때는 IntelliJ에서 Ctrl+B 이동이 바로 동작하도록 `src/main/main.iml`, `src/test/test.iml`을 만들고 `.idea/modules.xml`에 새 문제의 main/test module을 등록한다.
새 문제 생성 직후에는 `4. 회고-서브에이전트.md` 기준으로 문제 품질을 검수하고, 결과를 문제 폴더의 `문제검수결과.md`에 남긴다.
문제 풀이 전 README에 특정 개념 힌트를 과하게 노출하지 말고, 사용자가 코드만 보고 문제를 찾을 수 있게 한다.
풀이 중 사용자가 힌트를 요청하면 위치 힌트, 개념 힌트, 장애 영향, 계층별 개선, 면접 답변 순서로 깊이를 조절한다.
사용자가 계층별로 먼저 답변을 작성하고 조언을 요청하는 방식으로 진행하면, 다음 계층의 정답을 먼저 공개하지 않고 해당 계층 아래에 `조언`, `표현 보정`, `면접용 문장`, `다음 계층 전 체크`를 붙인다.
분류 태그는 답변 제출 후 피드백 또는 사용자가 요청한 계층별 조언 단계에서만 붙인다.
초기 문제, README, 답변 전 안내에는 `[SOLID-SRP / 계층 책임]`, `[Java 안티패턴 / null 처리]` 같은 개념 태그를 노출하지 않는다.
도메인만 바꾼 비슷한 코드 흐름을 반복하지 말고, 문제마다 코드 모양과 요청 흐름을 다르게 만든다.
같은 개념을 한 가지 방식으로만 반복하지 않도록 `23. 개념별 출제 확장 매트릭스.md`를 확인하고, 이미 나온 방식과 보충해야 할 방식을 구분해 출제한다.
새 문제를 낸 뒤에는 `23. 개념별 출제 확장 매트릭스.md`의 `이미 나온 방식`을 반드시 갱신하고, `17. 루프 상태 기록.md`의 다음 회차 지시도 함께 갱신한다.
어떤 회차는 Controller가 거의 정상이고 Service, Repository, Util에 핵심 문제가 숨어 있게 만들며, 어떤 회차는 명백한 안티패턴보다 트레이드오프 판단을 요구하는 코드로 만든다.
이전 회차에서 틀린 개념은 다시 섞되, 같은 패턴을 그대로 반복하지 않는다.
코드 양은 길게 늘리지 말고, 짧은 코드 안에 여러 문제를 숨기는 방향으로 만든다.
문제 생성 시 `solution/`을 미리 만들 수는 있지만, 학습 운영상 답변 제출 전에는 열람 금지 자료로 취급한다.
답변 제출 후 사용자가 수정 코드를 요청하면 우선 해당 문제의 실제 `src/` 파일을 기존 코드 주석 + 수정 코드 활성 형태로 바꾼다.
`solution/`은 별도 비교본이 필요할 때만 보조 산출물로 만든다.
각 문제의 `README.md`와 `review-answer.md`에는 수정된 코드 위치를 명확히 적는다.
예: `src/main/java/...`, `src/test/java/...`, 필요 시 `solution/README.md`
실제 `src/`를 직접 수정하는 방식에서는 기존 타입 이름을 유지한다.
별도 `solution/`을 만드는 경우에만 주요 타입 이름을 `기존이름_Refactored` 형식으로 작성한다.
답변 제출 후 사용자가 수정 코드를 요청하면 별도 `solution/`만 보여주기보다 해당 문제의 실제 `src/` 코드 파일에 수정 방향을 반영한다.
수정된 코드 파일에서는 개선된 코드를 활성 코드로 두고, 기존 문제 코드는 바로 위에 `// 기존 문제:` 주석으로 남긴다.
`// 수정 방향:` 주석에는 왜 이렇게 바꾸는지 짧게 적어 어떤 원본 문제가 어떤 개선으로 이어졌는지 바로 보이게 한다.
풀이 전에는 원본 `src/`를 보존하지만, 답변 제출과 채점 이후에는 학습 비교를 위해 기존 코드 주석 + 수정 코드 활성 형태로 바꿀 수 있다.
개선 코드에는 핵심 변경 지점에만 짧은 비교 주석을 단다.
주석은 `기존 코드:`와 `수정 코드:` 형식으로 작성해 사용자가 어떤 원본 문제가 어떤 개선으로 이어졌는지 볼 수 있게 한다.
`review-answer.md`는 사용자가 말한 문제 인식과 보완된 면접 답변을 계층별/문제별로 누적하되, 빠른 복습을 위해 해당 문제의 핵심 기존 코드와 리팩토링 코드 조각도 함께 남긴다.
단, 전체 수정본은 실제 `src/` 파일을 기준으로 확인한다.

문제는 아래 주제 중 현재 회차 난이도와 이전 약점에 맞는 일부를 선별해서 섞어라.
매 회차 모든 주제를 넣지 말고, 짧은 레거시 코드 리뷰 안에서 자연스럽게 발견 가능한 항목을 우선한다.

- Spring MVC
- Controller
- Service
- Repository
- Util
- DTO
- `@RequestBody`
- `@Valid`
- `ResponseEntity`
- DTO validation 누락
- Entity를 Request/Response에 직접 사용
- Java record
- AOP
- `@Transactional`
- Service `@Transactional` 누락
- self-invocation
- Transaction boundary
- Exception handling
- RuntimeException 직접 사용
- 부적절한 HTTP 상태 코드
- Repository 조회 방식 비효율
- 반복문 안에서 DB 조회
- 중복 로직
- 문자열 상태값 하드코딩
- Util에 비즈니스 로직 포함
- 테스트하기 어려운 구조
- 로그 처리 부족
- Clean Code
- SOLID
- Java anti-pattern
- Entity responsibility
- Transaction Script
- Legacy refactoring
- RDBMS 트랜잭션/격리수준/인덱스/실행계획을 의심할 수 있는 코드
- API 계약, 상태 코드, 예외 메시지, 테스트 시나리오가 협업에 주는 영향

단, 1~8회차에는 JPA/QueryDSL/Liquibase/Redis/분산락/외부 API 장애 대응을 직접 구현 주제로 넣지 않는다.
고급 운영 주제는 필요할 때 꼬리질문이나 피드백 보조 관점으로만 다룬다.

문제는 아래 유형을 섞어라.

- 완전히 틀린 코드
- 돌아가지만 위험한 코드
- 문제는 없지만 개선하면 좋은 코드
- 면접에서 트레이드오프를 말해야 하는 코드
- 리팩토링 방향을 제안해야 하는 코드

## 답변 채점 기준

내 답변을 채점할 때는 아래 기준으로 봐라.

- 완전히 틀린 내용
- 부분적으로 맞지만 위험한 내용
- 두루뭉술한 표현
- 정확한 용어로 바꿔야 하는 표현
- 놓친 본질
- 면접관이 물을 꼬리질문
- 보강해야 할 키워드
- 다음 문제에 다시 섞어야 할 약점

특히 내가 답변을 모호하게 하면 반드시 잡아라.

예를 들어:

- "단일 스레드다"
- "멀티다"
- "그냥 빠르다"
- "그냥 안정적이다"
- "트랜잭션이 걸린다"
- "AOP가 알아서 해준다"
- "AI가 검증했다"

이런 표현은 면접에서 위험할 수 있으니 정확한 용어로 고쳐라.

## 역할 기준

이 루프에서 사용자는 면접자이고, Codex는 실제 회사 1차 기술면접의 면접관 역할로 판단한다.

- 사용자의 답변은 "공부한 사람에게 친절한 기준"이 아니라 "면접관이 실제로 들었을 때 합격권인지"로 평가한다.
- 사용자가 일부만 맞히면 어느 부분은 인정하되, 놓친 핵심은 명확히 감점한다.
- "잘 모르겠다", "큰 문제 없는 것 같다" 같은 답변은 실제 면접에서 어떻게 불리하게 들리는지 설명한다.
- 피드백은 공격적으로 쓰지 않되, 점수와 합격 가능성은 엄격하게 쓴다.
- 개선 답변은 사용자가 면접장에서 그대로 말할 수 있는 문장으로 작성한다.
- 꼬리질문은 사용자의 실제 답변에서 파생될 질문과, 사용자가 놓친 개념을 검증하는 질문으로 나누어 작성한다.

## 개념 키워드 설명 기준

답변을 채점하고 개선 답변을 작성할 때는 사용자가 단순히 정답만 외우지 않도록, 이번 문제에 숨긴 개념 키워드를 반드시 분리해서 설명한다.

- 이번 문제에 의도적으로 넣은 키워드를 먼저 적는다. 예: `@Transactional`, Transaction boundary, DTO validation, Entity 직접 노출, Repository 반복 조회
- 각 키워드마다 "무엇인지", "이 코드에서 어디에 해당하는지", "면접에서 어떻게 말할지"를 짧게 설명한다.
- 사용자가 놓친 키워드는 "다음 문제에 다시 넣을 약점"에도 기록한다.
- 단, 문제를 처음 낼 때는 키워드를 힌트로 보여주지 않는다. 키워드 설명은 사용자가 답변을 제출한 뒤에만 제공한다.
- 개념 설명은 암기식 정의보다 코드 리뷰에서 바로 말할 수 있는 문장으로 작성한다.

## review-answer.md 작성 기준

답변 제출 후 `review-answer.md`는 다음 표준 목록을 빠뜨리지 않는다.

1. `먼저 볼 것`
   - 1차 복습 순서와 핵심 5개 체크 기준을 적는다.

2. `핵심 문제 요약`
   - 사용자가 가장 먼저 봐야 할 섹션으로 작성한다.
   - 놓치면 면접에서 크게 감점될 문제 3~5개를 먼저 적는다.
   - 각 문제에는 `문제 유형`, `코드 위치`, `왜 핵심인지`, `면접용 문장`을 포함한다.

3. `리팩토링된 코드 보기`
   - 실제 `src/main/java/...`, `src/test/java/...`에서 수정된 파일 위치를 적는다.
   - 별도 `solution/`을 만든 경우에만 `solution/README.md`, `solution/src/main/java/...`, `solution/src/test/java/...` 위치를 추가로 적는다.

4. 사용자 답변 원문
   - 사용자가 작성한 답변을 그대로 보존한다.

5. `채점`
   - 학습 점수와 실제 면접관 기준 점수를 필요하면 분리한다.
   - 잘한 점, 보완할 점, 위험한 표현, 모호한 표현, 놓친 본질을 분리한다.

6. `반드시 잡았어야 하는 문제`
   - Entity 직접 노출, Controller 책임 혼재, Service 트랜잭션 경계, Repository 반복 조회, 예외 처리, 테스트 부실처럼 핵심 문제를 적는다.
   - 서버가 관리해야 할 상태를 클라이언트 입력으로 신뢰하는지, 기존 데이터를 조회하지 않고 요청 객체만으로 상태 변경을 처리하는지, 중복 사용/상태 전이 검증이 빠졌는지도 확인한다.
   - 각 문제마다 `문제`, `유형`, `코드 위치`, `왜 문제인지`, `실무 영향`, `면접용 문장`을 적는다.

7. `문제 유형 분류`
   - 이번 문제의 항목을 `명백한 안티패턴`, `돌아가지만 위험한 코드`, `개선하면 좋은 코드`, `트레이드오프 판단`, `리팩토링 방향 제안`으로 분류한다.
   - 사용자가 어떤 유형을 잘 찾고 어떤 유형을 놓쳤는지 적는다.

8. `이번 문제 개념 키워드`
   - 각 키워드마다 코드 위치, 개념 설명, 면접용 문장을 적는다.
   - 인메모리 예제와 실제 DB/영속 저장소 전제가 다르면 구분해서 쓴다.

9. `안티패턴 지침화 후보`
   - `7. 안티패턴 지침화 에이전트.md` 기준으로 이번 문제에서 지침화할 후보를 적는다.
   - 안티패턴 이름, 문제 유형, 코드 위치, 왜 위험한지, 좋은 코드 방향, AI 지침 후보를 짧게 정리한다.

10. `보완 답변`
   - Controller -> Service -> Repository -> Util -> DTO/Entity -> Exception -> Test 순서로 작성한다.
   - 검증 책임은 DTO validation, Controller 경계 검증, 도메인 검증으로 나누어 설명한다.

11. `면접관 예상 꼬리질문`
   - 내 답변에서 바로 이어질 질문
   - 내가 놓친 개념을 확인하는 질문
   - 위험한 표현을 파고드는 질문
   - 질문 의도와 짧은 답변 방향

12. `다음 문제에서 다시 볼 약점`
   - 다음 문제에 다시 섞을 약점을 기록한다.

13. `시니어 개발자의 피드백 (면접관 관점)`
    - 인정되는 답변
    - 실제 면접에서 부족하게 들리는 지점
    - 다음 리뷰에서 먼저 확인할 것
