# 23. 개념별 출제 확장 매트릭스

이 문서는 문제를 낼 때 같은 개념을 한 가지 형태로만 반복하지 않도록 관리하는 출제 기준이다.

목표는 금요일 면접 전까지 짧은 Java/Spring Boot 레거시 코드를 많이 보면서도, 각 개념을 다양한 코드 모양과 깊이로 반복 학습하는 것이다.

## 사용 원칙

- 면접 전 마지막 날에는 남은 풀이 횟수가 2~3회뿐이므로, 한 문제에 자주 나오는 경우의 수를 더 많이 압축한다. 기본 과밀 방지 기준보다 반복 노출과 문서화가 우선이다.
- 마지막 날 문제는 핵심 리뷰 포인트를 7~9개까지 허용하되, README와 문제 코드에는 정답 키워드를 직접 노출하지 않는다.
- 테스트 코드는 더 이상 단순 성공 호출만 두지 않는다. 최소한 상태 변화 assertion, 실패 케이스, 중복 요청, 테스트 격리 문제 중 2개 이상이 보이게 만든다.
- 이미 나온 방식은 다시 낼 수 있지만, 도메인명만 바꾸는 반복은 피한다.
- 같은 개념을 다른 계층, 다른 코드 모양, 다른 실무 장애로 바꿔 출제한다.
- 사용자가 약하게 답한 개념은 다음 문제에 다시 넣되, 같은 줄/같은 패턴으로 숨기지 않는다.
- 문제 코드와 직접 연결되는 깊은 이론은 반드시 설명한다.
- 1~8회차에는 JPA/QueryDSL/Liquibase/Kafka/AWS/Redis/분산락/외부 API 장애 대응을 직접 구현 주제로 넣지 않고, 9회차 전후에 확장 제안한다.
- 기본적으로 한 문제의 핵심 출제 개념은 3~5개로 제한하고, 나머지는 보조 관점이나 꼬리질문으로 분리한다.
- 단, 006~015 빠른 반복 구간에서는 사용자가 많은 개념을 눈에 익히는 것이 우선이므로 핵심 리뷰 포인트를 6~7개까지 허용한다. 이때도 새 개념은 1~2개로 제한하고, 나머지는 반복 약점이나 보조 관점으로 둔다.
- 문제 출제 후에는 이 문서의 `이미 나온 방식`과 `보충해야 할 방식`을 반드시 갱신한다.

## 문제 출제 후 갱신 규칙

새 문제를 낸 뒤에는 아래 순서로 이 문서를 반드시 갱신한다.

1. 이번 문제에 실제로 숨긴 핵심 개념을 확인한다. 기본은 3~5개, 006~015 빠른 반복 구간은 6~7개까지 허용한다.
2. 해당 개념의 `보충해야 할 방식`에서 실제로 출제한 항목을 회차 태그와 함께 `이미 나온 방식`으로 옮기거나 추가한다.
3. 사용자가 놓쳤거나 표현이 약했던 방식은 `보충해야 할 방식`에 유지하거나 더 구체화한다.
4. 아직 출제하지 않은 변형은 `보충해야 할 방식`에 남긴다.
5. 새로 떠오른 변형은 `개념별 출제 아이디어 풀`에 추가한다.
6. 다음 문제 생성 전에는 `이미 나온 방식`과 겹치는 경우 코드 모양, 계층, 업무 흐름, 장애 영향 중 최소 1개를 바꾼다.
7. `17. 루프 상태 기록.md`에도 이번 회차에서 나온 방식과 다음 회차 보충 방식을 함께 기록한다.

갱신 예시:

```text
006에서 Optional 필드/파라미터 오남용을 냈다면
Optional의 이미 나온 방식에 `006: 필드/파라미터 Optional 오남용`을 추가한다.
보충해야 할 방식에는 `orElse vs orElseGet`, `Optional<List<T>>`, `optional.get()`을 남긴다.
```

## 출제 변형 축

출제 아이디어를 늘릴 때는 단순히 사례를 계속 추가하지 말고 아래 축을 바꾼다.

- 계층 변형: Controller, Service, Repository, Util, DTO/Entity, Exception, Test
- 코드 모양 변형: private helper, stream, Optional, try-catch, mapper, enum, wrapper DTO, static factory
- 업무 흐름 변형: 생성, 조회, 수정, 취소, 상태 전이, 검색, 정산, 이력 조회
- 판단 난이도 변형: 명백한 안티패턴, 돌아가지만 위험한 코드, 개선하면 좋은 코드, 트레이드오프, 리팩토링 우선순위

## 과밀 방지 기준

한 문제에는 아래 기준을 적용한다.

- 핵심 결함: 기본 3개, 006~015 빠른 반복 구간은 4~5개까지 허용
- 보조 결함: 기본 2개 이하, 006~015 빠른 반복 구간은 전체 리뷰 포인트가 6~7개를 넘지 않게 조절
- 꼬리질문 후보: 2개 이하
- 새 개념: 최대 1개
- 반복 개념: 2~4개
- SOLID는 한 문제에서 전체를 묻지 않고 SRP/OCP/DIP 중 1~2개만 선택한다.
- Transaction/AOP와 외부 호출 순서 문제는 1~8회차에는 실제 외부 API 장애 대응 구현이 아니라 가짜 의존성이나 주석 수준의 호출 순서 판단으로만 다룬다.

## 개념별 출제 아이디어 풀

아래 아이디어는 한 문제에 모두 넣지 않는다.
문제 하나에는 기본적으로 핵심 3~5개만 선택하고, 나머지는 다음 회차나 꼬리질문으로 넘긴다.
다만 006~015 빠른 반복 구간에서는 핵심 리뷰 포인트를 6~7개까지 허용하되, 코드 길이와 답변 부담이 40분 범위를 넘지 않게 한다.

### Controller

- `POST` 생성인데 200 OK만 반환하고 생성 위치/식별자를 알려주지 않는 코드
- 실패를 `Map.of("success", true)`로 감싸는 코드
- Controller가 요청 DTO를 수정해 서버 계산값을 넣는 코드
- Controller가 권한/소유자 검증을 직접 하는 코드
- Controller가 try-catch로 모든 예외를 200 응답으로 바꾸는 코드
- Controller는 얇지만 Response DTO가 없어 FE/QA가 계약을 알기 어려운 코드

### Service

- 여러 정책, 저장, 알림, 응답 조립을 한 메서드에서 모두 처리하는 코드
- private helper가 많아서 책임 분리가 된 것처럼 보이지만 실제로는 한 Service에 정책이 몰린 코드
- 조회 실패 시 새 Entity 생성, 기본값 반환, 빈 응답 반환으로 실패를 숨기는 코드
- 클라이언트가 보낸 status, amount, ownerId를 서버 상태처럼 신뢰하는 코드
- Service가 `Map`, 문자열, `ResponseEntity` 등 HTTP 응답 구조를 반환하는 코드
- 상태 변경 후 저장은 하지만 트랜잭션 경계와 실패 처리 순서가 애매한 코드

### Repository

- `findAll()` 후 Service에서 userId/status/date를 필터링하는 코드
- 반복문 안에서 `findById`, `findByOrderId`, `existsBy...`를 호출하는 코드
- 조회 실패를 `null`로 반환해 Service가 정상 흐름으로 숨기게 만드는 코드
- 내부 mutable collection을 그대로 반환하는 코드
- `save`가 생성/수정/중복 기준을 구분하지 않고 항상 add만 하는 코드
- Repository에서 할인 가능 여부, 상태 전이 가능 여부 같은 정책을 판단하는 코드
- 실제 DB라면 조건 조회, 인덱스, 페이징을 의심해야 하는 목록/검색 코드

### Util / Policy / Component

- 순수 문자열/날짜 변환처럼 보이지만 도메인 정책을 판단하는 Util
- 알림, 로그, 외부 전달을 static Util로 처리해 테스트 대체가 어려운 코드
- `isValid`, `calculateFee`, `sendNotice`가 한 Util에 섞인 코드
- 공통 Util이라는 이름으로 여러 도메인 정책이 섞여 변경 영향이 커진 코드
- Util로 둘지 Policy/Validator/Notifier Component로 분리할지 트레이드오프가 있는 코드

### DTO / Entity

- Request DTO에 서버가 계산해야 할 최종 금액, 상태, 권한, 생성일시가 들어 있는 코드
- Response DTO 없이 Entity를 그대로 반환하는 코드
- Response DTO가 너무 많은 내부 필드를 노출하는 코드
- record DTO를 무조건 정답처럼 쓰지만 validation/변환 책임이 애매한 코드
- DTO -> Entity 변환을 Controller가 직접 하거나, Entity static factory가 요청 DTO에 과하게 의존하는 코드
- public field로 어디서든 상태를 바꿀 수 있는 Entity
- Entity가 너무 빈약해 모든 상태 변경 규칙이 Service if문에 몰린 코드
- Entity가 반대로 알림/외부 호출/응답 조립까지 담당하는 코드

### Validation

- `@Valid`가 빠져 형식 검증이 Controller 전에 걸리지 않는 코드
- DTO null 검증은 있지만 도메인 상태 전이 검증이 빠진 코드
- Bean Validation으로 처리할 수 없는 소유자/상태/기간 검증을 DTO 어노테이션으로 억지 처리한 코드
- Controller, Service, Util에 같은 검증 로직이 중복된 코드
- 검증 실패와 조회 실패를 같은 예외/같은 메시지로 처리하는 코드

### Optional

- Repository가 `Optional<T>`를 반환하지만 Service에서 바로 `get()`을 호출하는 코드
- `Optional`을 Entity 필드나 DTO 필드로 사용하는 코드
- 메서드 파라미터를 `Optional<T>`로 받아 호출부를 복잡하게 만드는 코드
- 값이 없으면 비용 큰 기본 객체를 만드는 데 `orElse`를 사용한 코드
- `Optional<List<T>>`로 빈 목록과 조회 실패 의미를 혼동시키는 코드
- Optional을 쓰지만 결국 `isPresent()` 후 `get()`으로 null 체크와 다를 바 없는 코드

### Exception

- `RuntimeException("invalid")`처럼 원인과 처리 방법이 드러나지 않는 코드
- catch 후 로그 없이 성공 응답이나 빈 결과로 바꾸는 코드
- 400, 404, 409가 구분되어야 하는데 모두 같은 예외로 처리하는 코드
- 커스텀 예외는 있지만 에러 코드와 HTTP 매핑이 없는 코드
- 예외 메시지에 개인정보나 내부 구현 정보가 노출되는 코드
- Service마다 예외 처리 방식이 달라 API 실패 계약이 흔들리는 코드

### Transaction / AOP

- 상태 변경과 저장이 여러 번 있는데 트랜잭션 경계가 보이지 않는 코드
- 조회만 하는 메서드에 과도하게 넓은 트랜잭션이 걸린 코드
- 트랜잭션 안에서 알림/외부 호출을 수행해 실패 순서가 애매한 코드
- 같은 클래스 내부 메서드에 `@Transactional`을 붙이고 self-invocation 효과를 기대하는 코드
- `@Async`, cache, security 같은 AOP 기반 기능도 내부 호출에 기대하는 꼬리질문
- 트랜잭션이 필요한 이유를 "그냥 안정적"이라고 말하기 쉬운 코드

### SOLID / Clean Code

- SRP: Controller/Service/Repository/Util이 서로의 책임을 대신하는 코드
- OCP: 상태/타입이 추가될 때마다 if/switch를 계속 수정해야 하는 코드
- DIP: Service가 구체 Repository/Notifier/Policy 구현을 직접 생성하는 코드
- ISP: 하나의 큰 인터페이스가 필요 없는 메서드까지 강제하는 코드
- LSP: 초반에는 직접 출제보다 상속 사용 코드가 나왔을 때 꼬리질문으로만 다룬다.
- Clean Code: 의미 없는 이름, 중복 조건, 매직 넘버, 깊은 중첩, 부정 조건 남발 코드

### Test / 운영 / 협업

- 테스트 메서드가 있지만 `@Test`나 assertion이 없어 검증이 안 되는 코드
- 성공 케이스만 있고 조회 실패, 권한 실패, 중복 요청, 상태 불가 케이스가 없는 코드
- static 저장소 때문에 테스트 간 데이터가 섞이는 코드
- `LocalDateTime.now()`나 랜덤값을 직접 호출해 테스트가 불안정한 코드
- 외부 의존성을 직접 `new`로 만들어 fake/mock 대체가 어려운 코드
- 취소 가능 시간, 만료 시간, 생성/수정 시간을 서버 기준으로 검증하지 않는 코드
- 요청의 userId를 그대로 믿고 저장된 소유자와 비교하지 않는 코드
- 실패를 200으로 반환해 FE/QA/운영 지표가 실패를 감지하기 어려운 코드
- 로그가 없거나 `System.out.println`만 있어 원인 추적이 어려운 코드
- API 응답 필드가 매번 달라 FE/QA가 계약을 잡기 어려운 코드
- 리팩토링할 문제가 많을 때 먼저 고칠 2~3개를 고르지 못하게 만드는 코드

## 수진님 현재 약점과 반복 복습 대상

아래 항목은 현재 문서와 001~005 회차 기준으로 반복 보강이 필요한 부분이다.
새 문제를 만들 때는 이 표에서 2~4개를 반복 개념으로 섞고, 새 개념은 최대 1개만 추가한다.

| 개념 | 아직 보충 필요한 것 | 반복 출제 방식 |
| --- | --- | --- |
| Optional | `orElse` vs `orElseGet`, `optional.get()` 직접 호출 위험성 | Repository 조회, Service 기본값 처리, DTO/파라미터 오남용으로 변형 |
| SOLID - OCP | if/switch가 계속 늘어나는 코드 -> 전략/정책 분리 | 상태/타입/할인/배송 방식 분기 코드로 반복 |
| Transaction | readOnly, 외부 호출과 트랜잭션이 섞인 순서 문제, `@Transactional`의 DB 롤백 범위와 외부 API 비롤백 구분 | 저장 후 알림, 상태 변경 후 예외, 조회 전용 메서드, PG/API 호출 순서로 반복 |
| 시간/상태 변경 | 서버 시간 기준 만료 처리, 취소 가능 시간 | 만료 쿠폰, 예약 취소, 배송 변경 가능 시간으로 반복 |
| 권한/소유자 검증 | 요청 userId와 저장된 ownerId를 비교하지 않는 문제 | 환불, 주소 변경, 쿠폰 사용, 예약 취소로 반복 |
| Validation | `@Valid` 누락, Bean Validation으로 부족한 도메인 검증 | 형식 검증과 도메인 검증이 섞인 요청으로 반복 |
| HTTP 상태코드 | 200/204/400/404/409 구분 기준, 성공 응답과 실패 응답의 책임 | 취소/삭제/수정 API에서 응답 바디 유무와 실패 원인별 상태코드로 반복 |
| 로직 흐름 추적 | Repository 조회 결과를 Service가 어떻게 처리하는지, 요청값/저장값/외부호출/저장 순서 연결 | 조회 실패 후 기본값 생성, 요청 금액 신뢰, 외부 호출 후 저장, 상태 변경 누락 흐름으로 반복 |

면접 대비로 당장 챙길 우선순위는 아래와 같다.

1. 조회 실패를 어떻게 표현하고 처리하는지: `null`, `Optional`, 예외 흐름
2. 요청값을 어디까지 신뢰하면 안 되는지: userId, status, amount, time
3. 상태 변경 유스케이스의 경계가 어디인지: Service, Transaction, 저장/알림 순서
4. 분기 코드가 늘어날 때 어떤 설계 문제가 생기는지: OCP, Policy, Strategy
5. DTO 검증과 도메인 검증을 어디서 나눌지: `@Valid`, Service/Domain validation
6. HTTP 상태코드를 어떻게 고르는지: 성공 바디 있음/없음, 잘못된 요청, 리소스 없음, 상태 충돌
7. 로직 흐름을 어떻게 따라갈지: 입력값 -> 조회 -> 검증 -> 계산 -> 외부 호출 -> 저장 -> 결과 반환

## 006~015 운영 방식

006~015는 회차별로 주제를 딱 나누지 않는다.
이 문서의 전체 출제 매트릭스에서 매 문제마다 기본 3~5개를 고르되, 006~015 빠른 반복 구간에서는 핵심 리뷰 포인트를 6~7개까지 허용한다.
아래 목록은 고정 범위가 아니라 빠른 반복을 위한 우선순위 예시다.
아래 목록에 없더라도 이 문서의 출제 매트릭스에 있거나 직전 채점에서 드러난 약점이면 출제할 수 있다.

- Controller 책임, 상태코드, Response DTO, `Map<String, Object>` 응답
- Service 책임, Transaction Script, HTTP 응답 책임 침범
- Repository 조건 조회, `null` 반환, 내부 컬렉션 노출, 반복 조회
- Optional 오남용, `optional.get()`, `orElse` vs `orElseGet`
- 예외 흐름, `RuntimeException`, 400/404/409 구분, 공통 예외 매핑
- DTO/Entity 분리, 서버 결정값을 클라이언트 입력으로 받는 문제
- Validation, `@Valid`, Bean Validation으로 부족한 도메인 검증
- SOLID 중 SRP/OCP/DIP
- Transaction boundary, readOnly, 외부 호출과 트랜잭션 순서
- 권한/소유자 검증, 요청 userId 신뢰 문제
- 시간/상태 변경, 서버 시간 기준 만료/취소 가능 시간
- Util/Policy/Component 경계
- 테스트 가능성, 실패 케이스 누락, static 저장소 테스트 격리 문제
- 그 외 매트릭스 개념: Java record/DTO 설계, Mapper/변환 책임, AOP/self-invocation, 문자열 상태값, Entity responsibility, Logging/Monitoring, 협업/API 계약, 리팩토링 우선순위 등

016 이후는 같은 개념을 더 깊게 반복한다.

- Controller가 거의 정상이고 Service/Repository에 문제가 숨어 있는 문제
- Service가 그럴듯하지만 트랜잭션/외부 호출 순서가 애매한 문제
- Repository가 단순해 보이지만 조회 조건, 페이징, 정렬이 빠진 문제
- Util이 순수 유틸처럼 보이지만 정책과 알림 책임을 가진 문제
- DTO/Entity가 겉보기에는 깔끔하지만 서버 결정값을 클라이언트에서 받는 문제
- 테스트가 존재하지만 핵심 실패 케이스를 고정하지 못하는 문제

## 출제 전 체크

새 문제를 만들기 전에 아래를 확인한다.

1. 이전 문제와 같은 코드 모양을 반복하고 있지 않은가?
2. 같은 개념이라도 다른 계층 또는 다른 장애 영향으로 출제했는가?
3. 사용자가 최근 모호하게 말한 표현을 다시 검증할 수 있는가?
4. 문제 코드와 직접 연결되는 깊은 이론을 피드백에서 설명할 수 있는가?
5. 30~40분 안에 읽고 답변할 수 있는 코드량인가?
