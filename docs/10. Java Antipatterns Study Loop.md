# 10. Java Antipatterns Study Loop

## 목적

Java 안티패턴과 실무 판단력을 반복 문제 풀이 방식으로 훈련한다.

## 이 루프를 통해 얻는 것

이 루프의 목적은 단순히 Java/Spring Boot 지식을 암기하는 것이 아니라, 반복 답변과 피드백을 통해 다음 항목을 축적하는 것이다.

- 내가 자주 틀리는 안티패턴
- 내가 모호하게 말하는 표현
- 면접에서 위험한 답변 습관
- Spring MVC / AOP / Transaction / Java 기본기의 빈틈
- 나중에 AI에게 줄 수 있는 자바용 하네스/코딩 지침
- 면접에서 "저는 AI를 이렇게 학습 루프에 적용했습니다"라고 말할 수 있는 근거

## 진행 방식

1. 문제 생성 에이전트가 코드/상황 문제를 낸다.
2. 수진님이 직접 답변한다.
3. 답변 채점 에이전트가 답변을 채점한다.
4. 틀린 점, 모호한 표현, 놓친 본질을 분리한다.
5. 설명 에이전트가 개선 답변을 만든다.
6. 풀이 후 리팩터링 계획을 `refactoring-plan.md`로 만든다.
7. 개선 코드 예시를 `solution/` 아래에 만든다.
8. 원본 코드와 개선 코드의 차이를 비교한다.
9. 면접관이 물을 수 있는 꼬리질문을 만든다.
10. 이번 문제에서 나온 핵심 키워드를 정리한다.
11. 프롬프트 개선 에이전트가 다음 문제 생성 프롬프트를 보강한다.
12. TIL/로그 에이전트가 문제, 답변, 피드백, 보강 키워드를 기록한다.
13. 누적 약점과 다음 문제 생성 상태를 업데이트한다.
14. 필요하면 교차검증 프롬프트로 피드백을 검수한다.
15. 다음 루프를 진행한다.

이 루프를 최소 20회 이상 반복한다.

## 복습 우선순위

한 회차가 끝난 뒤 모든 개념과 산출물을 한 번에 외우려고 하지 않는다.

처음에는 아래 5개를 안정적으로 찾는 것을 목표로 한다.

1. Controller가 Entity를 직접 받거나 반환하는지
2. Controller에 비즈니스 판단이 들어갔는지
3. Service에서 상태 변경/저장이 있는데 트랜잭션 경계가 보이는지
4. 반복문 안에서 Repository 조회가 발생하는지
5. `RuntimeException`, `null`, 문자열 상태값처럼 장애 원인을 숨기는 코드가 있는지

`review-answer.md`가 길면 아래 순서만 먼저 보고, `solution/`과 `refactoring-plan.md`는 나중에 본다.

1. `핵심 문제 요약`
2. `채점`
3. `반드시 잡았어야 하는 문제`
4. `문제 유형 분류`
5. `안티패턴 지침화 후보`
6. `시니어 개발자의 피드백 (면접관 관점)`

## 문제 운영 규칙

- 문제는 실제 회사 면접에서 받을 수 있는 레거시 코드 리뷰처럼 만든다.
- 이전 회차 약점은 다음 문제에 다시 섞되, 같은 코드 흐름과 같은 문제 위치를 반복하지 않는다.
- README에는 특정 개념 힌트를 과하게 노출하지 않고, 사용자가 코드만 보고 문제를 찾게 한다.
- 풀이 전에는 원본 `src/` 코드만 보고, `refactoring-plan.md`와 `solution/`은 답변 제출과 채점 이후에 본다.
- 개선 코드 예시는 유일한 정답이 아니라 가능한 개선 방향으로 작성한다.
- 개선 코드 주요 타입 이름은 `기존이름_Refactored` 형식으로 작성한다.
- 개선 코드에는 핵심 변경 지점에만 `기존 코드:` / `수정 코드:` 비교 주석을 단다.
