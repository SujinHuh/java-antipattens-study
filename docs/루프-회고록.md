## ✍️ 회차별 상세 회고록

### 📌 면접 전 마지막 날 운영 기준
* **작성 일시**: 2026-07-01
* **상황**: 면접 전 마지막 날이므로 새 문제를 많이 늘리기보다, 2~3문제 안에 최대한 많은 실무 경우의 수를 압축하고 문서화해서 반복 복습한다.

#### [출제 방식 전환]

* 앞으로 남은 문제는 "한 개념을 깊게 하나"보다 "면접에서 자주 묻는 경우의 수를 한 문제 안에서 많이 마주치게 하는 것"을 우선한다.
* 단, 정답 주석이나 개념명을 문제 코드에 직접 노출하지 않는다.
* README는 상황과 읽는 순서만 제공하고, 개념 키워드는 답변 후 문서나 피드백에서 정리한다.
* 한 문제의 핵심 리뷰 포인트는 7~9개까지 허용한다. 마지막 날에는 과밀 방지보다 반복 노출이 더 중요하다.
* 테스트 코드도 더 이상 단순 성공 호출만 두지 않는다. 최소한 상태 변화, 실패 케이스, 중복 요청, 경계값 중 2개 이상은 코드에 보이게 한다.

#### [남은 2~3문제에서 반드시 눈에 익힐 축]

* **동시성/RDBMS 정합성**: 재고 차감, 쿠폰 발급, 중복 요청, Race Condition, 낙관적 락/비관적 락, 격리수준.
* **트랜잭션/예외 전파**: self-invocation, checked exception 롤백 누락, catch 후 성공/보류 응답, 외부 호출과 DB 저장 순서.
* **멱등성**: idempotency key, 같은 사용자 중복 요청, 재시도 시 같은 결과를 돌려줘야 하는 흐름.
* **API 계약/상태코드**: 실패를 200 계열 성공 DTO로 감싸는 문제, 400/404/409 구분, Response DTO 필드 노출 범위.
* **DTO/Entity/Validation**: 클라이언트가 보내면 안 되는 수량/상태/버전/권한/시간값, `@Valid`와 도메인 검증의 분리.
* **Repository/성능**: 조건 조회, 페이징, 정렬 안정성, 인덱스 의심, 전체 조회 후 애플리케이션 필터링.
* **Test**: 단일 성공 테스트만 있는지, 동시 요청/중복 요청/실패 케이스/테스트 격리가 빠졌는지.

#### [007 문제에 기대하는 역할]

007은 `재고 차감 / 선착순 쿠폰 발급` 문제로, 최근 004~006의 단순 상태 변경 반복에서 벗어나 아래 경우의 수를 한 번에 보게 하는 문제다.

* 재고 수량 조회 후 차감하는 read-modify-write 흐름.
* 다중 요청에서 초과 발급 또는 음수 재고가 생길 수 있는 흐름.
* 같은 사용자 중복 발급과 idempotency key 재시도 처리가 섞인 흐름.
* `@Transactional`이 내부 메서드 호출에서는 기대처럼 적용되지 않을 수 있는 구조.
* 실패를 예외로 드러내지 않고 `PENDING`, `FAILED`, `INACTIVE`, `NOT_OPEN` 같은 성공 응답처럼 돌려주는 구조.
* 이벤트/외부 알림 발행과 DB 저장을 한 try-catch 안에서 처리하는 구조.
* 테스트는 일부 assertion이 있지만, 동시 요청/격리/idempotency key 기준 재시도 검증이 빠진 구조.

### 📌 004 ~ 006회차 통합 회고 (중요 피드백 반영)
* **작성 일시**: 2026-07-01
* **평가 대상**: 004. Point Refund Legacy, 005. Delivery Address Legacy, 006. Order Cancel Legacy

#### [문제 품질 및 구조적 한계 평가]

1. **계층 아키텍처의 고착화**
   * 4, 5, 6번 문항 모두 `Controller -> Service -> Repository -> Util` 형태의 고정된 파일 템플릿과 명명 스펙을 유지함.
   * 세 문제 모두 상태 변경 유스케이스다. 004는 포인트 환불, 005는 배송지 변경, 006은 주문 취소로 도메인은 다르지만 요청값 신뢰, 조회 실패 처리, 상태 변경, 저장, 응답 DTO를 찾는 사고 흐름이 거의 같다.
   * 비즈니스 로직을 Anemic Domain Model 방식으로 Service 계층에 절차지향적으로 구현하여 구조적 참신함이 부족했다.

2. **안티패턴 중복률 과다**
   * **Controller**: Entity를 API 응답으로 직접 유출하거나 `Map<String, Object>` 형태의 무타입 응답 노출.
   * **Service**: Repository/Util을 직접 생성하여 강결합을 만들고, 요청 DTO 값을 서버 상태처럼 신뢰함.
   * **Repository**: 인메모리 수정 기능에서 `ArrayList.add`를 활용해 데이터 정합성이 깨지는 수정 버그 유발.
   * **Util**: static 메서드 안에 도메인 정책, 외부 호출, 감사 로그가 섞임.
   * **Test**: `@Test`와 assertion이 없거나, 성공 호출만 있고 실패 케이스가 없음.

3. **문제 코드와 해설 코드 혼재**
   * 004는 레거시 문제 형태를 유지하고 있어 사용 가능하다.
   * 005는 `src/main/java`에 이미 리팩토링된 코드와 "기존 문제", "수정 방향" 주석이 들어 있다. 학습자가 문제를 찾는 구조가 아니라 정답을 읽는 구조가 되었다.
   * 005는 `DeliveryAddress.java`에 `private String memo;나` 오타가 있어 컴파일도 깨진다.
   * 006은 도메인과 의도는 좋지만 `OrderController`, `OrderService`, `OrderCancelUtil`에 기존 코드, 문제점, 리팩토링 방향이 주석으로 노출되어 있다.

4. **학습 효과 한계**
   * 사용자가 비즈니스 설계나 트레이드오프를 고심하기보다 "ArrayList.add", "static Util", "Entity 반환" 등 동일한 결함을 기계적으로 스캔하게 만들 위험이 있다.
   * 결함 개수는 많았지만 문제 스타일 다양성은 부족했다.

#### [문제별 판단]

* **004 Point Refund**
  * 상태: 사용 가능.
  * 면접 대비 중요도: 높음.
  * 좋은 점: Entity 직접 Request/Response 사용, 요청 객체를 환불 대상으로 신뢰하는 문제, 조회 실패를 요청 객체로 보정하는 문제가 실무 장애와 직접 연결된다.
  * 보강점: 소유자 검증 누락을 핵심 정답으로 올리고, `PointRefundRequest`가 있는데도 `PointPayment`를 받는 문제를 더 명확히 설명한다. `refundReason`과 `reason` 필드 혼재도 의도된 결함인지 정리한다.

* **005 Delivery Address**
  * 상태: 문제로 사용 전 복구 필요.
  * 면접 대비 중요도: 의도는 중간 이상이지만 현재 상태로는 낮음.
  * 좋은 의도: 배송중 주소 변경 불가, 우편번호 형식 검증, 주소 정규화, 배송 이력 조회는 좋은 소재다.
  * 보강점: 리팩토링 코드와 정답 주석을 `solution/` 또는 답변 후 문서로 이동하고, `src/main/java`는 순수 레거시 코드로 되돌린다. 004와 달라 보이도록 배송 도메인 특화 결함을 전면에 둔다.

* **006 Order Cancel**
  * 상태: 문제로 사용 전 복구 필요.
  * 면접 대비 중요도: 높음. 세 문제 중 의도 자체는 가장 좋다.
  * 좋은 의도: 상태 전이, 취소 가능 시간, 환불 금액 계산, 외부 PG 호출, DB 저장 순서, OCP 분기까지 함께 묻기 좋다.
  * 보강점: 정답 주석을 제거하고, 실제 레거시 코드는 `orElse(createTemporaryOrder(request))`, 외부 PG 선호출, public field 직접 변경, 문자열 타입 분기, 요청 `requestedRefundAmount/requestedStatus/userId` 신뢰 형태로 남긴다. 개선 코드는 `solution/`으로 분리한다.

#### [빠진 안티패턴 및 설명 보강 항목]

* **빠진 안티패턴**
  * 멱등성: 환불/취소 API 재시도 시 중복 환불, 중복 취소 방지.
  * 동시성: 같은 주문/결제/재고에 다중 요청이 들어올 때 Race Condition이 생기는 흐름.
  * RDBMS 동시성 제어: 낙관적 락, 비관적 락, 격리수준, 데드락 가능성.
  * 인증 컨텍스트 기반 인가: request `userId`가 아니라 `Principal`, `SecurityContext`, 저장된 ownerId 비교.
  * 예외 전파와 트랜잭션 롤백: checked exception, 예외 삼킴, rollbackFor 누락, `UnexpectedRollbackException`.
  * 개인정보/민감정보 로깅: 토큰, 카드번호, 주소 원문, 비밀번호 노출.
  * API 하위 호환성: 응답 필드 제거, enum 값 추가, nullable 필드 추가.
  * 목록/검색 API: 페이징, 정렬 안정성, 조건 조회, 인덱스 관점.

* **설명이 더 필요한 부분**
  * Service에서 `ResponseEntity`를 반환하면 왜 계층 경계가 흐려지는가.
  * Controller는 언제 `ResponseEntity<DTO>`를 쓰고, 언제 DTO 직접 반환으로 충분한가.
  * DTO 개선은 Lombok getter/setter 문제가 아니라 API 계약, validation, 서버 관리 상태 분리 문제다.
  * Builder는 Entity 상태 전이를 해결하는 정답이 아니다.
  * JPA N+1과 단순 반복 Repository 조회는 구분해서 말해야 한다.
  * 인메모리 예제에서 `@Transactional`을 말할 때는 "실제 DB 기반이라면"이라는 전제가 필요하다.
  * 외부 PG/API 호출은 DB 트랜잭션으로 롤백되지 않는다.
  * `final List`는 불변 List가 아니다.
  * `@Valid`는 형식 검증이고, 소유자/상태/기간 검증은 Service/Domain 규칙이다.

#### [다음 문제 출제 원칙]

* **원본 문제와 해설 분리**: `src/main/java`에는 학습자가 읽을 순수 레거시 코드만 둔다. 정답 주석, 개선 코드, 리팩토링 방향은 `review-answer.md`, `feedback.md`, `solution/`, `답변후-개념공부.md`로 분리한다.
* **구조적 변형**: 단순 3계층 구조를 차단하고, 이벤트 기반 비동기 리스너, 배치형 스케줄러, 외부 API 연동 어댑터, 필터/인터셉터 등 다른 진입점을 도입한다.
* **출제 금지 안티패턴**: 문항 4~6에서 반복된 `List.add` 저장 버그, static Util 만능주의, Controller 단순 Entity/Map 반환, DTO-Entity 변환 위치 위반은 다음 문제의 핵심 결함으로 쓰지 않는다.
* **핵심 결함 제한**: 한 문제의 핵심 결함은 4~5개, 보조 결함은 2개 이하, 꼬리질문은 2개 이하로 제한한다.
* **신규 고난도 키워드 도입**
  1. **AOP Self-invocation**: 클래스 내부 호출로 인해 프록시가 작동하지 않아 `@Transactional`이 무시되는 결함.
  2. **RDBMS 동시성 제어**: Race Condition 상황을 제시하고 낙관적 락 vs 비관적 락의 설계적 선택과 성능 트레이드오프 유도.
  3. **예외 전파와 트랜잭션 롤백**: checked exception에 대한 `@Transactional` 롤백 누락 또는 예외 삼킴 현상.

#### [다음 문제 후보]

* **추천 1순위: 재고 차감 / 선착순 쿠폰 발급**
  * 이유: 최근 3문제의 상태 변경 흐름을 유지하되, 단순 `save add`가 아니라 다중 요청, Race Condition, RDBMS 락, 재시도, 멱등성으로 사고 수준을 올릴 수 있다.
  * 핵심 결함: 재고 수량을 조회한 뒤 감소시키는 read-modify-write 경쟁, 동시 요청에서 음수 재고/초과 발급, 트랜잭션 경계와 락 위치 혼동, 실패 시 예외 삼킴, 테스트가 단일 요청만 검증.

* **추천 2순위: 주문 목록 / 관리자 검색 API**
  * 이유: 상태 변경 API에서 벗어나 목록/검색/성능/API 계약 중심으로 문제 체감을 바꿀 수 있다.
  * 핵심 결함: `findAll()` 후 stream filter/sort/page 처리, 페이징 없는 전체 목록 반환, 정렬 기준 불안정, Response DTO 개인정보 과다 노출, 조건 조회/인덱스 관점 누락.

* **추천 3순위: 회원 권한 변경**
  * 이유: 요청값 신뢰 문제를 보안/인가 관점으로 바꿀 수 있다.
  * 핵심 결함: 요청의 `role/isAdmin` 신뢰, 인증 컨텍스트와 요청 userId 불일치, 403/404/409 구분 누락, 감사 로그 민감정보 노출.

## 💡 회고 운영 규칙 (AI & 사용자 협업 약정)

1. **풀이 완료 직후 회고 기록**: 사용자가 한 문항의 답변을 완전히 제출하고 채점이 끝날 때마다, 본 `루프-회고록.md` 문서에 해당 회차의 회고 항목을 추가한다.
2. **자가 복제 감지**: 새로운 문제를 출제하기 전, AI는 반드시 이 `루프-회고록.md`를 먼저 읽고 직전 3개 회차와 동일한 결함 코드 모양이 반복되는지 검수한 뒤 출제한다.
3. **점진적 난이도 상향**: 사용자의 답변 완성도가 높아질수록 단순 문법적 결함에서 **아키텍처 설계 결함 -> 동시성/인프라적 장애 유발 결함 -> 기술 스택적 트레이드오프 결함** 순으로 점진적 상향 조절한다.
