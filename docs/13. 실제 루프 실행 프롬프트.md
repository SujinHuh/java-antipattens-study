# 13. 실제 루프 실행 프롬프트

이 문서는 실제 문제 풀이 루프를 시작할 때 사용하는 짧은 실행 프롬프트다.
세부 운영 기준은 `13-1. 실제 루프 상세 레퍼런스.md`에 보관한다.

## 실행 목표

오늘부터 Java/Spring 백엔드 면접 대비 학습 루프를 진행한다.

목표는 짧은 Java/Spring Boot 레거시 코드를 보고 문제점, 안티패턴, 개선점, 트레이드오프를 면접장에서 설명할 수 있게 되는 것이다.

면접 전까지는 아래 두 가지를 같이 가져간다.

1. 실무에서 나올 법한 짧은 레거시 코드 문제를 많이 풀어 반복 패턴을 익힌다.
2. 문제 코드에서 실제로 나온 개념은 면접 답변 수준까지 깊게 설명한다.

## 문제 생성 원칙

- Java + Spring Boot 기반으로 낸다.
- 구조는 가능하면 Controller -> Service -> Repository -> Util 흐름을 따른다.
- DTO/Entity, Exception, Test는 필요할 때 보조 파일로 둔다.
- 30~40분 안에 리뷰 가능한 코드량으로 유지한다.
- 실제 서비스에서 쓰였을 법한 레거시 코드처럼 작성한다.
- 정답, 문제점 목록, 개선 코드, 개념 태그는 답변 전에는 공개하지 않는다.
- README에는 힌트를 과하게 노출하지 않는다.
- 여러 개념을 섞은 문제는 `답변후-개념공부.md`를 만들 수 있다. 단, 답변 전에는 열람 금지 자료로 두고 README에는 답변 후에만 보라고 안내한다.
- IntelliJ에서 Ctrl+B 이동이 가능하도록 source root와 module 설정을 유지한다.
- 문제 생성 직후 `문제검수결과.md`를 남긴다.

## 회차 범위

- 1~8회차에는 JPA/QueryDSL/Liquibase/Kafka/AWS/Redis/분산락/외부 API 장애 대응을 직접 구현 주제로 넣지 않는다.
- 009에서는 JPA/QueryDSL/Liquibase 확장 여부를 사용자에게 제안만 한다.
- 실제 JPA/QueryDSL/Liquibase 문제는 사용자가 동의한 뒤 010 이후 별도 트랙으로 시작한다.

## 풀이 중 운영

- 사용자가 먼저 Controller -> Service -> Repository -> Util -> DTO/Entity -> Exception -> Test 순서로 답변한다.
- 사용자가 힌트를 요청하기 전에는 힌트를 먼저 주지 않는다.
- 사용자가 계층별로 답변을 작성하고 조언을 요청하면 해당 계층 아래에만 조언을 붙인다.
- 분류 태그는 답변 제출 후 피드백 또는 사용자가 요청한 계층별 조언 단계에서만 붙인다.
- 초기 문제, README, 답변 전 안내에는 `[SOLID-SRP / 계층 책임]`, `[Java 안티패턴 / null 처리]` 같은 개념 태그를 노출하지 않는다.

## 답변 작성 기준

사용자 답변과 피드백은 아래 흐름을 따른다.

```text
### 계층명

1. 문제 제목
   - 문제 위치:
   - 문제 유형:
   - 왜 문제인지:
   - 실무 영향:
   - 개선 방향:
   - 면접용 문장:

2. 문제 제목
   - 문제 위치:
   - 문제 유형:
   - 왜 문제인지:
   - 실무 영향:
   - 개선 방향:
   - 면접용 문장:
```

`review-answer.md`에는 최소한 아래 항목을 포함한다.

1. 먼저 볼 것
2. 핵심 문제 요약
3. 사용자 답변 원문
4. 채점
5. 반드시 잡았어야 하는 문제
6. 문제 유형 분류
7. 이번 문제 개념 키워드
8. 안티패턴 지침화 후보
9. 보완 답변
10. 면접관 예상 꼬리질문
11. 다음 문제에서 다시 볼 약점
12. 시니어 개발자의 피드백 (면접관 관점)

`먼저 볼 것`에는 Service/Repository 흐름 추적 체크를 반복해서 적는다.

```text
입력값 -> 조회 -> 검증 -> 계산 -> 외부 호출 -> 저장 -> 결과 반환
```

특히 Service나 Repository 답변을 작성할 때는 각 문제를 위 흐름 중 어디에서 발생한 문제인지 연결한다.
