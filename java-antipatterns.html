안티패턴으로 판단하는 기준

높음

  • Entity를 API 응답으로 직접 반환한다.
  • 조회 실패를 임시 Entity 생성으로 숨긴다.
  • 클라이언트가 보낸 금액/상태를 서버 결정값처럼 믿는다.
  • 외부 PG 호출을 먼저 하고 DB 저장 실패 가능성을 방치한다.

상황 의존

  • Service의 if 분기는 작을 때는 허용되지만, 타입/정책이 늘어나면 OCP 위반 후보가 된다.
  • 조회 실패 표현은 Optional, null, 예외, 빈 객체, 빈 목록 등 여러 방식이 있다. 표현 자체보다 Service 흐름에서 실패를 숨기는지가 중요하다.
  • Map은 API 응답 계약에는 부적절하지만, 내부 임시 속성이나 로그 맥락에는 사용할 수 있다.
  • Builder는 생성에는 좋지만 기존 Entity 상태 전이를 대신하는 정답은 아니다.

흐름 단계별 문제 찾기

면접에서 말하는 순서

먼저 전체 요청 흐름을 Controller -> Service -> Repository -> Util 순서로 보겠습니다.
그 다음 입력값 -> 조회 -> 검증 -> 계산 -> 외부 호출 -> 저장 -> 결과 반환 순서로 따라가며,
클라이언트 요청값을 신뢰하는지, 조회 실패를 숨기는지, 외부 호출과 DB 저장 순서가 안전한지,
마지막 응답이 Entity인지 Response DTO인지 확인하겠습니다.
